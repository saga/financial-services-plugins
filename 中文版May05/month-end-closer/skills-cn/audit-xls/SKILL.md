# 审计电子表格

审计公式与数据的准确性及错误。审计深度由范围决定——从针对选中区域的快速公式检查，到完整的财务模型完整性审计。

## 第 1 步：确定审计范围

如果用户已指定范围，请直接使用。否则，**询问用户**：

> 您希望审计哪个范围？
> - **selection (选中区域)** — 仅针对当前选定的单元格范围
> - **sheet (工作表)** — 仅针对当前活动的工作表
> - **model (模型)** — 整个工作簿，包括财务模型完整性检查（资产负债表平衡、现金勾稽、滚动计算、逻辑合理性）

**model (模型)** 范围是最深层的——在将 DCF、LBO、三表联动、并购、可比公司分析或任何综合财务模型发送给客户或投委会 (IC) 之前，请使用此项。

---

## 第 2 步：公式级检查（适用于所有范围）

无论何种范围，均需执行以下检查：

| 检查项 | 重点关注内容 |
|---|---|
| 公式错误 | `#REF!`, `#VALUE!`, `#N/A`, `#DIV/0!`, `#NAME?` |
| 公式中的硬编码 | 例如 `=A1*1.05` — 其中的 `1.05` 应当是一个单元格引用（假设值） |
| 公式不一致 | 某个公式破坏了所在行或列中邻近单元格的计算模式 |
| 范围错位 | `SUM`/`AVERAGE` 等公式遗漏了首行或末行 |
| 误将公式覆盖为数值 | 单元格看起来像公式，但实际是硬编码的数值 |
| 循环引用 | 无论是有意还是无意的循环引用 |
| 跨表链接断裂 | 引用了已被移动或删除的单元格 |
| 单位/量级不匹配 | 千位与百万位混用，或将百分比存为整数 |
| 隐藏行/工作表 | 可能包含覆盖值或陈旧的计算逻辑 |

## 第 3 步：模型完整性检查（仅限模型范围）

如果审计范围是 **model**，请识别模型类型（DCF / LBO / 三表联动 / 并购 / 可比公司 / 自定义）并执行以下相应的完整性检查。

### 3a. 结构评审

| 检查项 | 重点关注内容 |
|---|---|
| 输入与公式的分离 | 输入值是否与计算逻辑清晰分开？ |
| 配色规范 | 是否遵循“蓝色为输入、黑色为公式、绿色为链接”或其他统一配色方案？ |
| 工作表流转 | 逻辑顺序是否合理（假设表 → 损益表 → 资产表 → 现金流量表 → 估值表）？ |
| 日期标题 | 所有工作表的日期表头是否一致？ |
| 单位 | 全表单位是否统一（千位 vs 百万位 vs 元）？ |

### 3b. 资产负债表 (BS)

| 检查项 | 测试方法 |
|---|---|
| 资产负债表平衡 | 每个期间：总资产 = 总负债 + 股东权益 |
| 留存收益滚动 | 期初留存收益 + 净利润 − 股利 = 期末留存收益 |
| 商誉/无形资产 | 是否来源于并购假设（如果是并购模型） |

如果资产负债表不平，**请量化每个期间的差额并追踪断裂位置** —— 在解决此问题之前，其他一切检查均无意义。

### 3c. 现金流量表 (CF)

| 检查项 | 测试方法 |
|---|---|
| 现金勾稽 | 每个期间：现金流量表期末现金 = 资产负债表货币资金 |
| 现金流量加总 | 经营现金流 (CFO) + 投资现金流 (CFI) + 筹资现金流 (CFF) = 现金净增量 |
| 折旧与摊销 (D&A) 匹配 | 现金流量表中的 D&A 是否等于损益表中的 D&A |
| 资本开支 (CapEx) 匹配 | 现金流量表中的 CapEx 是否与资产负债表中的固定资产 (PP&E) 滚动计划匹配 |
| 营运资本变化 | 符号（正负号）是否与资产负债表变动一致（如应收账款增加代表现金减少） |

### 3d. 损益表 (IS)

| 检查项 | 测试方法 |
|---|---|
| 营收构成计算 | 是否与细分板块/产品明细表勾稽 |
| 税项 | 税所得税费用 = 税前利润 × 税率（需考虑递延所得税调整） |
| 股数计算 | 是否与稀释表一致（期权、可转债、回购） |

### 3e. 循环引用

- 在 LBO 或三表模型中，“利息 → 债务余额 → 现金 → 利息”是常见的有意循环。
- 如果是有意的：验证是否存在“迭代计算”开关且工作正常。
- 如果是无意的：追踪链路并指出如何打破循环。

### 3f. 逻辑与合理性

| 检查项 | 触发警报的情况 |
|---|---|
| 增长率 | 营收增长率 >100% 且无合理解释 |
| 利润率 | 明显超出行业常规水平 |
| 终值占比过高 | 在 DCF 模型中，终值 (TV) > 企业价值 (EV) 的 75%（需警惕） |
| 曲棍球棒增长 | 预测期后几年的增长速度不切实际地大幅提升 |
| 复利效应 | EBITDA 在第 10 年滚动到天文数字 |
| 极端情况测试 | 当增长率为 0% 或负增长、EBITDA 为负或杠杆为负时，模型是否崩溃 |

### 3g. 特定模型类型的常见漏洞

**DCF (现金流折现):**
- 折现率应用期间错误（年中折现 vs 年末折现）
- 终值未折现回当前期间
- WACC 使用账面价值而非市场价值
- 自由现金流 (FCF) 中包含了利息支出（应使用无杠杆现金流）
- 税盾价值被重复计算

**LBO (杠杆收购):**
- 债务偿还与现金清扫 (Cash Sweep) 机制不匹配
- 实物支付 (PIK) 利息未计入本金
- 管理层展期 (Rollover) 未在回报中体现
- 退出倍数应用了错误的 EBITDA（最近 12 个月 LTM vs 未来 12 个月 NTM）
- 初始外部融资中未扣除费用/开支

**并购 (Merger):**
- 增厚/稀释分析使用了错误的股数（交易前 vs 交易后）
- 协同效应未能按阶段实现
- 购买价格分配 (PPA) 不平衡
- 未考虑现金对价的利息损失
- 交易费用未包含在资金来源与运用表中

**三表联动:**
- 营运资本变动正负号错误
- 折旧与固定资产 (PP&E) 明细表不匹配
- 债务到期计划与本金偿付不一致
- 股利超过净利润且无合理解释

## 第 4 步：审计报告

输出审计发现表：

| # | 工作表 | 单元格/范围 | 严重程度 | 类别 | 发现问题 | 建议修复 |
|---|---|---|---|---|---|---|

**严重程度说明：**
- **Critical (严重)** — 产出错误（资产负债表不平、公式断裂、现金勾稽失败）
- **Warning (警告)** — 存在风险（硬编码、公式不一致、极端情形失效）
- **Info (提示)** — 格式/最佳实践（配色方案、布局、命名）

针对 **model** 范围，请先输出一行总结：

> 模型类型: [DCF/LBO/三表/...] — 总体评价: [洁净 / 存在细微问题 / 存在重大问题] — [N] 项严重, [N] 项警告, [N] 项提示

**未经询问请勿修改任何内容** —— 先报告，待要求后再进行修复。

## 备注

- **资产负债表平衡是核心** — 如果财务报表不平，下游的所有计算都值得怀疑。
- **硬编码的覆盖值是“静默错误”的首要来源** — 需强力审计。
- **正负号习惯错误**（如现金流出的正负号方向）非常普遍。
- 如果模型使用 VBA 宏，请注明哪些由宏驱动的计算是公式审计无法覆盖的。

> **💡 Appendix: 领域知识小贴士**
>
> 1. **三表联动 (3-Statement Model)**：财务模型的核心。它像是有三根隐形的线将损益表（赚了多少）、资产负债表（剩了多少）和现金流量表（钱怎么流的）缝在了一起。损益表的**净利润**会增加资产负债表的**留存收益**，同时又是现金流量表的起点。
> 2. **资产负债表为什么必须“平”？**：这是会计界的“质量守恒定律”。**资产 = 负债 + 所有者权益**。如果一边多了一块钱，另一边没变，模型就“不平”了。这通常意味着你虽然记了账，但忘了这笔钱是从哪儿来或到哪儿去了。
> 3. **现金勾稽 (Cash Tie-out)**：这是校验模型的终极绝招。现金流量表算出的期末现金金额，必须和资产负债表里的现金数一模一样。如果不一致，说明你的模型逻辑在某个地方“漏水”了。
> 4. **硬编码 (Hardcodes)**：在公式中直接写数字（如 `=A1*0.25`）。这种做法被称为财务建模的“禁忌”，因为如果以后税率变了，你很难找到并修改所有藏在公式里的 `0.25`。专业的做法是把 `0.25` 放在一个专门的“假设表”里。
> 5. **DCF (现金流折现)**：简单来说，就是预测公司未来每年能赚多少现金，然后考虑“明年的 100 块不如现在的 100 块值钱”，把未来的钱全部“打折”回现在的价值加总。
