# 差异原因追溯

给定单个差异行（包含键值、总账数值、明细账数值、差异分桶、疑似原因），将其追溯至源头并生成根本原因说明。

## 追溯路径

1. **调取总账侧数据** —— 通过内部总账（internal-gl）MCP，获取产生该总账分录的日记账分录或过账明细：包括分录 ID、过账日期、源系统、批次 ID 及编制人。
2. **调取明细账侧数据** —— 通过明细账（subledger）MCP，获取匹配的交易：包括交易 ID、交易/结算日期、交易对手、源数据流及所用汇率。
3. **属性对比** —— 对齐过账日期、汇率/日期、科目映射、数量正负号以及金额正负号。通常属性不一致的地方即为差异原因。

## 原因 → 说明描述

将根本原因写成一个单句，格式为 **“⟨哪一方⟩ ⟨做了什么⟩ 因为 ⟨原因⟩”**，例如：

- “总账在结算日（T+2）过账，而明细账在交易日过账 —— 属于时间性差异，将于 2026-05-07 结清。”
- “明细账使用了 WM/R 下午 4 点汇率，总账使用了彭博收盘汇率 —— 汇率差异导致基准金额偏差 12 个基点。”
- “映射表中证券 ABC123 应当对应总账科目 11420，但明细账传送的是 11410 —— 属于映射错误，需提交参考数据团队处理。”
- “明细账重复录入交易（交易 ID 88412 和 88419 为重复项） —— 属于重复过账，需拦截 88419。”

## 输出格式

对于每个已追溯的差异，返回：

```json
{
  "key": "...",
  "root_cause": "上述格式的单句说明",
  "owner": "ops | reference-data | accounting | upstream-system",
  "expected_clear_date": "YYYY-MM-DD 或 null",
  "action": "monitor | adjust | raise-ticket | suppress"
}
```

注意：只有调账器（resolver）有权编写调整分录 —— 本功能仅用于诊断，不执行过账操作。

> **💡 Appendix: 领域知识小贴士**
>
> 1. **对账差异 (Break)**：金融界的“找茬儿”。就像你发现银行卡余额和记账 APP 对不上，中间差的那笔钱就叫 Break。
> 2. **总账 vs 明细账 (GL vs Subledger)**：总账是“大账本”，统计公司的总资产；明细账是“小清单”，记录每一单具体的买卖。两边对不上，通常是因为信息在传递过程中“变了样”。
> 3. **时间性差异 (Timing Break)**：这是由于“记账时差”引起的。比如你周五刷卡，银行周天还没结算，这种差异通常等两天自动就好了。
> 4. **映射差异 (Mapping Break)**：就像“分类投递”，系统把“苹果核”当成了“可回收物”。在金融里，如果系统把“股票”当成了“债券”去记账，就会出现这种错误。
> 5. **基点 (BPS)**：金融圈的微小度量衡。1 个基点（1 bp）等于 0.01%。虽然听起来很小，但在处理几十亿资金时，差几个基点也是一笔巨款！
