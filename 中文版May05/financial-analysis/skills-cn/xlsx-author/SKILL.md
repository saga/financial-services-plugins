# xlsx-author

当处于**无界面**（Headless，即托管代理/CMA 模式）环境下运行，且需要交付 Excel 工作簿作为**文件产物**（File artifact），而非通过 `mcp__office__excel_*` 接口编辑实时工作簿时，请使用此技能。

## 输出约定

- 写入至 `./out/<name>.xlsx`。如果 `./out/` 目录不存在，请先创建。
- 在最终回复中返回相对路径，以便编排层（Orchestration layer）进行文件提取。

## 如何构建工作簿

编写简短的 Python 脚本并通过 Bash 运行。推荐使用 `openpyxl` 库：

```python
from openpyxl import Workbook
from openpyxl.styles import Font, PatternFill

wb = Workbook()
ws = wb.active; ws.title = "Inputs"
ws["B2"] = "Revenue"; ws["C2"] = 1_250_000_000
ws["C2"].font = Font(color="0000FF")           # 蓝色 = 硬编码输入值
calc = wb.create_sheet("DCF")
calc["C5"] = "=Inputs!C2*(1+Inputs!C3)"        # 黑色 = 计算公式
wb.save("./out/model.xlsx")
```

## 建模惯例（参照 `audit-xls` 标准）

- **蓝/黑/绿颜色规范**：蓝色表示硬编码输入值（Hardcoded input），黑色表示计算公式（Formula），绿色表示跨表或跨文件链接（Link）。
- **计算单元格严禁硬编码**：所有计算逻辑必须通过公式实现；所有原始输入数据必须统一存放在 “Inputs” 标签页中。
- **命名区域（Named ranges）**：对于任何需要被演示文稿（Deck）或备忘录（Memo）引用的数据点，必须设置命名区域。
- **平衡校验（Balance checks）**：必须包含一个 “Checks” 标签页，用于进行财务钩稽核对（如资产负债表是否平衡、现金流量表净增加额与资产负债表货币资金变动是否一致等），并明确显示 TRUE/FALSE 状态。
- **一文件一模型**：除非有明确要求，否则不要向现有工作簿追加内容，应保持模型的独立性。

## 适用禁忌

如果当前环境下 `mcp__office__excel_*` 系列工具可用（即协作插件模式），请优先使用这些工具。它们可以直接驱动用户的实时工作簿，并支持人工审核检查点。本技能仅作为无界面运行模式下生成文件的备选方案。

> **💡 Appendix: 领域知识小贴士**
> 
> *   **硬编码 (Hardcoding)**：这在财务建模里是个“大忌”。想象一下，如果你把去年的收入直接写死在复杂的计算公式里，明年想改数字时就得翻遍整个表格。专业的做法是把所有会变的数字（比如增长率、单价）集中放在一个“输入页”，计算公式只去引用那里，这样修改一个数字，全表都能自动更新。
> *   **DCF (现金流折现模型)**：这是金融界的“测谎仪”。简单来说，就是预测一个公司未来能赚多少钱，然后考虑到“未来的钱没有现在的钱值钱”（因为通货膨胀和风险），把那一堆未来的钱折合成现在的价值，看看这个公司今天到底值多少。
> *   **财务钩稽关系 (Ties/Checks)**：会计是一门关于“平衡”的艺术。比如资产负债表左右必须相等，这就是一种钩稽。在 Excel 里设一个“Checks”页，就像是给模型装了个报警器，一旦数据对不上，它会立刻显示“FALSE”提醒你：模型出错了！
> *   **蓝/黑/绿颜色规则**：这是投资银行和咨询公司的“通用语言”。当你打开一个专业的财务模型，蓝色字体告诉你“这是我可以改的参数”，黑色字体警告你“这是公式，别乱动”。这种约定让跨团队协作变得极其高效。
