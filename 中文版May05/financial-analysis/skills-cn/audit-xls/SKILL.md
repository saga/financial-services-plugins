# 审计电子表格

审计公式与数据的准确性及错误。审计深度由范围决定——从针对选定区域的快速公式检查，到完整的财务模型一致性审计。

## 第一步：确定范围

如果用户已指定范围，请直接使用。否则，**请询问用户**：

> 您希望审计哪个范围？
> - **选定区域 (selection)** —— 仅针对当前选中的单元格范围
> - **工作表 (sheet)** —— 仅针对当前活动的单个工作表
> - **模型 (model)** —— 整个工作簿，包括财务模型一致性检查（资产负债表平衡、现金余额核对、科目滚动勾稽、逻辑合理性）

**模型 (model)** 范围最为深入——在向客户或投资委员会 (IC) 提交现金流折现 (DCF)、杠杆收购 (LBO)、三表链接、并购 (Merger)、可比公司分析 (Comps) 或任何综合财务模型前，请使用此范围。

---

## 第二步：公式级检查（适用于所有范围）

无论何种范围，均需运行以下检查：

| 检查项 | 关注要点 |
|---|---|
| 公式错误 | `#REF!`, `#VALUE!`, `#N/A`, `#DIV/0!`, `#NAME?` |
| 公式中的硬编码 | `=A1*1.05` —— 其中的 `1.05` 应使用单元格引用 |
| 不一致的公式 | 某个公式破坏了行或列中相邻单元格的处理模式 |
| 范围错位 | `SUM`/`AVERAGE` 漏掉了第一行或最后一行 |
| 公式被数值覆盖 | 单元格看起来像公式，但实际上是硬编码的数值 |
| 循环引用 | 无论是有意还是无意产生的 |
| 跨表链接断开 | 引用了已移动或已删除的单元格 |
| 单位/数量级不匹配 | 千位与百万位混用，或百分比被存为整数 |
| 隐藏行/工作表 | 可能包含覆盖值或过时的计算逻辑 |

## 第三步：模型一致性检查（仅适用于“模型”范围）

如果范围是 **模型 (model)**，请识别模型类型（DCF / LBO / 三表链接 / 并购 / 可比公司 / 自定义），并运行相应的各环节检查。

### 3a. 结构审查

| 检查项 | 关注要点 |
|---|---|
| 输入与公式分离 | 输入值是否与计算逻辑有清晰的物理隔离？ |
| 颜色规范 | 蓝色=输入，黑色=公式，绿色=链接 —— 观察模型是否一致地应用了这些规范？ |
| 工作表流向 | 逻辑顺序是否合理（假设 → 损益表 → 资产负债表 → 现金流量表 → 估值）？ |
| 日期表头 | 所有工作表的日期表头是否完全一致？ |
| 精度单位 | 单位是否统一（如：千位 vs 百万位 vs 绝对值）？ |

### 3b. 资产负债表 (BS)

| 检查项 | 测试逻辑 |
|---|---|
| 资产负债表平衡 | 资产总计 = 负债合计 + 股东权益（每个预测期均须平衡） |
| 留存收益滚动 | 期初留存收益 + 净利润 − 股利 = 期末留存收益 |
| 商誉/无形资产 | 是否来源于收购假设（针对并购模型） |

如果资产负债表不平，**请量化每个期间的差异并追踪断裂处** —— 在此问题解决前，其他检查均无意义。

### 3c. 现金流量表 (CF)

| 检查项 | 测试逻辑 |
|---|---|
| 现金余额核对 | 现金流量表期末现金 = 资产负债表货币资金（每个期间） |
| 现金流加总 | 经营现金流 + 投资现金流 + 筹资现金流 = 净现金流变化 |
| 折旧摊销 (D&A) 匹配 | 现金流量表中的 D&A = 损益表中的 D&A |
| 资本开支 (CapEx) 匹配 | 现金流量表中的 CapEx 与资产负债表中固定资产 (PP&E) 的滚动逻辑匹配 |
| 营运资本 (WC) 变动 | 正负号是否与资产负债表变动一致（如：应收账款增加代表现金减少） |

### 3d. 损益表 (IS)

| 检查项 | 测试逻辑 |
|---|---|
| 收入拆解 | 是否能勾稽到分部门/分产品的明细计算 |
| 所得税 | 所得税费用 = 税前利润 × 税率（允许存在递延所得税调整） |
| 股票数量 | 是否勾稽到稀释计算表（期权、可转债、回购等） |

### 3e. 循环引用

- 利息 → 债务余额 → 现金 → 利息，是 LBO/三表模型中常见的有意循环
- 如属有意：验证模型中是否存在“迭代计算”开关且工作正常
- 如属无意：追踪循环路径并标记如何打破循环

### 3f. 逻辑与合理性

| 检查项 | 预警情形 |
|---|---|
| 增长率 | 收入增长率 > 100% 且无合理解释 |
| 利润率 | 明显超出行业常规水平 |
| 终值占比过高 | 终值 (TV) 占 DCF 企业价值 (EV) 的比例 > 75%（黄色预警） |
| 曲棍球棒效应 (Hockey-stick) | 预测期后几年的增长斜率不切实际地陡增 |
| 复利异常 | EBITDA 在第 10 年因复利效应达到离谱的数值 |
| 极端情况测试 | 在增长率为 0%、负增长、负 EBITDA 或负杠杆时模型崩溃 |

### 3g. 特定模型类型的常见漏洞

**DCF (现金流折现):**
- 折现率应用的期间错误（期中折现 vs 期末折现）
- 终值未被折现到现值
- 加权平均资本成本 (WACC) 使用了账面价值而非市场价值
- 自由现金流 (FCF) 包含了利息费用（应为无杠杆现金流）
- 税盾效应被双重计算

**LBO (杠杆收购):**
- 债务偿还逻辑不符合现金清扫 (Cash Sweep) 机制
- PIK 利息未归入本金滚动
- 管理层展期 (Rollover) 未在收益率中体现
- 退出倍数应用于错误的 EBITDA（过去 12 个月 LTM vs 未来 12 个月 NTM）
- 费用/支出未从第一天的股权投资中扣除

**Merger (并购):**
- 增厚/稀释分析使用了错误的股数（交易前 vs 交易后）
- 协同效应未按阶段分步实现
- 购买价格分配 (PPA) 不平衡
- 未计入因现金支付而损失的利息收入 (Foregone Interest)
- 交易费用未包含在资金来源与用途表 (Sources & Uses) 中

**三表链接 (3-statement):**
- 营运资本变动的正负号错误
- 折旧费用与固定资产 (PP&E) 明细表不匹配
- 债务到期计划与本金偿付金额不匹配
- 在无合理解释的情况下，股利发放超过了净利润

## 第四步：输出报告

输出审计发现表：

| # | 工作表 | 单元格/范围 | 严重程度 | 分类 | 发现问题 | 建议修复方案 |
|---|---|---|---|---|---|---|

**严重程度划分：**
- **致命 (Critical)** —— 计算结果错误（资产负债表不平、公式损坏、现金流无法勾稽）
- **警告 (Warning)** —— 潜在风险（硬编码、公式不一致、极端情况崩溃）
- **建议 (Info)** —— 格式/最佳实践（颜色标注、布局、命名规范）

对于 **模型 (model)** 范围，请在开头增加一行总结：

> 模型类型: [DCF/LBO/三表/...] — 总体评估: [清晰 / 存在次要问题 / 存在重大问题] — [N] 项致命, [N] 项警告, [N] 项建议

**未经许可请勿修改任何内容** —— 先报告问题，待用户要求后再修复。

## 注意事项

- **资产负债表平衡优先** —— 如果表不平，下游的所有计算数据均不可信。
- **硬编码覆盖是静默漏洞的第一大来源** —— 需强力搜索。
- **正负号惯例错误**（特别是现金流出项）极为常见。
- 如果模型包含 VBA 宏，请注明哪些由宏驱动的计算是仅靠公式无法审计的。

> **💡 Appendix: 领域知识小贴士**
>
> 财务模型审计听起来高大上，其实核心就在玩一场“数字连连看”。这里有几个你需要了解的职场“黑话”：
>
> 1. **三表链接 (3-Statement Model)**：这是财务世界的基石。损益表（赚了多少）、资产负债表（剩了多少）和现金流量表（兜里有多少钱）这三张表是通过公式紧紧扣在一起的。比如，损益表最后的净利润会流进资产负债表的“留存收益”，这就是所谓的一致性。
> 2. **勾稽关系 (Tie-out)**：这是财务人的强迫症福音。如果说“资产=负债+所有者权益”，那这个等式两边必须一分钱都不差。如果差了一块钱，我们就说这表“不平”，必须审计找出哪里漏了。
> 3. **硬编码 (Hardcode)**：本该用公式计算的地方（比如 `A1+A2`），你直接随手填了个数字（比如 `100`）。这在模型里是大忌，因为当 A1 变动时，你的结果还是 100，这就是“静默漏洞”。
> 4. **现金流折现 (DCF)**：这是一种通过预测公司未来能赚多少现金，再把未来的钱“打折”到今天，算出公司现在值多少钱的方法。
> 5. **杠杆收购 (LBO)**：简单说就是“借钱买公司”。审计这种模型重点看贷款能不能按时还上，收益够不够覆盖借钱的利息。
> 6. **曲棍球棒效应 (Hockey-stick)**：有些人在做预测时，前几年业绩平平，最后一年突然垂直上升。就像曲棍球棒一样，这种预测通常被认为是在吹牛。
