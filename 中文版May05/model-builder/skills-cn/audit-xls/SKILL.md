# 审计电子表格

审计公式与数据的准确性，查找潜在漏洞。审计范围决定了检查深度——从对选定区域的快速公式检查，到对完整财务模型的集成性审计。

## 第 1 步：确定范围

如果用户已指定范围，则直接使用。否则，**请询问用户**：

> 您希望审计哪个范围？
> - **选定区域 (selection)** — 仅针对当前选中的单元格区域
> - **工作表 (sheet)** — 仅针对当前活动的工作表
> - **模型 (model)** — 整个工作簿，包括财务模型完整性检查（资产负债表平衡、现金勾稽、滚动分析、逻辑合理性）

**模型 (model)** 范围是最深度的审计——在发送给客户或投委会 (IC) 之前，请务必对 DCF、LBO、三表联动、并购模型、可比公司分析或任何集成财务模型运行此范围审计。

---

## 第 2 步：公式级检查（适用于所有范围）

无论范围如何，均须运行以下检查：

| 检查项目 | 关注点 |
|---|---|
| 公式错误 | `#REF!`, `#VALUE!`, `#N/A`, `#DIV/0!`, `#NAME?` |
| 公式中的硬编码 | 例如 `=A1*1.05` — `1.05` 应当使用单元格引用 |
| 公式不一致 | 某行或某列中的公式打破了相邻单元格的计算规律 |
| 范围选取偏移 | `SUM` 或 `AVERAGE` 漏掉了首行或末行 |
| 被覆盖的公式 | 看起来像公式但实际被直接粘贴成了硬编码数值的单元格 |
| 循环引用 | 无论是有意为之还是无意误操作 |
| 跨表链接断裂 | 引用了已移动或已删除的单元格 |
| 单位/数量级不匹配 | 千位与百万位混用，或百分比被存为整数 |
| 隐藏行/隐藏标签页 | 可能包含覆盖值或陈旧的计算逻辑 |

---

## 第 3 步：模型完整性检查（仅限模型范围）

如果范围是 **model**，请识别模型类型（DCF / LBO / 三表联动 / 并购 / 可比公司 / 自定义），并运行以下相应的完整性检查。

### 3a. 结构评审

| 检查项目 | 关注点 |
|---|---|
| 输入值与公式分离 | 输入参数是否与计算过程清晰分开？ |
| 颜色规范 | 是否遵循“蓝色=输入，黑色=公式，绿色=链接”或模型既定规范，并保持一致？ |
| 标签页流向 | 逻辑顺序是否合理（假设 → 利润表 → 资产负债表 → 现金流量表 → 估值）？ |
| 日期表头 | 所有标签页的日期表头是否一致？ |
| 单位 | 全局单位是否统一（千位 vs 百万位 vs 实际数值）？ |

### 3b. 资产负债表

| 检查项目 | 测试方法 |
|---|---|
| 报表配平 | 总资产 = 总负债 + 股东权益（每一期均需满足） |
| 留存收益滚动 | 期初留存收益 + 净利润 − 股利 = 期末留存收益 |
| 商誉/无形资产 | 是否与并购假设一致（若涉及 M&A） |

如果资产负债表不平，**请量化每一期的差额并追踪断裂处** —— 在解决此问题之前，其他任何检查都毫无意义。

### 3c. 现金流量表

| 检查项目 | 测试方法 |
|---|---|
| 现金勾稽 | 现金流量表期末现金 = 资产负债表货币资金（每一期均需满足） |
| 现金流加总 | 经营性现金流 + 投资性现金流 + 筹资性现金流 = 现金净增量 |
| 折旧摊销匹配 | 现金流量表中的 D&A = 利润表中的 D&A |
| 资本开支匹配 | 现金流量表中的 CapEx 与资产负债表中的 PP&E 滚动分析一致 |
| 营运资本变动 | 正负号与资产负债表变动一致（如应收/应付账款、存货的变动） |

### 3d. 利润表

| 检查项目 | 测试方法 |
|---|---|
| 收入拆解 | 是否勾稽到分板块/产品的详细计算 |
| 税项 | 所得税费用 = 税前利润 × 税率（允许递延所得税调整） |
| 股份数 | 是否勾稽到稀释分析表（期权、可转债、回购等） |

### 3e. 循环引用

- 在 LBO 或三表联动模型中，“利息 → 债务余额 → 现金 → 利息”是常见的有意循环
- 若为有意：验证是否存在迭代计算开关且运行正常
- 若为无意：追踪循环路径并标记如何打破循环

### 3f. 逻辑与合理性

| 检查项目 | 风险标记情形 |
|---|---|
| 增长率 | 毫无理由的 >100% 收入增长 |
| 利润率 | 偏离行业常态 |
| 终值占比过高 | 在 DCF 模型中，终值 (TV) > 企业价值 (EV) 的 75%（黄色警示） |
| 曲棍球棒曲线 | 预测期后端出现不切实际的爆发式增长 |
| 复利幻觉 | EBITDA 到第 10 年因复利累计到荒谬的量级 |
| 极端情况 | 模型在增长率为 0% 或负值、EBITDA 为负、杠杆率为负时崩溃 |

### 3g. 模型类型特定漏洞

**DCF (现金流折现模型):**
- 折现率应用于错误的期间（年中折现 vs 年末折现）
- 终值未进行折现
- WACC 使用账面价值而非市值
- 自由现金流 (FCF) 包含了利息费用（应使用无杠杆 FCF）
- 税盾收益重复计算

**LBO (杠杆收购模型):**
- 债务偿还与现金清扫 (Cash Sweep) 机制不匹配
- 实物支付利息 (PIK Interest) 未累加至本金
- 管理层滚动持股并未反映在回报率中
- 退出倍数应用于错误的 EBITDA（过去 12 个月 LTM vs 未来 12 个月 NTM）
- 费用/支出未从第一天的股本支出中扣除

**并购模型 (Merger):**
- 增厚/稀释分析使用了错误的股份数（交易前 vs 交易后）
- 协同效应未分段体现
- 买价分配 (PPA) 不平衡
- 未计入因使用现金而损失的利息收入
- 交易费用未包含在资金来源与用途表中

**三表联动 (3-statement):**
- 营运资本变动符号错误
- 折旧与固定资产 (PP&E) 计划表不匹配
- 债务到期计划表与本金支付不匹配
- 在无合理解释的情况下，分红超过了净利润

## 第 4 步：报告

输出审计结论表：

| # | 工作表 | 单元格/区域 | 严重程度 | 类别 | 问题描述 | 建议修复 |
|---|---|---|---|---|---|---|

**严重程度：**
- **致命 (Critical)** — 结果错误（报表不平、公式断裂、现金不勾稽）
- **警告 (Warning)** — 潜在风险（硬编码、公式不一致、极端情况崩溃）
- **提示 (Info)** — 格式规范/最佳实践（颜色标注、布局、命名）

对于 **模型 (model)** 范围，请在报告最上方附带一行摘要：

> 模型类型: [DCF/LBO/三表联动/...] — 总体状态: [清洁 / 轻微问题 / 重大问题] — [N] 项致命, [N] 项警告, [N] 项提示

**未经允许请勿进行任何修改** —— 先报告，根据要求再修复。

## 注意事项

- **报表配平优先** — 如果报表不平，下游的所有计算都值得怀疑
- **硬编码覆盖是静默 Bug 的头号来源** — 务必深度扫描
- **符号规则错误**（现金流出的正负号混淆）极其常见
- 如果模型使用了 VBA 宏，请注明哪些由宏驱动的计算是仅靠公式无法审计的

> **💡 Appendix: 领域知识小贴士**
>
> 欢迎来到财务模型的“质检室”！这里有几个硬核但很有趣的概念：
>
> 1.  **资产负债表配平 (BS Balance)**：你可以把它想象成天平。你的家产（资产）要么是借来的（负债），要么是自己的（所有者权益）。如果这两边不相等，说明你的“数学上帝”罢工了，模型一定有严重的逻辑漏洞。
> 2.  **硬编码 (Hardcodes)**：在公式里直接写数字（比如 `*1.05`）就像在迷宫里埋地雷。如果明年增长率变了，你得翻遍几千个单元格去改。标准的做法是把 `1.05` 放在一个显眼的“假设区”，公式只引用它。
> 3.  **DCF/LBO/三表联动**：这是财务界的三大“神兽”。DCF 是在算这家公司未来能赚多少钱、折合到今天值多少；LBO 是算“借钱买公司”能不能回本；三表联动则是把利润、资产、现金像齿轮一样咬合在一起，动一个数，全身跟着动。
> 4.  **现金勾稽 (Cash Tie-out)**：这是最后的防线。如果你的现金流量表说你年底剩下 100 块，但资产负债表里的银行存款只有 80 块，那消失的 20 块就是模型里的“黑洞”，必须找出来。
> 5.  **商誉 (Goodwill)**：如果你花 10 块钱买了一个净资产只值 6 块的公司，多出来的 4 块就是商誉。它代表了你看好这家公司的品牌、人才或者那些看不见但值钱的“软实力”。
