## 克劳德 (CLAUDE) 核心指令 - 请先阅读

### 环境：Office JS vs Python

**如果是在 Excel 内部运行（Office 加载项 / Office JS 环境）：**
- 直接使用 Office JS (`Excel.run(async (context) => {...})`) —— 不要使用 Python/openpyxl。
- 通过 `range.formulas = [["=B5*B6"]]` 写入公式 —— Office JS 公式会在实时工作簿中原生重新计算。
- 遵循“公式胜过硬编码”原则：设置 `range.formulas`，对于任何应属于计算的内容，绝不要设置 `range.values`。
- 使用 `range.format.font.color` / `range.format.fill.color` 来应用蓝/黑/紫/绿的字体颜色惯例。
- 无需单独的重算步骤 —— Excel 会原生处理计算。
- **合并单元格陷阱**：严禁先调用 `.merge()` 然后在合并后的区域设置 `.values`（这会抛出 `InvalidArgument` 错误，因为区域仍会报告原始尺寸）。正确做法：仅向左上角单元格写入值（`ws.getRange("A7").values = [["SOURCES & USES"]]`），然后对整个区域执行合并和格式化（`ws.getRange("A7:F7").merge(); ws.getRange("A7:F7").format.fill.color = "#1F4E79";`）。

**如果是生成独立的 .xlsx 文件（无实时 Excel 会话）：**
- 使用下文所述的 Python/openpyxl。
- 写入公式字符串（`ws["D20"] = "=B5*B6"`），在交付前运行 `recalc.py`。

本技能的其余部分虽以 openpyxl 为例编写，但同样适用于 Office JS —— 只需转换 API 调用即可。

### 核心原则
* **每项计算必须是 Excel 公式** - 严禁在 Python 中预计算值并将结果硬编码到单元格中。使用 openpyxl 时，应写入 `cell.value = "=B5*B6"`（公式字符串），而非 `cell.value = 1250`（计算结果）。模型必须是动态的，并随输入变化而更新。
* **遵循模板结构** - 遵循 `examples/LBO_Model.xlsx` 或用户提供模板中的组织结构。不要自创布局。
* **使用正确的单元格引用** - 所有公式都应引用适当的单元格。绝不要手动输入应来自其他单元格的数值。
* **保持正负号惯例一致性** - 遵循模板使用的任何正负号惯例（有些对现金流出使用负号，有些使用正号）。全文保持一致。
* **分模块作业，每步与用户核对** - 完整完成一个模块，向用户展示成果，运行该模块的验证检查，并在进入下一个模块之前获得确认。不要端到端地构建整个模型后再展示 —— 后面的模块依赖前面的模块，如果在构建完回报分析后才发现‘资金来源与用途’中有错，意味着到处都要重做。

### 公式颜色惯例
* **蓝色 (0000FF)**：硬编码输入项 - 手动输入的数字，不引用其他单元格。
* **黑色 (000000)**：带计算的公式 - 任何使用算术运算符或函数的公式（如 `=B4*B5`、`=SUM()`、`=-MAX(0,B4)`）。
* **紫色 (800080)**：链接到**同工作表**的单元格 - 直接引用且无计算（如 `=B9`、`=B45`）。
* **绿色 (008000)**：链接到**不同工作表**的单元格 - 跨表引用（如 `=Assumptions!B5`、`='Operating Model'!C10`）。

### 填充颜色调色板 — 专业蓝与灰（除非用户/模板另有规定，否则为默认值）
* **保持简约** — 单元格填充仅使用蓝色和灰色。严禁引入绿色、黄色、红色或多种强调色。专业的 LBO 模型讲究克制。
* **默认填充调色板：**
  * **标题行**（资金来源与用途、经营模型等）：深蓝色 `#1F4E79`，白色加粗字体。
  * **列标题**（第1年、第2年等）：浅蓝色 `#D9E1F2`，黑色加粗字体。
  * **输入单元格**：浅灰色 `#F2F2F2`（或保持白色）—— 蓝色**字体**是信号，填充是次要的。
  * **公式/计算单元格**：白色，无填充。
  * **核心输出项** (IRR, MOIC, 退出股本)：中蓝色 `#BDD7EE`，黑色加粗字体。
* **这就是全部调色板**：3种蓝色 + 1种灰色 + 白色。如果模板有自己的配色，请遵循模板。
* 注意：上述蓝/黑/紫/绿**字体颜色**是为了区分输入、公式和链接。这与此处的**填充**调色板是分开的 —— 两者相辅相成。

### 数字格式标准
* **货币**：根据模板使用 `$#,##0;($#,##0);"-"` 或 `$#,##0.0`。
* **百分比**：`0.0%`（保留一位小数）。
* **倍数**：`0.0"x"`（保留一位小数）。
* **MOIC/精密比率**：`0.00"x"`（保留两位小数以示精确）。
* **所有数字单元格**：右对齐。

## 模板分析阶段 - 务必先行

在填充任何公式前，彻底检查模板：

1. **地图化结构** - 识别每个模块的位置以及它们之间的关系。注意哪些模块的数据会流向其他模块。

2. **了解时间线** - 哪些列代表哪些期间？是否有“交割 (Closing)”或“备考 (Pro Forma)”列？预测期从哪里开始？

3. **识别输入单元格与公式单元格** - 模板通常使用颜色编码、边框或阴影来指示哪些单元格需要输入，哪些需要公式。请遵守这些惯例。

4. **仔细阅读现有标签** - 行标签会准确告诉你预期的计算内容。不要凭空猜测 - 阅读模板的要求。

5. **检查现有公式** - 有些模板可能已预填部分内容。除非用户明确要求，否则不要覆盖运行正常的公式。

6. **注意模板特定的惯例** - 正负号惯例、小计结构、模块组织方式、不同组件是否有独立的工作表等。

## 填充公式 - 通用方法

对于每个需要公式的单元格，遵循以下层级：

### 第 1 步：核对模板
* 该单元格是否已有公式？如果是，验证其正确性并跳过。
* 是否有注释或说明指示了预期的计算？
* 行/列标签是否使计算显而易见？

### 第 2 步：核对用户指令
* 用户是否指定了特定的计算方法？
* 是否有既定假设会影响此公式？
* 是否提到了任何特殊要求？

### 第 3 步：应用标准惯例
* 如果模板和用户均未指定，则使用标准 LBO 模型惯例。
* 记录您所做的任何假设。
* 如果确实不确定，请询问用户。

## 常见难点区域

以下计算模式在 LBO 模型中经常出现问题。遇到时请特别注意：

### 配平模块
* 当两个部分必须相等时（例如，资金来源 = 资金用途），通常有一个项目是“轧差项 (Plug)”（平衡值）。
* 识别哪个项目是轧差项，并将其计算为差值。

### 税务计算
* 税务公式应仅引用相关的收入行和税率。
* **不应**引用无关的部分（例如债务表）。
* 考虑亏损是产生税盾还是直接忽略。

### 利息与循环引用
* 如果利息计算引用的余额受现金流影响，可能会产生循环引用。
* 使用**期初余额**（而非平均或期末余额）来打破循环引用。
* 模式：利息 → 现金流 → 偿还 → 期末余额（如果利息使用期末余额，则会形成循环）。

### 债务偿还 / 现金清扫 (Cash Sweeps)
* 当存在多档债务时，通常有优先级顺序。
* 现金清扫应遵循优先级瀑布流。
* 余额不能为负数 - 适当使用 MAX 或 MIN 函数。

### 回报计算 (IRR/MOIC)
* 现金流必须正负号正确：投资额 = 负数，收益额 = 正数。
* 如果使用 XIRR，需要对应的日期。
* 如果使用 IRR，现金流应在连续的期间内。
* MOIC = 总收益 / 总投资。

### 敏感性分析表
* **使用奇数维度**（5×5 或 7×7）—— 绝不要用 4×4 或 6×6。奇数维度能确保有一个精准的中心单元格。
* **中心单元格 = 基准情况 (Base Case)**。围绕模型的实际假设对称构建行轴和列轴的数值（例如：如果基准退出倍数为 10.0x，则坐标轴为 `[8.0x, 9.0x, 10.0x, 11.0x, 12.0x]`）。中心单元格的 IRR/MOIC **必须**等于模型的实际 IRR/MOIC 输出 —— 这是证明表格勾稽正确的证据。
* **突出显示中心单元格** — 中蓝色填充 (`#BDD7EE`) + 加粗字体，使基准情况在视觉上起到锚定作用。
* Excel 的“模拟分析表 (DATA TABLE)”功能可能无法在 openpyxl 中运行 —— 此时请编写引用行/列标题的显式公式。
* 每个单元格应显示不同的值 —— 如果全部相同，则说明公式变量引用有误。
* 使用混合引用（例如：`$A5` 用于行输入，`B$4` 用于列输入）。

## 应避免的常见错误

| 错误 | 后果 | 修复方法 |
|-------|-----------------|------------|
| 对计算值进行硬编码 | 输入改变时模型不更新 | 始终使用引用源单元格的公式 |
| 复制后单元格引用错误 | 公式指向错误的单元格 | 验证所有链接，使用适当的 $ 绝对引用锁定 |
| 循环引用错误 | 模型无法计算 | 对利息类计算使用期初余额，打破循环 |
| 模块不配平 | 本应匹配的总计项不一致 | 确保有一个项目是轧差项（计算为差值） |
| 出现不可能的负数余额 | 偿还/使用额超过了可用额 | 适当使用 MAX(0, ...) 或 MIN 函数 |
| IRR/回报错误 | 正负号错误或范围不全 | 检查现金流符号，确保公式覆盖所有期间 |
| 敏感性分析表显示相同值 | 公式未随输入变量改变 | 检查单元格引用 - 需要混合引用 ($A5, B$4) |
| 滚算表不勾稽 | 期初余额 ≠ 上期期末余额 | 验证期间间的链接 |
| 正负号惯例不一致 | 加项变成了减项，反之亦然 | 全文严格遵循模板的正负号惯例 |

> **💡 Appendix: 领域知识小贴士**
>
> 即使你从未接触过金融，也可以通过以下几个通俗的比喻来理解这些高大上的概念：
>
> 1. **什么是杠杆收购 (LBO)？**
>    想象你想买一套 100 万的学区房用来出租，但你手里只有 20 万。于是你找银行借了 80 万（这就是“杠杆”）。你打算用租客每月交的房租来还房贷，等几年后房价涨了，你把房子卖掉，还清剩下的房贷，剩下的钱全是你的。LBO 就是“买公司版的贷款买房”：私人股权投资公司用少量的钱加上大量的借款买下一家公司，用这家公司赚的钱来还债，最后卖掉掉公司赚取高额差价。
>
> 2. **资金来源与用途 (Sources & Uses) 是什么？**
>    这就像是买房前的“账单明细”。“用途”告诉你买这房子一共花多少钱（房子原价、过户费、中介费等）；“来源”告诉你这些钱哪来的（你的存款、银行按揭、亲戚借款等）。这两边必须一分不差地对齐，表格才叫“配平”。
>
> 3. **IRR 和 MOIC 是什么？**
>    这是衡量投资赚不赚钱的两个尺子：
>    - **MOIC（投资倍数）**：看“翻了多少倍”。你投了 10 块，回本 30 块，MOIC 就是 3x。它告诉你最后口袋里多了多少钱。
>    - **IRR（内部收益率）**：看“赚钱的速度”。同样是赚 3 倍，如果是 1 年赚到，你就是股神；如果是 30 年赚到，你可能还不如存银行。IRR 考虑了时间价值。
>
> 4. **敏感性分析表 (Sensitivity Table) 是什么？**
>    财务人的“压力测试”。买房的时候你可能会想：“如果房租跌了 10% 怎么办？如果房贷利率涨了 1% 怎么办？”敏感性分析表就是把这些变化的变量列成一个格网，一眼看过去就能知道在不同倒霉（或幸运）的情况下，你最后还能赚多少钱。
