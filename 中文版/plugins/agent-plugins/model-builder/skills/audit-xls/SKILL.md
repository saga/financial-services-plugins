# 审计电子表格

审核公式和数据的准确性和错误。范围决定深度 — 从对选定内容的快速公式检查到完整的财务模型完整性审计。

## 第1步：确定范围

如果用户已经指定了范围，使用它。否则**询问他们**：

> 您希望我审核的范围是什么？
> - **选定内容** — 仅当前选定的区域
> - **工作表** — 仅当前活动工作表
> - **模型** — 整个工作簿，包括财务模型完整性检查（BS平衡、现金核对、结转、逻辑合理性）

**模型**范围是最深的 — 在发送给客户或IC之前，将其用于DCF、LBO、三表、并购、可比公司分析或任何综合财务模型。

---

## 第2步：公式级检查（所有范围）

无论范围如何，都要运行这些：

| 检查 | 要查找的内容 |
|---|---|
| 公式错误 | `#REF!`、`#VALUE!`、`#N/A`、`#DIV/0!`、`#NAME?` |
| 公式内的硬编码 | `=A1*1.05` — `1.05`应该是一个单元格引用 |
| 不一致的公式 | 在行/列中与其邻居模式不同的公式 |
| 范围偏移一 | `SUM`/`AVERAGE`遗漏了第一行或最后一行 |
| 粘贴覆盖公式 | 看起来像公式但实际上是硬编码值的单元格 |
| 循环引用 | 有意的或意外的 |
| 断开的跨表链接 | 引用已移动或已删除的单元格 |
| 单位/规模不匹配 | 千与百万混合，%存储为整数 |
| 隐藏行/标签 | 可能包含覆盖或过时计算 |

---

## 第3步：模型完整性检查（仅模型范围）

如果范围是**模型**，识别模型类型（DCF/LBO/三表/并购/可比公司/自定义）并运行以下适当的完整性检查。

### 3a. 结构审查

| 检查 | 要查找的内容 |
|---|---|
| 输入/公式分离 | 输入是否与计算明确分离？ |
| 颜色约定 | 蓝色=输入，黑色=公式，绿色=链接 — 或者模型使用的任何约定，应用一致吗？ |
| 标签流程 | 逻辑顺序（假设 → IS → BS → CF → 估值）？ |
| 日期标题 | 所有标签一致？ |
| 单位 | 一致（千 vs 百万 vs 实际）？ |

### 3b. 资产负债表

| 检查 | 测试 |
|---|---|
| BS平衡 | 总资产 = 总负债 + 权益（每个期间） |
| RE结转 | 期初RE + 净利润 − 股息 = 期末RE |
| 商誉/无形资产 | 从收购假设流动（如并购） |

如果BS不平衡，**量化每个期间的差距并追溯哪里断裂** — 在此修复之前，其他一切都值得怀疑。

### 3c. 现金流量表

| 检查 | 测试 |
|---|---|
| 现金核对 | CF期末现金 = BS现金（每个期间） |
| CF汇总 | CFO + CFI + CFF = Δ现金 |
| D&A匹配 | CF上的D&A = IS上的D&A |
| CapEx匹配 | CF上的CapEx与BS上的PP&E结转匹配 |
| WC变化 | 符号与BS变动匹配（ΔAR、ΔAP、Δ存货） |

### 3d. 利润表

| 检查 | 测试 |
|---|---|
| 收入构建 | 链接到分部/产品明细 |
| 税务 | 税务费用 = 税前收入 × 税率（允许递延税务调整） |
| 股份数 | 链接到稀释计划（期权、可转换、回购） |

### 3e. 循环引用

- 利息 → 债务余额 → 现金 → 利息是LBO/三表模型中常见的故意循环
- 如果有意的：验证迭代开关存在且工作
- 如果无意的：追溯循环并标记如何打破它

### 3f. 逻辑与合理性

| 检查 | 标记如果 |
|---|---|
| 增长率 | >100%收入增长没有解释 |
| 利润率 | 超出行业正常水平 |
| 终值主导 | TV > ~75% DCF EV（黄色标记） |
| 曲棍球棒 | 预测在不切实际的年份急剧上升 |
| 复合 | EBITDA复合到第10年变成荒谬的$ |
| 边缘情况 | 模型在0%或负增长、负EBITDA、杠杆变为负时崩溃 |

### 3g. 特定于模型类型的错误

**DCF：**
- 折现率应用于错误的期间（年中 vs 年末）
- 终值未折现
- WACC使用账面价值而非市场价值
- FCF包含利息费用（应该是无杠杆）
- 税盾重复计算

**LBO：**
- 债务偿还与现金清扫机制不匹配
- PIK利息未应计到本金
- 管理层展期未反映在回报中
- 退出倍数应用于错误的EBITDA（LTM vs NTM）
- 费用/支出未从第1天权益中扣除

**并购：**
- 增值/稀释使用错误的股份数（交易前 vs 交易后）
- 协同效应未分阶段引入
- 购买价格分配不平衡
- 未包括放弃的现金利息
- 交易费用不在来源与运用中

**三表：**
- 营运资本变化符号错误
- 折旧与PP&E明细表不匹配
- 债务到期计划与本金支付不匹配
- 股息超过净利润没有解释

## 第4步：报告

输出发现表：

| # | 工作表 | 单元格/范围 | 严重性 | 类别 | 问题 | 建议修复 |
|---|---|---|---|---|---|---|

**严重性：**
- **关键** — 错误输出（BS不平衡，公式损坏，现金不核对）
- **警告** — 有风险（硬编码、公式不一致、边缘情况失败）
- **信息** — 风格/最佳实践（颜色编码、布局、命名）

对于**模型**范围，在前面加一行摘要：

> 模型类型：[DCF/LBO/三表/...] — 总体：[干净/小问题/大问题] — [N]个关键，[N]个警告，[N]个信息

**在询问之前不要更改任何内容** — 首先报告，然后在要求时修复。

## 注意事项

- **BS平衡第一** — 如果不平衡，每个下游都值得怀疑
- **硬编码覆盖是#1静默bug来源** — 积极搜索
- **符号约定错误**（现金流出的正负）是极其常见的
- 如果模型使用VBA宏，标记任何无法仅从公式审计的宏驱动计算

## 金融术语和知识详细解释

### 1. 公式审计（Formula Audit）

**定义**：公式审计是检查电子表格中公式正确性和逻辑的过程。

**核心检查项**：
- 公式错误（#REF!、#DIV/0!等）
- 硬编码值
- 循环引用
- 跨表链接完整性

**为什么重要**：
- 确保模型准确性
- 发现潜在错误
- 验证数据完整性
- 符合审计标准

---

### 2. 财务模型完整性检查（Financial Model Integrity Checks）

**定义**：确保财务模型的各个部分正确链接和平衡的验证。

**核心检查**：
| 检查 | 公式 | 预期结果 |
|------|------|---------|
| BS平衡 | 资产 - 负债 - 权益 | = 0 |
| 现金核对 | CF期末现金 - BS现金 | = 0 |
| RE结转 | 期初RE + NI - 股息 - 期末RE | = 0 |

**为什么重要**：
- 确保模型逻辑正确
- 防止错误决策
- 符合投行标准
- 客户交付前必须通过

---

### 3. 循环引用（Circular Reference）

**定义**：当公式直接或间接引用自身时创建循环引用。

**财务模型中的常见循环**：
```
利息费用 → 净利润 → 现金 → 债务余额 → 利息费用
```

**解决方案**：
1. 使用期初余额计算利息
2. 启用迭代计算
3. 添加断路器开关

**为什么棘手**：
- 可能导致计算不收敛
- 影响计算速度
- 需要仔细模型构建

---

### 4. 硬编码覆盖（Hardcoded Override）

**定义**：用固定值替换公式的行为。

**风险**：
- 当输入变化时模型不更新
- 静默错误传播
- 难以追踪问题来源
- 模型失去灵活性

**为什么是#1 bug来源**：
- 看似正常但实际静态
- 审计难以发现
- 下游计算全部受影响
- 可能导致严重估值错误

---

### 5. 合理性检查（Sanity Check）

**定义**：验证模型输出是否在合理范围内的测试。

**常见检查项**：

| 检查 | 标记如果 |
|------|---------|
| 增长率 | >100%无解释 |
| 利润率 | 超出行业正常范围 |
| 终值占比 | >75%的企业价值 |
| hockey-stick | 未来年份急剧上升 |

**为什么重要**：
- 快速识别异常
- 防止不切实际的预测
- 支持投资决策
- 行业最佳实践

---

### 6. 敏感性分析 vs 合理性检查

**敏感性分析**：
- 研究假设变化如何影响输出
- 通常使用数据表
- 展示价值范围

**合理性检查**：
- 验证模型逻辑正确
- 检查是否在合理范围
- 标记需要调查的异常

**两者关系**：
- 合理性检查先于敏感性分析
- 通过合理性检查才能进行敏感性分析
- 确保输入合理，输出才可靠

---

### 7. 工作表/模型范围

**工作表范围**：
- 仅审核当前活动工作表
- 快速公式检查
- 不验证跨表链接

**模型范围**：
- 审核整个工作簿
- 完整性检查
- 包括BS平衡、现金核对
- 投行客户交付前必须

**选定内容范围**：
- 仅审核选定区域
- 最快但最浅
- 用于快速验证

---

### 8. 颜色编码约定

**行业标准**：

| 颜色 | 含义 | 示例 |
|------|------|------|
| 蓝色 | 硬编码输入 | 历史数据、假设 |
| 黑色 | 公式 | 计算、小计 |
| 绿色 | 跨表链接 | 引用其他工作表 |
| 紫色 | 同表链接 | 引用同一工作表 |

**为什么重要**：
- 快速识别单元格类型
- 防止意外覆盖公式
- 加速模型审查
- 行业最佳实践

---

### 9. 符号约定（Sign Convention）

**定义**：关于如何表示现金流入和流出的约定。

**常见约定**：
- **方案A**：流出为负，流入为正
- **方案B**：所有金额为正，流出用括号表示

**为什么重要**：
- 确保计算一致
- 防止符号错误
- 便于理解模型逻辑
- 行业标准做法

---

### 10. 边缘情况测试（Edge Case Testing）

**定义**：测试模型在极端或异常条件下的行为。

**常见边缘情况**：
- 0%增长
- 负增长
- 负EBITDA
- 杠杆变为负（净现金）

**为什么重要**：
- 确保模型稳健性
- 发现隐藏的错误
- 防止极端情况崩溃
- 符合审计要求

## Appendix: 金融背景知识

这份文件是"Excel 审计（Audit XLS）"技能的详细说明。在 Model Builder 的上下文中，Audit XLS 是模型从零建造完成后的"出厂检验"——验证每一个公式、每一个引用、每一个勾稽关系。

---

### 1. 为什么 Model Builder 的审计特别重要？

**核心原因**：Model Builder 从零建造模型——没有"前一个版本"可以参考。所有公式都是新建的——错误概率比更新现有模型高得多。

**比喻**：新盖的楼必须做验收，不能"相信建筑工人没错"。

---

### 2. 从零模型的"审计清单"

| 检查项 | 方法 |
|--------|------|
| 公式覆盖率 | 不应该有任何"硬编码"的数字——全部应该是公式 |
| 三表勾稽 | BS 平衡吗？净利润流入 CF 了吗？期末现金 = BS 现金吗？ |
| 引用完整性 | 有 #REF! 吗？有外部引用丢失吗？ |
| WACC 合理性 | 股权成本、债务成本、税率都合理吗？ |
| 终值合理性 | 终值占 DCF 估值超过 80% 吗？（如果是，假设太乐观） |

---

### 给小白的一句话

> Aaudit XLS 在 Model Builder 中就是"楼房验收师"——从零建造的模型就像一栋刚盖好的楼，你必须逐层检查结构安全（三表勾稽）、电路是否通了（所有公式链接）、承重墙有没有问题（终值占估值 80%+意味着你的假设支撑了整个估值，极其脆弱）。跟 renovation（更新现有模型）不同，new construction（从零建造）没有"前一个版本"做参照——所以审计必须彻底。
