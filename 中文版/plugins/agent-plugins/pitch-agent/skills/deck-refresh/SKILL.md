# 演示文稿更新

更新演示文稿中的数字。演示文稿是格式的真实来源；您只更改数值。

## 环境检查

此技能在 PowerPoint 插件和聊天环境中都能工作。开始前先确定您所在的环境——编辑机制不同，但意图相同：

- **插件** — 演示文稿实时打开；直接编辑文本运行、表格单元格和图表数据。
- **聊天** — 演示文稿是上传的文件；通过使用新值重新生成受影响的幻灯片并将结果写回来进行编辑。

无论哪种方式：尽可能最小化更改，保持现有格式不变。

这是一个四阶段过程，第三阶段是审批门。在用户看到计划之前不要进行编辑。

## 阶段 1 — 获取数据

使用 `ask_user_question` 了解新数字的提供方式：

- **粘贴映射** — 用户输入或粘贴"revenue $485M → $512M, EBITDA $120M → $135M"。最清晰的情况。
- **上传的Excel** — 旧/新列，或用户想要从中提取的新输出工作表。读取它，在信任之前确认哪一列是哪一列。
- **仅新值** — "Q4收入为$512M，利润率为22%"。您需要确定每个值替换什么。可行，但在接触任何内容之前确认映射——您映射到收入的"$512M"如果用户指的是毛利润，将是一场无声的灾难。

还要询问**派生数字**：如果收入变动，用户是否希望重新计算增长率和份额百分比，还是保持不变？大多数演示文稿中都有"+15% YoY"这样的内容，现在可能已经过时。是否处理这些应由用户决定，而不是您。

## 阶段 2 — 读取所有内容，查找所有实例

阅读每张幻灯片。对于每个旧值，找到所有实例——包括那些看起来不同的：

| 变体 | 示例 |
|---|---|
| 规模 | `$485M`, `$0.485B`, `$485,000,000` |
| 精度 | `$485M`, `$485.0M`, `~$485M` |
| 单位样式 | `$485M`, `$485MM`, `$485 million`, `485M` |
| 嵌入 | "revenue grew to $485M", "a $485M business", 轴标签 |

一个演示文稿在幻灯片3上显示`$485M`，在幻灯片8的图表轴上显示`485`，在幻灯片15的脚注中显示`$485.0 million`，这是同一个数字的三个实例。查找替换会遗漏其中两个。您不应遗漏。

**数字隐藏的地方：**
- 文本框（明显）
- 表格单元格
- 图表数据标签和轴标签
- 图表源数据 — 驱动条形图的数字，而不仅仅是上面的标签
- 脚注、来源行、小字体
- 演讲者备注（如果用户关心）

构建一个列表：对于每个旧值，它出现的每个位置、出现的确切文本以及它将变成什么。这个列表就是计划。

## 阶段 3 — 呈现计划，获得批准

**这是对某人花时间制作的演示文稿进行的破坏性操作。** 在编辑任何内容之前显示完整的变更列表。格式化使其易于扫描：

```
$485M → $512M (Revenue)
  Slide 3  — Title box: "Revenue grew to $485M"
  Slide 8  — Chart axis label: "485"
  Slide 15 — Footnote: "$485.0 million in FY24 revenue"

$120M → $135M (Adj. EBITDA)
  Slide 3  — Table cell
  Slide 11 — Body text: "$120M of Adj. EBITDA"

FLAGGED — 可能是派生的，不在您的映射中：
  Slide 3  — "+15% YoY" (增长率 — 如果基准年没有变化，可能已过时？)
  Slide 7  — "12%市场份额" (这是从$485M / 市场规模计算的吗？)
```

标记部分很重要。您不仅仅是执行查找替换——您正在捕捉用户在晚上11点可能会错过的二阶效应。如果映射说`$485M → $512M`，而幻灯片3旁边还有`+15% YoY`，那么这个增长率可能现在是错误的。标记它；不要默默地修复它，也不要默默地留下它。

使用 `ask_user_question` 获取批准：按显示的方式进行、进行但跳过标记的项目，或让他们先修改映射。

## 阶段 4 — 执行、保留、报告

对于每个变更，进行最小化的编辑以完成任务。具体如何操作取决于您的环境：

- **插件** — 在实时演示文稿中直接编辑特定的运行、单元格或图表系列。
- **聊天** — 使用新值重新生成受影响的幻灯片，保持所有其他元素完全原样，然后将其写回文件。

无论哪种方式，标准都是相同的：

- **形状中的文本** — 更改值，保持字体/大小/颜色/粗体状态完全不变。如果`$485M`在句子中是14pt深蓝色粗体，那么`$512M`在同一个句子中也是14pt深蓝色粗体。
- **表格单元格** — 更改单元格，保持表格不变。
- **图表数据** — 更新底层系列值，使条形图/线图实际移动。仅编辑标签而不编辑数据会留下一个说谎的图表。

不要重新格式化任何您不需要接触的内容。演示文稿的现有样式本质上是正确的；您是外科医生，不是翻新者。

最后一次编辑后，报告实际发生的情况：

```
在8张幻灯片上更新了11个值。

已更改：
  [阶段3的列表，现在是过去式]

仍标记 — 未更改：
  Slide 3 — "+15% YoY" (派生；需单独确认)
  Slide 7 — "12%市场份额"
```

对每张编辑过的幻灯片运行标准视觉验证检查。变长的数字（`$485M` → `$1,205M`）现在可能会溢出其文本框或推动表格列宽。在用户发现之前抓住它。

## 您不做的事情

- **不重建幻灯片** — 如果幻灯片的叙述在新数字下不再有意义（"利润率压缩"但利润率上升），标记它，不要重写它。
- **除非被要求，否则不重新计算** — 派生数字由用户决定。您的阶段1问题涵盖了这一点。
- **不接触格式** — 如果演示文稿使用`$MM`而用户的映射说`$M`，匹配演示文稿，而不是映射。值会改变；样式保持不变。

## 金融术语解释

### 1. 演示文稿更新（Deck Refresh）

**定义**：演示文稿更新是指在不重建整个演示文稿的情况下，更新其中的数字数据。这在财务演示中非常常见，尤其是在季度财报发布后或市场数据更新时。

**常见场景**：
- **季度更新**：将Q3数据替换为Q4数据
- **收益更新**：在财报发布后更新收益数据
- **可比公司滚动**：更新可比公司的财务数据
- **市场数据更新**：刷新市场规模和份额数据

**关键原则**：
- **最小化更改**：只更改数值，保持格式不变
- **完整性**：找到所有数字的所有实例
- **准确性**：确认映射关系，避免错误替换
- **透明度**：在执行前展示计划并获得批准

### 2. 派生数字（Derived Numbers）

**定义**：派生数字是基于其他数字计算得出的数值，如增长率、百分比份额、比率等。

**示例**：
- **增长率**：(本期收入 - 上期收入) / 上期收入
- **市场份额**：公司收入 / 市场总收入
- **利润率**：净利润 / 收入

**处理注意事项**：
- 当基础数据更新时，派生数字可能变得过时
- 在更新前询问用户是否需要重新计算派生数字
- 标记可能需要更新的派生数字供用户确认

### 3. 可比公司分析（Comps Analysis）

**定义**：可比公司分析是一种估值方法，通过比较目标公司与类似上市公司的财务指标和估值倍数来评估公司价值。

**步骤**：
1. 选择可比公司（同行业、相似规模）
2. 收集财务数据（收入、利润、估值等）
3. 计算估值倍数（P/E、EV/Revenue、EV/EBITDA等）
4. 将倍数应用于目标公司

**常见倍数**：
- **P/E比率**：市盈率 = 股价 / 每股收益
- **EV/Revenue**：企业价值与收入比
- **EV/EBITDA**：企业价值与EBITDA比

**用途**：
- 确定公司的相对估值
- 支持投资决策
- 并购交易定价参考

### 4. 收益更新（Earnings Update）

**定义**：收益更新是指公司发布季度或年度财务报告后，更新相关的财务数据和分析。

**关键数据点**：
- **收入**：公司的销售收入
- **净利润**：扣除所有费用后的利润
- **每股收益（EPS）**：每股普通股的收益
- **EBITDA**：息税折旧摊销前利润
- **毛利率**：毛利占收入的百分比
- **经营利润率**：经营利润占收入的百分比

**更新流程**：
1. 获取最新财报数据
2. 更新演示文稿中的相关数字
3. 重新计算派生指标（增长率、比率等）
4. 验证所有更改的准确性

### 5. 市场份额（Market Share）

**定义**：市场份额是指公司在特定市场中的销售额占整个市场销售额的百分比。

**计算公式**：
- 市场份额 = (公司收入 / 市场总收入) × 100%

**重要性**：
- **竞争地位**：高市场份额通常意味着强大的竞争优势
- **增长机会**：市场份额变化反映公司的竞争表现
- **估值影响**：市场份额是评估公司价值的重要因素

**计算注意事项**：
- 确保使用一致的市场定义
- 考虑时间范围（年度、季度）
- 注意数据来源的可靠性

### 6. 增长率（Growth Rate）

**定义**：增长率是衡量指标（如收入、利润）在一段时间内变化的百分比。

**计算公式**：
- 同比增长率 = (本期值 - 上期值) / 上期值 × 100%

**类型**：
- **收入增长率**：衡量销售收入的增长
- **利润增长率**：衡量盈利能力的增长
- **用户增长率**：衡量用户基础的增长

**分析要点**：
- 增长率是否可持续
- 与行业平均水平的比较
- 增长率的变化趋势

### 7. EBITDA

**定义**：EBITDA代表息税折旧摊销前利润，是衡量公司核心业务盈利能力的指标。

**计算公式**：
- EBITDA = 营业收入 + 折旧 + 摊销

**优点**：
- 消除了不同资本结构和税收政策的影响
- 便于跨公司比较
- 作为经营现金流的近似值

**局限性**：
- 忽略资本支出需求
- 不考虑利息和税费
- 可能被管理层操纵

### 8. 季度更新（Quarterly Refresh）

**定义**：季度更新是指每季度对财务模型、演示文稿或分析报告进行的数据更新。

**内容**：
- 最新的财务数据
- 更新的市场数据
- 修订的预测
- 新的分析和见解

**重要性**：
- 保持信息的时效性
- 支持决策制定
- 满足报告要求

**最佳实践**：
- 建立标准化的更新流程
- 验证数据的准确性
- 记录所有变更

### 9. 估值倍数（Valuation Multiples）

**定义**：估值倍数是用于评估公司价值的比率，通常基于财务指标。

**常用倍数**：
- **P/E比率**：市盈率
- **EV/Revenue**：企业价值与收入比
- **EV/EBITDA**：企业价值与EBITDA比
- **Price/Book**：市净率

**应用场景**：
- **相对估值**：与同行业公司比较
- **投资决策**：评估投资机会的吸引力
- **交易定价**：并购交易中的定价参考

### 10. 视觉验证（Visual Verification）

**定义**：视觉验证是指在编辑完成后，通过视觉检查确保更改正确且格式保持一致。

**检查要点**：
- 数字是否正确更新
- 文本框是否溢出
- 表格列宽是否需要调整
- 图表是否正确显示新数据
- 整体格式是否保持一致

**重要性**：
- 避免格式错误影响专业性
- 确保演示文稿的视觉质量
- 防止数字溢出等明显错误

## Appendix: 金融背景知识

这份文件是"演示文稿更新（Deck Refresh）"技能的详细说明。在投资银行中，一个 Pitch Deck 不会只用一次——每次见新客户、每季度公司出财报、每个月市场变化，都需要更新数字。Deck Refresh 就是高效更新投行 PPT 的"流水线"。

---

### 1. 什么是 Deck Refresh？

**类比：**
想象你经营一家餐馆。你的菜单（Pitch Deck）已经设计好了——排版精致、照片好看。但每个月食材价格会变化，你需要把菜单上"牛排套餐 $68"改成"牛排套餐 $72"，但不改变菜单的设计和排版。

Deck Refresh 就是投行的"菜单数字更新"——只改数字，不动设计。

---

### 2. Deck Refresh 的"四阶段流程"

| 阶段 | 内容 | 关键动作 |
|------|------|---------|
| 阶段 1：扫描 | 识别所有需要更新的数字 | 列出所有变化的数字 |
| 阶段 2：计划 | 告诉用户"我会改什么" | 用户审批后才能动手 |
| 阶段 3：执行 | 批量替换数字 | 保持格式不变 |
| 阶段 4：验证 | 确认所有数字正确 | 截图对比、手动检查 |

**核心原则：** 在用户看到计划之前，绝不动手改——因为客户可能不想改某个数字。

---

### 3. Deck Refresh 的"三大场景"

| 场景 | 触发 | 例子 |
|------|------|------|
| 季度更新 | 公司发布新财报 | Q3 数据 → Q4 数据 |
| 市场滚动 | 可比公司股价变化 | P/E 从 25x 变成 28x |
| 数据修正 | 发现错误 | $485M → $512M |

---

### 4. 投行 PPT 中常见的"数字陷阱"

| 陷阱 | 说明 |
|------|------|
| 硬编码文本 | 数字不是来自 Excel 链接，而是手动敲进去的——更新时容易漏 |
| 不同页同一数字 | 同一产品在不同页出现，改了首页忘了第 10 页 |
| 图表数据 | 图表的源数据在 Excel 里，只改 PPT 文本不够 |
| 累计值 | 改了一个月的数据，但季度累计值没更新 |

---

### 5. 真实案例：一次失败的 Deck Refresh

**背景**：某投行分析师被要求"把 Q3 数字更新到 Q4"，他花了一晚上改完了 50 页的 Pitch Deck。

**第二天早上**：MD 发现第 3 页的可比公司 P/E 还是 Q3 的数字（没更新），因为图表数据在 Excel 里，分析师只改了 PPT 文本。

**后果**：Pitch 推迟一天，分析师被骂。

**教训**：Deck Refresh 不仅仅是"找到数字改数字"，而是要理解"每个数字是从哪里来的"。

---

### 给小白的一句话

> Deck Refresh 就是投行界最"无聊但致命"的工作——你不做，Pitch Deck 里的数字就是过时的（客户看到会问"你们为什么用上季度的数据？"）。但做起来又极其繁琐：一个 50 页的 Pitch Deck 可能有 300 个数字要更新，改完首页忘了第 30 页就完蛋了。记住：格式是神圣不可侵犯的（字体、颜色、排版丝毫不能动），数字是唯一要换的东西。先列清单、后动手、最后逐页验证——这就是 Deck Refresh 的"三字经"。
