# IB演示文稿检查器

对演示文稿执行全面QC，涵盖四个维度。读取每张幻灯片，然后报告发现。

## 环境检查

本技能在PowerPoint加载项和聊天中均可工作。开始前识别你所在的环境：

- **加载项** — 从实时打开的演示文稿读取。
- **聊天** — 从上传的`.pptx`文件读取。

这是只读报告 — 无编辑 — 所以两种环境中的工作流相同。

## 工作流

### 读取演示文稿

从每张幻灯片提取文本，跟踪每行来自哪张幻灯片。你需要幻灯片级归属来定位每个发现（"$500M出现在幻灯片3和8，但幻灯片15显示$485M"）。30张幻灯片的演示文稿内容太多，无法可靠保存在工作内存中 — 将提取的文本写入文件，以便数字检查脚本可以处理它。

脚本期望带幻灯片标记的markdown输入。格式为：

```
## 幻灯片1
[幻灯片1文本内容]

## 幻灯片2
[幻灯片2文本内容]
```

### 1. 数字一致性

对你收集的内容运行提取脚本：

```bash
python scripts/extract_numbers.py /tmp/deck_content.md --check
```

它标准化单位（$500M vs $500MM vs $500,000,000 → 相同数字），将值分类（收入、EBITDA、倍数、利润率），并标记当同一指标类别在不同幻灯片上显示冲突值时。这是人类在第五次通读时最可能遗漏的部分。

除了脚本标记的内容，验证：
- 计算正确（总计求和、百分比相加、增长率匹配端点）
- 单位风格一致 — 演示文稿应选择$M或$MM并坚持使用
- 时间段对齐 — FY vs LTM vs 季度，明确标记

### 2. 数据与叙述对齐

将主张映射到 supposed 支持它们的数据。这是演示文稿悄悄出错的地方 — 有人编辑了幻灯片7上的图表但忘记了幻灯片4上的叙述。

- 趋势陈述（"利润率下降"） → 图表确实朝那个方向吗？
- 市场地位主张（"#1玩家"） → 收入和份额数据支持它吗？
- 合理性 — "100亿美元市场中的#1" 收入2亿美元是0.2%份额；那不是#1

### 3. 语言润色

IB演示文稿有语域。扫描任何打破它的内容：随意措辞（"相当好"、"很多"）、缩写、感叹号、没有数字模糊量词、同一概念的不一致术语。

参见 `references/ib-terminology.md` 了解替换模式。

### 4. 视觉和格式QC

对每张幻灯片运行标准视觉验证检查。你在寻找：缺失的图表来源引用、缺失的轴标签、排版不一致、数字格式漂移（同一演示文稿内1,000 vs 1K）、日期格式漂移、脚注和免责声明缺口。

视觉验证捕捉文本提取中不会出现的重叠、溢出和对比度问题。不要跳过它 — 没有来源引用的图表在文本转储中看起来与正确引用的图表相同。

## 输出

使用 `references/report-format.md` 作为结构。按严重性分类：

- **关键** — 数字不匹配、事实错误、数据与叙述矛盾。这些阻碍客户交付。
- **重要** — 语言、缺失来源、术语漂移。应该修复。
- **轻微** — 字体大小、间距、日期格式。润色。

以关键项开头。如果没有任何关键项，明确说明 — "未发现数字不一致"是一个发现，不是没有发现。

## Appendix: 金融背景知识

这份文件是"投行 PPT 检查（IB Check Deck）"技能的详细说明。在投资银行中，一个 Pitch Deck 发出去之前必须经过严格的"安全检查"——就像飞机起飞前的检查清单。

---

### 1. 什么是 IB Check Deck？

**类比：**
想象你是杂志社的校对编辑。一篇深度报道写完，不能直接发表——必须经过校对：
1. 事实核查（数据对不对？来源有没有？）
2. 格式检查（字号、行距、标题统一了吗？）
3. 合规审查（有没有违规内容？）

IB Check Deck 就是投行 PPT 的"校对系统"——确保 PPT 在发出之前没有任何错误。

---

### 2. PPT 检查的"三大维度"

#### 维度 1：数据一致性
- 同一数字在不同页面是否一致？
- PPT 数字是否和 Excel 模型一致？
- 百分比加起来是不是 100%？

#### 维度 2：格式统一性
- 所有页面的字体、字号、颜色是否统一？
- 表格格式是否一致？
- 图表配色是否符合品牌色？

#### 维度 3：合规完整性
- 每张图表是否标注了数据来源？
- 是否有免责声明？
- 是否有不当的业绩承诺？

---

### 3. 投行 PPT 的"十大核查项"

| # | 检查项 | 通过标准 |
|---|--------|---------|
| 1 | 封面信息 | 客户名、日期、Logo 正确 |
| 2 | 页码 | 连续、无缺失 |
| 3 | 字体 | 统一为 Arial/Calibri |
| 4 | 数字格式 | 千分位逗号、百分比格式统一 |
| 5 | 表格对齐 | 数字右对齐、文字左对齐 |
| 6 | 图表来源 | 每张图下方有 Source 行 |
| 7 | 数字交叉检查 | PPT 数字 = Excel 数字 |
| 8 | 免责声明 | 最后一页必须有 |
| 9 | 超链接 | 全部有效 |
| 10 | 拼写检查 | 无拼写错误 |

---

### 4. 投行最不能接受的"三大错误"

| 错误 | 严重等级 | 后果 |
|------|---------|------|
| 客户公司名字拼错 | 致命 | 直接丢掉客户 |
| 估值数字算错 | 严重 | 失去信誉 |
| 用错 Logo/品牌色 | 严重 | 合规审查 |

---

### 5. 真实案例：一个代价 $10M 的 PPT 错误

**背景**：某投行在给客户竞标时，PPT 中是数量级错误——把 $50M 写成了 $500M。

**客户反应**："你连基本的数学都搞不清楚，我怎么相信你能帮我的公司卖个好价钱？"

**结果**：投行丢掉了这笔交易，估计损失佣金 $10M+。

---

### 给小白的一句话

> IB Check Deck 就是投行 PPT 的"出厂检验"——就像汽车出厂前要经过 100 道质量检查一样，投行的 Pitch Deck 发出前也要经过逐页、逐字、逐数字的严格审查。为什么这么严格？因为一份 PPT 可能价值几百万甚至几千万的佣金——一个拼写错误可能让你丢掉客户，一个数字错误可能让你被告上法庭。在投行，没有"差不多"——只有"完美"和"被骂"。
