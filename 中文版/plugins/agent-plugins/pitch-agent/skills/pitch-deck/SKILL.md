## 工作流程决策树

**这是什么类型的任务？**

```
┌─ 使用源数据填充空模板？
│  └─→ 遵循下面的"模板填充工作流程"
│
├─ 编辑已填充的现有幻灯片？
│  └─→ 提取当前内容，修改，重新验证
│
└─ 修复现有幻灯片的格式问题？
   └─→ 参见"常见失败"表格，应用有针对性的修复
```

## ⚠️ 关键渲染限制

**LibreOffice 用于验证，但不能准确渲染 PowerPoint 文件。** 它会破坏字体、渐变、形状位置、文本换行和一些表格格式。

**这意味着什么：** 在 LibreOffice 中通过视觉验证的幻灯片在 Microsoft PowerPoint 中可能仍然存在问题。验证循环可以捕获结构问题（缺失内容、损坏的表格、占位符格式保留），但 **无法** 捕获字体替换、细微的对齐偏移或渐变问题。

**必需操作：** 交付输出时始终包含以下声明：
> "此文件使用 LibreOffice 验证。分发前请在 Microsoft PowerPoint 中查看，因为可能存在渲染差异。"

## 模板填充工作流程

复制并跟踪进度：

```
演示文稿进度：
- [ ] 阶段1：提取并验证源数据
- [ ] 阶段2：将内容映射到模板部分
- [ ] 阶段3：用正确的格式填充幻灯片
- [ ] 阶段4：验证 → 修复 → 重复直到干净
- [ ] 阶段5：最终验证
```

### 阶段1：数据提取
1. **创建备份** 在进行任何修改之前，备份原始模板 — 复制到 `[filename]_backup.pptx`。直接 XML 编辑或意外错误可能会损坏文件。
2. 识别所有源材料（Excel、CSV、PDF 报告、Word 文档、数据库、网络来源）
3. 从每个来源提取相关数据点
4. 对照原始来源验证所有数字
5. 标准化单位和货币（将所有数字转换为模板中使用的主要单位/货币）
6. 记录需要验证的任何计算 → 参见 [`calculation-standards.md`](reference/calculation-standards.md) 获取公式

### 阶段2：内容映射
1. **打开并视觉审查模板** — 在修改前了解其结构、样式和现有内容
2. 分析模板结构 — 识别所有占位符区域和内容框
3. 将源数据映射到相应的模板部分 → 参见 [`slide-templates.md`](reference/slide-templates.md) 获取映射指南
4. 识别占位符指导框（任务创建者提供的彩色说明框）
5. 记录任何数据差距或不匹配 → 参见 [`slide-templates.md`](reference/slide-templates.md#handling-data-template-mismatches) 获取解决方案

### 阶段3：模板填充
1. **移除或重新格式化占位符框** — 彩色说明框显示要创建什么，而不是如何格式化。删除它们并在其位置创建格式正确的内容。参见 [关键反模式](#critical-anti-patterns-never-do-these)。
2. 用映射的内容填充每个部分（先关注内容）
3. **然后应用格式** 以匹配模板样式 → 参见 [`formatting-standards.md`](reference/formatting-standards.md)
4. 将表格创建为实际表格对象（切勿使用竖线/制表符分隔的文本）→ 参见 [`xml-reference.md`](reference/xml-reference.md#table-implementation)
5. 将箭头/形状创建为 PowerPoint 对象 → 参见 [`xml-reference.md`](reference/xml-reference.md#arrow-shapes)
6. 如果任务文件中提供了公司标志，则插入；如果不可用，向用户标记："[LOGO NOT PROVIDED - please supply company logo]"

### 阶段4：验证 → 修复 → 重复

**这是一个反馈循环。重复直到所有检查通过或触发升级。**

```bash
# 转换为图像进行视觉验证
soffice --headless --convert-to pdf presentation.pptx
pdftoppm -jpeg -r 150 presentation.pdf slide
```

**验证清单（检查每张幻灯片图像）：**
- [ ] 文本在背景上可读吗？
- [ ] 表格是实际对象（列对齐，不是竖线/制表符分隔的文本）？
- [ ] 图表/表格填充指定区域？
- [ ] 项目符号格式在部分内一致？
- [ ] 字体大小在同级框中匹配？
- [ ] 没有内容超出幻灯片边界？
- [ ] **没有保留占位符格式**（没有大的彩色框包含数据）？
- [ ] **没有基于文本的"表格"**（没有 `|` 或制表符分隔符创建假列）？
- [ ] **跨幻灯片一致性**：相同指标/数字在所有出现的幻灯片上相同？

**修复循环协议：**

| 循环 | 操作 |
|------|------|
| 1 | 修复所有已识别的问题，重新验证 |
| 2 | 修复剩余问题，重新验证 |
| 3 | 如果问题仍然存在，记录剩余问题并升级给用户 |

**3个循环后，如果问题仍然存在：**
1. 列出每个未解决的问题，包括幻灯片编号和描述
2. 解释已尝试的操作
3. 交付文件并附上明确的免责声明："以下问题无法自动解决：[列表]。需要手动审核。"

**不要**无限期继续循环。一些问题（字体渲染、复杂形状对齐）可能需要在 PowerPoint 中手动干预。

### 阶段5：最终验证

交付前运行 [最终质量检查表](#final-quality-checklist)。

### 幻灯片层级（典型）

这些是典型范围——根据模板规格调整：

| 层级 | 示例 | 典型大小 | 样式 |
|------|------|----------|------|
| 标题 | 幻灯片标题 | 40-48pt | 粗体 |
| 副标题 | 市场定义，幻灯片描述符 | 18-22pt | 粗体 |
| 部分标题 | "关键预测"，"评论" | 14-16pt | 常规 |
| 块标签 | "包含的细分市场"，"定义"侧边栏 | 12-14pt | 常规 |
| 块内容 | 项目符号，正文 | 11-14pt | 常规 |
| 表格标题 | 列标题 | 10-12pt | 粗体 |
| 表格正文 | 单元格内容 | 9-11pt | 常规 |
| 脚注 | 来源，注释 | 8-9pt | 斜体 |

### 字体一致性匹配

**同一层级**的框必须使用相同的字体大小：

| 同级 | 必须与以下匹配 |
|------|----------------|
| "包含的细分市场" | "排除的细分市场" |
| "定义" | "范围理由" |
| 左列项目符号 | 右列项目符号 |
| 所有块标签 | 彼此 |
| 所有部分标题 | 彼此 |

### 演示文稿四舍五入

这些是**典型惯例** — 根据数值大小和模板样式调整：

| 值类型 | 典型四舍五入 | 示例 |
|--------|--------------|------|
| 大型市场规模($100亿+) | 最接近的$10亿 | 18.5 → $190亿 |
| 较小市场规模(<$100亿) | 最接近的$5亿或$1亿 | 2.3 → $25亿 |
| 规模范围 | 匹配来源精度 | 14.9-22.1 → $150-220亿 |
| CAGR | 整数%或0.5% | 16.4% → 16%或16.5% |
| 市场份额 | 最接近的5%或匹配来源 | 21.4% → 20% |
| 倍数 | 1位小数 | 9.69 → 9.7x |

**原则：** 四舍五入不应实质性改变数字。对于较小的值，使用更精细的精度。

### 文本密度规则

- 每个内容框最多6-7个项目符号
- 每个项目符号最多2行
- 括号示例：同一行或缩进在下方
- 无孤立单词（新行上的单个单词）

### 对齐原则

**垂直堆叠的框**必须具有相同的：
- 左边距位置、项目符号缩进、文本起始位置、框宽度

**水平相邻的框**必须具有相同的：
- 顶部位置、高度（可能的话）、内部填充

### 多幻灯片一致性

当相同数据出现在多张幻灯片上时：
- 使用相同的数字、格式和术语
- 如果一个指标在一张幻灯片上更新，更新所有出现的地方
- 验证时交叉引用以捕获不匹配

## 必须要求

无论模板如何，这些要求都是不可协商的：

| 要求 | 详情 |
|------|------|
| **文本可读性** | 所有文本必须与背景有足够的对比度。示例：在深蓝色、深绿色、黑色背景上使用白色/浅色文本；在白色、浅灰色、浅黄色背景上使用黑色/深色文本。 |
| **实际表格对象** | 表格数据必须是表格对象，而不是制表符分隔的文本。参见 [`xml-reference.md`](reference/xml-reference.md#table-implementation)。 |
| **正确的图表/表格大小** | 粘贴的视觉元素必须填充指定区域。参见 [`formatting-standards.md`](reference/formatting-standards.md#chart-and-image-handling)。 |
| **一致的格式** | 部分内的项目符号必须匹配（符号、大小、缩进）。同级框必须使用相同的字体大小。 |
| **内容边界** | 所有内容必须保持在幻灯片边缘内。脚注框宽度：16:9约32.5厘米，4:3约24厘米。 |
| **无占位符格式** | 删除彩色说明框。主体：按照模板在浅色背景上使用深色文本。 |

## 关键反模式：切勿这样做

当占位符格式被误认为是输出格式时，就会出现这些失败。识别这些模式至关重要。

### 反模式1：将数据填充到占位符框中

**发生的情况：** 模板有彩色说明框（黄色、橙色等），带有指导文本。模型用实际数据替换指导文本，但**保留彩色框**。

**为什么错误：** 彩色框是占位符。它告诉你应该放什么内容。输出应该有不同的格式 — 通常是在白色/浅色背景上使用深色文本，或适当样式的形状。

**识别测试：** 如果您填充的幻灯片有装满数据文本的大彩色矩形，那么您复制了占位符格式而不是替换它。

**关键区别 — 两种"占位符"类型：**

| 类型 | 如何识别 | 该做什么 |
|------|----------|----------|
| **说明框** | 明亮颜色（黄色、橙色），包含指导文本如"在此插入X"，彩色背景上的白色/浅色文本 | **删除**整个形状，然后用生产格式创建新内容 |
| **布局占位符** | 幻灯片母版/布局的一部分，与模板主题匹配的中性颜色，"点击添加文本" | **保留**形状，仅**替换**文本内容 |

如果不确定：检查同一模板的空幻灯片上是否存在该形状。布局占位符会保留；说明框是常规形状。

### 反模式2：基于文本的"表格"

**发生的情况：** 模型使用分隔符字符（`|`、制表符、空格）创建类似表格的内容，而不是实际的表格对象。

**为什么错误：** 这**不是**表格。列永远无法正确对齐，无法一致地格式化，看起来不专业。

**识别测试：** 如果您正在输入 `|` 字符或依赖空格/制表符来创建列，那么您正在创建文本，而不是表格。

**必须验证：** 创建任何表格后，验证它是实际的表格对象。参见 [`xml-reference.md`](reference/xml-reference.md#critical-verify-tables-are-actual-table-objects) 获取验证方法。

### 反模式3：继承占位符对比度

**发生的情况：** 占位符在彩色背景上使用浅色文本（例如白色在黄色上）。模型填充数据但保留此配色方案，导致难以阅读的输出。

**为什么错误：** 占位符颜色故意与众不同，以表示"替换我"。生产幻灯片通常在正文内容中使用浅色背景上的深色文本。

**识别测试：** 如果您填充的内容在正文区域（不是标题）的明亮彩色背景上有浅色/白色文本，那么您继承了占位符格式。

**正确方法：** 应用生产格式 — 通常正文内容在白色或浅色背景上使用深色文本（#000000 或 #333333）。标题和强调区域可以使用品牌颜色。

### 摘要：占位符 vs 生产

| 元素 | 占位符（输入） | 生产（输出） |
|------|----------------|--------------|
| 说明框 | 彩色背景，指导文本 | 删除或重新格式化 |
| 数据区域 | "[在此插入数据]"文本 | 具有清晰格式的实际数据 |
| 表格 | 描述表格应包含的内容 | 具有行/列的实际表格对象 |
| 正文 | 彩色背景上的浅色文本 | 浅色背景上的深色文本 |

**占位符告诉您要创建什么，而不是如何格式化它。**

## 错误处理

**如果PDF/图像转换失败：**
1. 检查LibreOffice是否安装：`which soffice`
2. 尝试替代方法：`libreoffice --headless --convert-to pdf presentation.pptx`
3. 如果仍然失败，在PowerPoint/LibreOffice中手动打开并导出

**如果源数据存在不一致或冲突：**
1. **优先级顺序**：首先使用任务文件中明确提供的数据
2. 如果使用其他来源的数据（网络搜索、外部文档），向用户标记
3. 明确记录任何差异
4. 添加脚注解释数据源选择

**如果计算与来源预测不匹配：**
1. 显示您的计算方法
2. 记录差异和可能的原因（不同的基准年、方法）
3. 如果差异很大，呈现两个值
4. 向用户标记以解决

## 表格结构指南

创建表格时（必须是实际表格对象）：

**列对齐：**
- 文本列：左对齐（标题和内容）
- 数字列：右对齐或居中对齐（标题与内容匹配）

**标题行：**
- 粗体文本
- 阴影背景（模板的品牌颜色）
- 白色或对比色文本

**共识/总计行：**
- 粗体文本
- 上方有分隔线
- 明显的背景阴影

**宽度：** 完全填充指定部分的宽度。

有关XML实现，请参见 [`xml-reference.md`](reference/xml-reference.md#table-implementation)。

## 脚注格式

**格式：**
```
来源：[来源1]（年份），[来源2]（年份）。
注释：(1) [第一条注释]；(2) [第二条注释]。
```

**示例：**
```
来源：Grand View Research（2024），Mordor Intelligence（2024），Markets and Markets（2023）。
注释：(1) 不包括硬件收入；(2) 包括B2B和B2C细分市场。
```

幻灯片正文中的所有上标数字（¹, ², ³）必须有相应的注释条目。

## 金融术语解释

### 1. 演示文稿（Pitch Deck）

**定义**：演示文稿是创业公司或企业用于向投资者展示商业计划和融资需求的演示文件。

**组成部分**：
- **执行摘要**：公司概述和融资需求
- **市场分析**：市场规模、增长趋势、竞争格局
- **产品/服务**：核心产品介绍
- **商业模式**：盈利模式和收入来源
- **财务预测**：收入预测、利润预期
- **团队介绍**：核心团队背景

**用途**：
- 吸引风险投资
- 展示商业机会
- 路演和投资者会议

### 2. TAM（总体可寻址市场）

**定义**：TAM（Total Addressable Market）是指公司产品或服务可以服务的最大市场规模。

**计算方法**：
- 基于市场规模和目标客户数量
- 考虑产品定价和市场渗透率

**重要性**：
- 评估市场机会大小
- 支持商业计划和融资
- 帮助战略规划

### 3. CAGR（复合年增长率）

**定义**：CAGR（Compound Annual Growth Rate）是衡量投资或指标在特定时期内平均年增长率的指标。

**计算公式**：
- CAGR = (终值 ÷ 初值)^(1/年数) - 1

**特点**：
- 假设增长率是恒定的
- 平滑了期间的波动
- 便于比较不同时期的增长率

### 4. 市场份额（Market Share）

**定义**：市场份额是指公司在特定市场中的销售额占整个市场销售额的百分比。

**计算公式**：
- 市场份额 = (公司收入 / 市场总收入) × 100%

**重要性**：
- 反映竞争地位
- 评估增长机会
- 影响估值

### 5. 估值倍数（Valuation Multiples）

**定义**：估值倍数是用于评估公司价值的比率，通常基于财务指标。

**常用倍数**：
- **EV/Revenue**：企业价值与收入比
- **EV/EBITDA**：企业价值与EBITDA比
- **P/E**：市盈率

**应用场景**：
- 相对估值
- 投资决策
- 并购定价

### 6. 企业价值（Enterprise Value, EV）

**定义**：企业价值是衡量公司总价值的指标，考虑了股权和债务。

**计算公式**：
- EV = 市值 + 债务 - 现金及等价物

**组成部分**：
- **市值**：普通股的市场价值
- **债务**：短期和长期债务
- **现金**：现金及现金等价物

**用途**：
- 并购估值
- 跨公司比较
- 财务建模

### 7. 同比增长率（YoY Growth）

**定义**：同比增长率是指与上一年相比的增长百分比。

**计算公式**：
- YoY增长率 = (本年值 - 上年值) / 上年值 × 100%

**优点**：
- 直观易懂
- 便于年度比较
- 反映季节性影响

### 8. 共识估计（Consensus Estimates）

**定义**：共识估计是多位分析师对财务指标的平均预测值。

**数据来源**：
- 卖方分析师报告
- 金融数据提供商
- 专业财经媒体

**类型**：
- 当前季度估计
- 年度估计
- 长期增长估计

### 9. 竞争格局（Competitive Landscape）

**定义**：竞争格局是指市场中竞争对手的分布和相对地位。

**分析维度**：
- 市场份额分布
- 竞争对手优势和劣势
- 市场进入壁垒
- 竞争策略

**工具**：
- SWOT分析
- 波特五力模型
- 竞争矩阵

### 10. 财务预测（Financial Projections）

**定义**：财务预测是对未来财务表现的估计，通常包括收入、利润和现金流。

**组成部分**：
- 收入预测
- 成本预测
- 利润预测
- 现金流预测

**方法**：
- 自上而下方法
- 自下而上方法
- 基于市场的方法

**用途**：
- 商业计划
- 融资申请
- 战略规划

## Appendix: 金融背景知识

这份文件是"推介材料模板填充（Pitch Deck Template Fill）"技能的详细说明。在投资银行中，Pitch Deck 是最常使用的客户材料——它把一个交易机会呈现给潜在客户，说服他们"选我们做这个交易"。

---

### 1. 什么是 Pitch Deck 填充？

**类比：**
想象你是一个"请柬设计师"。你已经有了精美的请柬模板（排版、配色、字体都已设计好），但每封请柬需要填不同的信息：新郎新娘名字、日期、地点。

Pitch Deck 填充就是投行的"请柬模板填充"——模板（设计、品牌、结构）是固定的，你只需要把"数字"和"名字"填进去。

---

### 2. Pitch Deck 的"标准结构"

| 页码 | 幻灯片 | 内容 |
|------|--------|------|
| 1 | 封面 | 投行 Logo + 客户名 + 日期 |
| 2 | 目录 | 本次 Pitch 的结构 |
| 3 | 执行摘要 | 为什么选我们 |
| 4 | 团队介绍 | 你将会和谁共事 |
| 5 | 行业概览 | 市场有多大、增长多快 |
| 6 | 公司概览 | 客户公司简介 |
| 7 | 估值分析 | DCF + Comps + Football Field |
| 8 | 交易方案 | 我们建议的价格范围和条款 |
| 9 | 时间表 | 交易流程和时间节点 |
| 10 | 附录 | 数据来源、免责声明 |

---

### 3. 模板填充的"三大参考文件"

| 参考文件 | 约束内容 |
|---------|---------|
| formatting-standards.md | 字体、字号、颜色、对齐、数字格式 |
| slide-templates.md | 每类幻灯片应该放什么内容 |
| calculation-standards.md | 财务数据的计算公式（CAGR、共识等） |
| xml-reference.md | 底层 XML 结构（表格、形状、箭头） |

**关键：** 不要凭感觉决定格式——参考文件已经定义了最佳实践。

---

### 4. Pitch Deck 的"致命禁忌"

| 禁忌 | 原因 |
|------|------|
| 自己设计新模板 | 模板是公司品牌资产，不能随意改 |
| 改变配色方案 | 配色属于合规范围（违规后果严重） |
| 用错 Logo | 可能涉及品牌侵权或合规 |
| 数据无来源 | 每个数字都必须有可追溯的数据来源 |
| 承诺具体回报 | 违反监管规定 |

### 5. 真实案例：一个完美的 Pitch Deck

**背景**：某投行竞标一家 $500M 科技公司的卖方顾问

**成功要素**：
1. 团队介绍页展示了 3 个在科技行业有 15+ 年经验的 MD
2. 行业概览页用 3 张图表展示了行业趋势和市场地位
3. 估值分析页用 Football Field 展示了 DCF + Comps + Precedent 三种方法
4. 交易方案页清晰列出了"推荐方案"和"备选方案"

**结果**：客户选择了这家投行，交易顺利交割。

### 给小白的一句话

> Pitch Deck 就是投行界的"竞标PPT"——客户要选投行做交易（比如上市、并购），各家投行必须用一份 PPT 来证明"我最懂你的行业、我最会卖好价钱、我有最强的团队"。模板是神圣的：字体、颜色、Logo 一丝一毫不能乱改。你的任务是"填数字和名字"而不是"搞设计"。如果 MD 发现你私自把蓝色 Logo 改成了绿色，后果可能比你算错估值还严重——因为在投行，"格式即专业"。
