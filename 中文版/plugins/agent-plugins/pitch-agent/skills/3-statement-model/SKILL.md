# 三表财务模型模板填写

用利润表、资产负债表和现金流量表之间适当的链接，完整填充综合财务模型模板。

## ⚠️ 关键原则 — 填写任何模板前必读

**环境 — Office JS vs Python：**
- **如果在Excel内运行（Office加载项/Office JS）：** 直接使用Office JS。通过`range.formulas = [["=D14*(1+Assumptions!$B$5)"]]`写入公式 — 永远不要对派生单元格使用`range.values`。无需单独重新计算；Excel原生计算。使用`context.workbook.worksheets.getItem(...)`导航工作表。
- **如果生成独立.xlsx文件：** 使用Python/openpyxl。写入`ws["D15"] = "=D14*(1+Assumptions!$B$5)"`，然后在交付前运行`recalc.py`。
- **Office JS合并单元格陷阱：** 不要在合并单元格后对合并范围调用`.merge()`然后在`.values`上设置值 — 会抛出`InvalidArgument`，因为该范围仍报告其合并前的维度。相反，只向左上角单元格写入值，然后合并并格式化完整范围：`ws.getRange("A1").values = [["损益表"]]; const h = ws.getRange("A1:G1"); h.merge(); h.format.fill.color = "#1F4E79";`
- 以下所有原则在两种环境中均适用。

**公式优先于硬编码（不可协商）：**
- 每个预测单元格、滚动结转、链接和小计都必须是Excel公式 — 永远不要使用预计算值
- 使用Python/openpyxl时：写入公式字符串（`ws["D15"] = "=D14*(1+Assumptions!$B$5)"`），而不是计算结果（`ws["D15"] = 12500`）
- 唯一应包含硬编码数字的单元格是：（1）历史实际数据，（2）假设选项卡中的假设驱动因素
- 如果发现自己用Python计算值然后写入单元格 — 停止。改为写入公式。
- 原因：模型必须在场景切换或假设更改时灵活变动。硬编码会悄悄破坏每个下游完整性检查。

**逐步与用户验证：**
1. **映射模板后** → 向用户展示已识别的工作表/部分，并在触碰任何单元格前确认
2. **填充历史数据后** → 向用户展示历史数据块并确认数值/期间与源数据匹配
3. **构建IS预测后** → 运行小计检查，向用户展示预测IS，在移至BS前确认
4. **构建BS后** → 向用户展示每个期间的余额检查（资产=负债+权益），在移至CF前确认
5. **构建CF后** → 向用户展示现金核对（CF期末现金=BS现金），在最终确定前确认
6. **不要端到端填充整个模型然后完整呈现** — 在每个报表处停顿，展示工作，提前发现错误

## 格式设置 — 专业蓝灰色调（除非模板/用户另有指定）

**保持颜色最少。** 仅对单元格填充使用蓝色和灰色。不要引入绿色、黄色、橙色或多种强调色 — 简洁的模型使用克制。

| 元素 | 填充 | 字体 |
|---|---|---|
| 部分标题（IS/BS/CF标题） | 深蓝色`#1F4E79` | 白色粗体 |
| 列标题（FY2024A、FY2025E等） | 浅蓝色`#D9E1F2` | 黑色粗体 |
| 输入单元格（历史数据、假设驱动因素） | 浅灰色`#F2F2F2`或白色 | 蓝色`#0000FF` |
| 公式单元格 | 白色 | 黑色 |
| 跨表链接 | 白色 | 绿色`#008000` |
| 检查行/关键合计 | 中蓝色`#BDD7EE` | 黑色粗体 |

**只需3种蓝色+1种灰色+白色。** 如果模板有自己的配色方案，请遵循模板。

字体颜色表示单元格的*类型*（输入/公式/链接）。填充颜色表示*位置*（标题/数据/检查）。

## 模型结构

### 识别模板工作表组织

模板的工作表命名约定和组织方式各不相同。填充前，审查所有工作表以了解模板结构。以下是常见的工作表名称及其典型内容：

| 常见工作表名称 | 内容 |
|----------------|------|
| IS、P&L、损益表 | 利润表 |
| BS、资产负债表 | 资产负债表 |
| CF、CFS、现金流量 | 现金流量表 |
| WC、营运资本 | 营运资本明细表 |
| DA、D&A、折旧、PP&E | 折旧与摊销明细表 |
| Debt、债务计划 | 债务明细表 |
| NOL、税务、DTA | 经营净亏损结转明细表 |
| Assumptions、Inputs、Drivers | 假设输入和驱动因素 |
| Checks、Audit、Validation | 错误检查仪表板 |

**模板审查清单**
- 识别模板中存在哪些工作表（并非所有模板都包含每个明细表）
- 注意任何未列出的特定于模板的工作表
- 了解工作表依赖关系（例如，哪些明细表流向主报表）
- 定位每个工作表上的输入单元格与公式单元格

### 理解模板结构

填充模板前，熟悉其现有布局以确保数据输入到正确位置且公式保持完整。

**识别行结构**
- 在每个工作表顶部找到模型标题
- 识别部分标题及其视觉分隔
- 找到表示百万美元、%、x等的单位行
- 注意区分实际数据与估计期间的列标题
- 确认期间标签（例如，FY2024A、FY2025E）
- 识别输入单元格与公式单元格（通常通过字体颜色区分）

**识别列结构**
- 确认最左列中的行项目标签
- 验证历史年份先于预测年份
- 注意分隔历史期间和预测期间的视觉边框
- 检查所有工作表间列顺序的一致性

**使用命名范围**
模板通常使用命名范围表示关键输入和输出。输入数据前：
- 审查模板中的现有命名范围（Excel中的公式 → 名称管理器）
- 常见命名范围包括：收入增长率、成本百分比、关键输出（净利润、EBITDA、总债务、现金）、场景选择器单元格
- 确保在流入这些命名范围的单元格中输入

### 预测期间
- 模板通常从最后一个历史年份开始预测5年
- 验证历史（A）与预测（E）列清晰分隔
- 确认列使用财年 notation（例如，FY2024A、FY2025E）

## 利润率分析

**注意：以下利润率分析仅在用户提示或模板明确要求时才执行。如果没有提示，请跳过此部分。**

在损益表（IS）工作表上计算并显示盈利能力利润率，以跟踪运营效率并 enable peer comparison。

### 核心利润率

| 利润率 | 公式 | 衡量内容 |
|--------|------|---------|
| 毛利率 | 毛利润/收入 | 定价能力、生产效率 |
| EBITDA利润率 | EBITDA/收入 | 核心运营盈利能力 |
| EBIT利润率 | EBIT/收入 | 折旧摊销后运营盈利能力 |
| 净利润率 | 净利润/收入 | 底线盈利能力 |

### 带利润率的损益表布局

在每个利润项目下方直接显示利润率百分比：
- 毛利润%在毛利润下方
- EBIT利润率%在EBIT下方
- EBITDA利润率%在EBITDA下方
- 净利润率%在净利润下方

## 信用指标

**注意：以下信用分析仅在用户提示或模板明确要求时才执行。如果没有提示，请跳过此部分。**

在资产负债表（BS）工作表上计算并显示信用/杠杆指标，以评估财务健康状况、债务能力和契约合规性。

### 核心信用指标

| 指标 | 公式 | 衡量内容 |
|--------|------|---------|
| 总债务/EBITDA | 总债务/EBITDA | 杠杆倍数 |
| 净债务/EBITDA | （总债务-现金）/EBITDA | 现金净额杠杆 |
| 利息保障倍数 | EBITDA/利息费用 | 偿债能力 |
| 债务/总资本 | 总债务/（总债务+权益） | 资本结构 |
| 债务/权益 | 总债务/总权益 | 财务杠杆 |
| 流动比率 | 流动资产/流动负债 | 短期流动性 |
| 速动比率 | （流动资产-存货）/流动负债 | 即时流动性 |

### 信用指标层次检查

验证Upside显示最强的信用状况：
- 杠杆：Upside < Base < Downside（越低越好）
- 覆盖：Upside > Base > Downside（越高越好）
- 流动性：Upside > Base > Downside（越高越好）

### 契约合规性跟踪

如果已知债务契约，添加将实际指标与契约阈值进行比较的明确合规检查。

## 场景分析（基础/上浮/下浮）

在假设选项卡中使用场景切换（下拉菜单）和CHOOSE或INDEX/MATCH公式。

| 场景 | 描述 |
|------|------|
| 基础情景 | 管理层指导或共识预期 |
| 上浮情景 | 高于指导的增长，利润率扩张 |
| 下浮情景 | 低于趋势增长，利润率压缩 |

**需要敏感化的关键驱动因素**：收入增长、毛利率、SG&A%、DSO/DIO/DPO、CapEx%、利率、税率。

**场景审计检查**：切换开关所有报表，所有场景中BS平衡，现金核对，层次保持（对于NI、EBITDA、FCF、利润率：Upside > Base > Downside）。

## 完成模型模板

本节提供完成任何三表财务模型模板的一般指导，同时保留现有公式并确保数据完整性。

### 第1步：分析模板结构

输入任何数据前，彻底审查模板以了解其架构：

**识别输入与公式单元格**
- 寻找区分输入单元格与公式单元格的视觉提示（字体颜色、单元格阴影）
- 常见约定：蓝色字体=输入，黑色字体=公式，绿色字体=链接到其他工作表
- 使用Excel的追踪从属/引用（公式 → 追踪引用）了解单元格关系
- 检查可能控制关键输入的命名范围（公式 → 名称管理器）

**映射模板流程**
- 识别哪些工作表流向其他工作表（例如，Assumptions → IS → BS → CF）
- 注意任何支持明细表及其与主报表的链接
- 在填充前记录模板的特定行项目和数据结构

### 第2步：填写数据而不破坏公式

**数据输入黄金法则**

| 规则 | 描述 |
|------|------|
| 仅编辑输入单元格 | 永远不要覆盖包含公式的单元格，除非有意替换公式 |
| 保留单元格引用 | 复制数据时，使用粘贴值（Ctrl+Shift+V）以避免用源格式覆盖公式 |
| 匹配模板单位 | 输入数据前验证模板使用千、百万还是实际值 |
| 遵守符号惯例 | 遵循模板现有符号约定（费用为正还是负） |
| 检查循环引用 | 如果模板使用迭代计算，确保开启"启用迭代计算" |

**安全数据输入流程**
1. 识别指定用于输入的确切单元格（通常高亮或标记）
2. 首先输入历史数据，然后验证这些期间的公式计算正确
3. 输入流向预测计算的假设驱动因素
4. 审查计算输出以确认公式按预期工作
5. 如果必须修改公式单元格，在更改前记录原始公式

**处理预建公式**
- 如果公式引用尚未填充的单元格，在所有输入完成前会出现临时错误（#REF!、#DIV/0!）
- 当公式产生意外结果时，追踪引用以识别缺失或错误的输入
- 在检查所有工作表的公式依赖项之前不要删除行/列

### 第3步：验证公式

**公式完整性检查**

依赖模板输出前，验证公式是否正确运作：

| 检查类型 | 方法 |
|------------|------|
| 追踪引用 | 选择公式单元格 → 公式 → 追踪引用以验证它引用正确的输入 |
| 追踪从属 | 验证关键输入流向预期的输出单元格 |
| 评估公式 | 使用公式 → 评估公式逐步执行复杂计算 |
| 检查硬编码 | 预测公式应引用假设，而不是包含硬编码值 |
| 用已知值测试 | 输入简单测试值以验证公式产生预期结果 |
| 跨表一致性 | 确保相同的公式逻辑适用于所有预测期间 |

**需要注意的常见公式问题**
- 混合绝对/相对引用在跨期间复制时导致错误结果
- 链接到外部文件或已删除范围的断开链接（#REF!错误）
- 早期期间收入尚未增长时的除以零（#DIV/0!错误）
- 循环引用警告（可能是有意的，用于利息计算）
- 跨预测列的公式不一致（使用Ctrl+\查找差异）

**验证跨表链接**
- 确认出现在多个工作表上的值是链接的（不是重复的）
- 验证明细表合计与主报表上的相应行项目一致
- 检查期间标签在所有工作表上对齐

### 第4步：按工作表进行质量检查

填充模板后在每个工作表上执行这些验证检查：

**损益表（IS）质量检查**
- 历史期间收入数据与源数据匹配
- 所有费用项目汇总到报告合计
- 小计（毛利润、EBIT、EBT、净利润）计算正确
- 税务计算逻辑适当（正确处理亏损）
- 预测驱动因素引用假设选项卡（无硬编码）
- 期间变化方向合理

**资产负债表（BS）质量检查**
- 每个期间的资产=负债+权益（主要检查）
- 现金余额与现金流量表期末现金匹配
- 营运资本账户与支持明细表一致（如适用）
- 留存收益正确结转：期初RE + 净利润 - 股息 +/- 调整 = 期末RE
- 债务余额与债务明细表一致（如适用）
- 所有资产负债表项目具有适当的符号（资产为正，大多数负债为正）

**现金流量表（CF）质量检查**
- CFO顶部的净利润与损益表净利润匹配
- 非现金加回（D&A、SBC等）与其源明细表/报表一致
- 营运资本变化具有正确符号（资产增加=现金使用=负）
- CapEx与PP&E明细表或固定资产结转一致
- 融资活动与BS上债务和权益账户的变化一致
- 期末现金与资产负债表现金匹配
- 期初现金等于上期期末现金

**支持明细表质量检查**
- 期初余额等于上期期末余额
- 结转逻辑完整（期初 + 增加 - 减少 = 期末）
- 明细表合计与主报表行项目一致
- 计算中使用的假设与假设选项卡匹配

### 第5步：跨报表完整性检查

验证单个工作表后，确认三表正确整合：

| 检查 | 公式 | 预期结果 |
|-------|------|---------|
| 资产负债表平衡 | 资产 - 负债 - 权益 | = 0 |
| 现金核对 | CF期末现金 - BS现金 | = 0 |
| 净利润链接 | IS净利润 - CF期初净利润 | = 0 |
| 留存收益 | 期初RE + NI - 股息 - BS期末RE | = 0（根据需要为SBC/其他项目调整） |

### 第6步：最终审查

在认为模型完成前：
- 在所有场景间切换（如适用）以验证每个场景中检查通过
- 审查并解决所有#REF!、#DIV/0!、#VALUE!和#NAME?错误或记录
- 确认所有输入单元格已填充（搜索占位符值）
- 验证所有工作表间的单位一致
- 在进行任何其他修改前保存干净版本

## 模型验证和审计

本节汇总所有验证检查和已完成模板的审计程序。

### 核心链接（必须始终成立）

请参阅[references/formulas.md](references/formulas.md)获取所有公式详情。

| 检查 | 公式 | 预期结果 |
|-------|------|---------|
| 资产负债表平衡 | 资产 - 负债 - 权益 | = 0 |
| 现金核对 | CF期末现金 - BS现金 | = 0 |
| 现金月度vs年度 | 期末现金（月度）- 期末现金（年度） | = 0 |
| 净利润链接 | IS净利润 - CF期初净利润 | = 0 |
| 留存收益 | 期初RE + NI + SBC - 股息 - BS期末RE | = 0 |
| 权益融资 | Δ普通股/APIC（BS）- 权益发行（CFF） | = 0 |
| 第0年权益 | 权益募集（第0年）- 期初权益资本（第1年） | = 0 |

### 符号惯例参考

| 报表 | 项目 | 符号惯例 |
|-----------|------|---------|
| CFO | D&A、SBC | 正（加回） |
| CFO | ΔAR（增加） | 负（现金使用） |
| CFO | ΔAP（增加） | 正（现金来源） |
| CFI | CapEx | 负 |
| CFF | 债务发行 | 正 |
| CFF | 债务偿还 | 负 |
| CFF | 股息 | 负 |

### 循环引用处理

利息费用产生循环：利息 → 净利润 → 现金 → 债务余额 → 利息

在Excel中启用迭代计算：文件 → 选项 → 公式 → 启用迭代计算。将最大迭代次数设置为100，最大更改设置为0.001。在假设选项卡中添加断路器切换。

### 检查类别

**第1节：货币一致性**
- 在假设中识别并记录货币
- 所有工作表使用一致的货币符号和比例
- 单位行与模型货币匹配

**第2节：资产负债表完整性**
- 每个期间的资产 = 负债 + 权益
- 公式：资产 - 负债 - 权益（必须 = 0）

**第3节：现金流量表完整性**
- 现金与BS核对（CF期末现金 = BS现金）
- 现金月度vs年度：期末现金（月度）= 期末现金（年度）
- NI与IS核对（CF净利润 = IS净利润）
- D&A与明细表核对
- SBC与IS核对
- ΔAR、Δ存货、ΔAP与WC明细表核对
- CapEx与DA明细表核对

**第4节：留存收益**
- RE结转检查：期初RE + NI + SBC - 股息 = 期末RE
- 显示组件分解以便调试

**第5节：营运资本**
- AR、存货、AP与BS核对
- DSO、DIO、DPO合理性检查（如果在正常范围外则标记）

**第6节：债务明细表**
- 总债务与BS核对（流动 + 长期债务）
- 利息计算与IS核对

**第6b节：权益融资**
- 权益发行收益与BS普通股/APIC增加核对
- 权益现金增加 = 权益账户增加（必须平衡）
- 权益募集核对：Δ普通股/APIC（BS）= 权益发行（CFF）（必须 = 0）
- 第0年权益核对：募集的权益（第0年）= 期初权益资本（第1年）

**第6c节：NOL明细表**
- 第1年/成立期的期初NOL = 0（新企业以零NOL开始）
- NOL仅在EBT < 0时增加（亏损必须实现才能产生NOL）
- DTA与BS核对（NOL明细表DTA = BS递延税项资产）
- NOL利用 ≤ EBT的80%（2017年后联邦限制）
- NOL余额为非负数（不能利用超过可用金额）
- NOL仅在EBT < 0时产生
- 当应税收入 ≤ 0时，税费 = 0

**第7节：场景层次**
- 绝对指标：Upside > Base > Downside（NI、EBITDA、FCF）
- 利润率：Upside > Base > Downside（GM%、EBITDA%、NI%）
- 信用指标：杠杆的Upside < Base < Downside（反向）

**第8节：公式完整性**
- COGS、S&M、G&A、R&D、SBC由收入百分比驱动（无硬编码）

## 金融术语和知识详细解释

### 1. 三表财务模型（Three-Statement Financial Model）

**定义**：三表财务模型是将利润表（Income Statement）、资产负债表（Balance Sheet）和现金流量表（Cash Flow Statement）通过数学逻辑和会计关系链接在一起的综合财务预测模型。

**核心关系**：
- **利润表**提供净利润，流入资产负债表的留存收益
- **资产负债表**提供资产、负债和权益结构
- **现金流量表**追踪现金变化，期末现金必须等于资产负债表的现金

**为什么重要**：三表模型是所有财务建模的基础，因为它确保：
- 会计等式始终成立（资产=负债+权益）
- 现金始终核对
- 各报表间的数字逻辑一致

---

### 2. 净利润（Net Income）

**定义**：净利润是企业在扣除所有费用后的最终利润，包括：
- 收入
- 销货成本（COGS）
- 运营费用（SG&A、R&D）
- 利息费用
- 税费

**计算公式**：
```
净利润 = 收入 - COGS - 运营费用 - 利息费用 - 税费
```

**在模型中的作用**：
- 流入资产负债表的留存收益
- 是现金流量表运营活动的起点
- 用于计算各种盈利能力指标（净利润率等）

---

### 3. EBITDA

**定义**：EBITDA代表"利息、税费、折旧和摊销前利润"（Earnings Before Interest, Taxes, Depreciation, and Amortization）。

**计算公式**：
```
EBITDA = 净利润 + 利息费用 + 税费 + 折旧 + 摊销
或
EBITDA = 毛利润 - 运营费用（不含D&A）
```

**为什么重要**：
- 去除资本结构（利息）和会计政策（折旧）的影响
- 更好地反映企业的核心运营盈利能力
- 常用于计算企业价值倍数（EV/EBITDA）
- 用于计算杠杆比率（债务/EBITDA）

---

### 4. 营运资本（Working Capital）

**定义**：营运资本是企业在日常运营中使用的资本，计算公式为：
```
营运资本 = 流动资产 - 流动负债
```

**组成部分**：
- **应收账款（AR）**：客户欠企业的钱
- **存货（Inventory）**：待销售的商品
- **应付账款（AP）**：企业欠供应商的钱

**关键周转天数**：
| 指标 | 公式 | 含义 |
|------|------|------|
| DSO（应收账款周转天数） | (AR/收入)×365 | 收回客户欠款所需天数 |
| DIO（存货周转天数） | (存货/COGS)×365 | 销售存货所需天数 |
| DPO（应付账款周转天数） | (AP/COGS)×365 | 支付供应商所需天数 |

**现金循环**：企业希望DSO和DIO尽可能短，DPO尽可能长，以优化现金效率。

---

### 5. 自由现金流（Free Cash Flow, FCF）

**定义**：自由现金流是企业在满足所有运营需求和资本支出后可自由支配的现金。

**计算公式**：
```
FCF = 经营现金流（CFO）- 资本支出（CapEx）
```

**经营现金流（CFO）的构成**：
```
CFO = 净利润 + 非现金费用（D&A、SBC）- Δ营运资本
```

**为什么重要**：
- 衡量企业实际产生的现金
- 用于企业估值（DCF模型的输入）
- 反映企业偿债和分红能力

---

### 6. 资本支出（CapEx vs. OpEx）

**资本支出（CapEx）**：用于购买或维护长期资产（如房产、设备）的支出
- 记录在现金流量表的"投资活动"
- 影响固定资产（PP&E）
- 折旧作为费用逐步扣除

**运营支出（OpEx）**：用于日常运营的支出（如工资、租金）
- 记录在利润表
- 当期费用化

**为什么重要**：
- CapEx前期投入大但税盾效果长期
- OpEx当期扣除但无资产增值
- 投资者关注两者比例以评估企业特性

---

### 7. 留存收益（Retained Earnings）

**定义**：留存收益是企业历年来累积的净利润中未作为股息分配的部分。

**滚动公式**：
```
期末留存收益 = 期初留存收益 + 净利润 + SBC - 股息
```

**在资产负债表上的位置**：
- 属于股东权益部分
- 体现企业历史盈利积累
- 可用于再投资或偿还债务

---

### 8. 企业价值倍数（EV/EBITDA）

**企业价值（Enterprise Value, EV）**：
```
EV = 市值 + 净债务
   = 普通股市值 + 优先股 + 少数股东权益 + 总债务 - 现金
```

**EV/EBITDA倍数**：
- 用于比较不同规模和资本结构的公司
- 排除折旧摊销会计政策差异
- 数值越低表示相对便宜

**为什么重要**：
- 投行估值的基础倍数
- 用于计算可比公司交易
- LBO模型中决定可承受的债务水平

---

### 9. 杠杆比率（Leverage Ratios）

**主要杠杆比率**：

| 比率 | 公式 | 解读 |
|------|------|------|
| 债务/EBITDA | 总债务/EBITDA | 杠杆倍数，越高风险越大 |
| 净债务/EBITDA | (总债务-现金)/EBITDA | 去除现金影响 |
| 利息保障倍数 | EBITDA/利息费用 | 偿债能力 |

**为什么重要**：
- 评估企业财务风险
- 贷款人设定契约的基础
- 影响信用评级
- LBO分析核心指标

---

### 10. 场景分析（Scenario Analysis）

**定义**：场景分析是改变模型关键假设以评估不同结果的过程。

**三种场景**：
| 场景 | 描述 | 典型特征 |
|------|------|---------|
| 基础情景 | 最可能的结果 | 管理层指引或共识预期 |
| 上浮情景 | 乐观结果 | 高增长、高利润率 |
| 下浮情景 | 悲观结果 | 低增长、利润率压缩 |

**层次关系（必须满足）**：
- 收入：Upside > Base > Downside
- 利润率：Upside > Base > Downside
- 杠杆：Upside < Base < Downside

**为什么重要**：
- 帮助理解关键驱动因素敏感性
- 风险评估
- 投资决策参考

## Appendix: 金融背景知识

这份文件是"三报表财务模型（3-Statement Model）"技能的详细说明。在投资银行和股权研究中，三表模型是所有财务分析的基础——它把公司的利润表、资产负债表、现金流量表整合成一个自洽的预测模型。

---

### 1. 什么是三表模型？

**类比：**
想象你是一个家庭的"CFO"。你需要记录三本账：
1. **收入支出账（利润表）**：每月赚多少、花多少、剩多少
2. **家产账（资产负债表）**：有多少存款、房子值多少、欠银行多少
3. **现金流水账（现金流量表）**：钱从哪来、花到哪去、现金多了还是少了

三本账必须能对上：收入支出账上的"利润"减去"花了买了东西的钱"= 现金流水账上的"现金变动"。现金变动加上期初现金 = 家产账上的"期末现金"。

三表模型就是把这三张表变成一个互相咬合的整体。

---

### 2. 三张表的"内容速记"

| 表 | 核心公式 | 回答的问题 |
|----|---------|-----------|
| 利润表 | 收入 - 成本 = 净利润 | 公司赚多少钱？ |
| 资产负债表 | 资产 = 负债 + 所有者权益 | 公司有什么、欠什么？ |
| 现金流量表 | 经营现金 + 投资现金 + 筹资现金 = 现金变动 | 钱跑哪去了？ |

---

### 3. 三张表的"勾稽关系"

```
利润表末尾的"净利润"
  ↓
加到资产负债表的"留存收益"
  ↓
现金流量表从"净利润"出发
  ↓ 调整非现金项目（折旧、摊销）
  ↓ 调整营运资本变动
  ↓ 减去资本支出
= 自由现金流（FCF）
  ↓
FCF 加上期初现金 = 资产负债表期末现金
```

**关键：** 如果资产负债表不平（资产 ≠ 负债 + 权益），说明模型有错误。

---

### 4. 三表建模的"五大铁律"

| 铁律 | 说明 |
|------|------|
| 所有公式可追溯 | 每个数字都必须能追溯回源数据——不能用"硬编码" |
| 历史数据不改 | 过去真实数据不能动，预测才有假设 |
| 假设集中管理 | 所有假设在一个 Sheet 里统一管理 |
| 一致性检查 | 收入预测的增速和费用预测的逻辑要匹配 |
| 资产负债表自动配平 | 设计好"配平项"（通常用"循环贷款"或"超额现金"） |

---

### 5. 真实案例：三表模型中一个常见的"陷阱"

**场景**：分析师预测公司收入增长 20%，同时假设应收账款（AR）增长 10%。

**问题**：如果收入增长 20%，应收账款的增长应该差不多——因为更多销售意味着更多"卖了但没收到的钱"。如果 AR 只增长 10%，意味着公司收款能力神奇地提升了。

**修正**：AR 增长率应该与收入增长率一致，除非有特殊原因（比如公司改进了收款政策）。

---

### 给小白的一句话

> 三表模型就是财务的"三脚架"——利润表、资产负债表、现金流量表任何一张倒了，另外两张也站不住。这三张表是互相"咬合"的：利润表里的净利润（赚了多少钱）不等于现金流水账里的现金（因为赚的钱可能还在客户那里没收到），两者的差异体现在资产负债表里（应收账款增加）。一个好的三表模型就像一个精密的齿轮系统——转动一个齿轮（修改收入增长率），所有其他齿轮（费用、利润、现金流、资产负债）都会自动跟着转，并且始终保持"平衡"。如果平衡被打破，说明有 Bug。
