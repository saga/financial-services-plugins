# 现金流量折现 (DCF) 模型排障指南

**适用场景：** 当 recalc.py 显示错误、估值结果不合理或情景选择器（Case Selector）无法正常工作时，请参考本指南。

## 模型返回错误值

### #REF! 错误
- 通常由于插入表头后公式引用的行号发生偏移导致。
- 解决方案：根据正确的行引用重建公式，或依照布局规划重新开始。
- 预防措施：在编写公式**之前**，先确定所有行号的位置。

### #DIV/0! 错误
- 除数为零或引用了空单元格。
- 解决方案：添加 IF 语句处理零值：`=IF([分母]=0,0,[分子]/[分母])`

### #VALUE! 错误
- 计算中使用了错误的数据类型（例如将文本误当作数字）。
- 解决方案：核实所有输入项均已格式化为数值。

## 估值结果不合理

### 隐含股价显著虚高
- 检查终值（Terminal Value）在企业价值（EV）中的占比是否超过 80%。
- 核实永续增长率（Terminal Growth）是否小于加权平均资本成本（WACC）。
- 审查增长假设是否符合实际。
- 考虑利润率（Margins）是否过于乐观。

### 隐含股价显著偏低
- 核实净债务（Net Debt）与净现金（Net Cash）的计算处理是否正确。
- 检查 WACC 是否设置过高。
- 审查财务预测是否过于保守。
- 考虑永续增长率是否设定过低。

## 情景选择器失效

### 切换情景时汇总列未同步更新
- 核实情景选择器单元格中的数值是否为 1、2 或 3。
- 检查 INDEX/OFFSET 函数引用的行范围（Row Range）及选择器单元格是否正确。
- 确保在引用选择器单元格时使用了绝对引用（如 $B$6）。
- 测试方法：手动更改选择器单元格的数值，观察预测值是否随之更新。

> **💡 Appendix: 领域知识小贴士**
>
> **什么是 DCF 模型？**
> 想象你有一只会下金蛋的鹅。DCF（现金流量折现模型）就是计算这只鹅未来一生能下多少蛋，并将这些还没见着的蛋，按照一定的“折扣”换算成现在的价值。简单来说，就是通过预测公司未来的赚钱能力，来决定公司现在的身价。
>
> **WACC 是什么“成本”？**
> 公司想要运转，得找银行借钱（债）或者找股东投资（股）。大家肯定不是白帮忙，银行要利息，股东要分红。这就是公司使用资金的“成本”。在 DCF 里，它是折算未来价值的那个“折扣率”。
>
> **终值（Terminal Value）为什么重要？**
> 我们很难一眼看到永远，所以通常只详细预测未来 5-10 年的表现。但公司在 10 年之后还会继续经营，“终值”就是把在那之后的无限未来统一估算成一个数。如果这个数占了总身价的一大半，说明你的估值非常依赖对遥远未来的臆测，这可是个风险信号。

## Appendix: 金融背景知识

这份文件是"DCF 模型排障指南（Troubleshooting Guide）"的详细说明。DCF 模型是投行最核心的估值工具，但也是最容易出错的——一个公式错误可能导致估值偏差几亿美元。

---

### 1. DCF 排障的"三不原则"

| 原则 | 说明 |
|------|------|
| 不假设 Excel 没错 | Excel 的绿色不一定对——公式可能引用了错误行 |
| 不跳过 sanity check | 算完 DCF 后，用 Comps 交叉验证 |
| 不对客户暴露错误 | 所有模型出去之前必须审计 |

---

### 2. 最常见的 DCF 错误

| 错误 | 后果 |
|------|------|
| #REF! | 行号偏移，公式引用丢失 |
| #DIV/0! | 分母为零——加 IFERROR 保护 |
| WACC 算错 | 估值可能翻倍/腰斩 |
| 终端价值用错方法 | 用了 Perpetuity 但不知道该用 Exit Multiple |
| 半期折现忘用 | 现金流折现时间不对 |

---

### 3. Sanity Check

排出所有技术错误后，还要做 sanity check：
- 隐含 P/E 倍数是否在 10-30x 之间？
- 隐含 EV/EBITDA 是否合理？
- 如果算出来一个 $1000 亿的公司值 $50 亿——大概率是你的模型错了

---

### 给小白的一句话

> DCF 排障指南就是投行的"急救手册"——当你的 DCF 模型跑出奇怪数字（公司值 -$500 万或 $10000 亿），不要慌，按顺序排查：#REF!（行号偏移）、#DIV/0!（分母为零）、WACC 算错（最常见）、半期折现忘用、终值方法选错。修完后必须做 sanity check——如果算出来的 P/E 只有 2x 或者 200x，那大概率不是你发现了"宝藏"，而是你的模型有个 bug。
