# 追溯差异根本原因

给定单个差异行（键、总账值、子分类账值、类别、可能原因），追溯到源并生成根本原因说明。

## 追溯路径

1. **提取总账方** — 通过internal-gl MCP，获取生成此总账行的日记账分录或过账：分录ID、过账日期、源系统、批次ID、编制人。
2. **提取子分类账方** — 通过子分类账MCP，获取匹配的交易：交易ID、交易/结算日期、交易对手、源馈送、使用的汇率。
3. **比较属性** — 对齐过账日期、汇率/日期、账户映射、数量符号、金额符号。不同的属性通常就是原因。

## 原因 → 说明

将根本原因写为一句话，格式为 **"⟨哪一方⟩ ⟨做了什么⟩ 因为 ⟨原因⟩"**，例如：

- "总账在结算日(T+2)过账，而子分类账在交易日过账 — 时间差异，将于2026-05-07结清。"
- "子分类账使用WM/R下午4点汇率；总账使用Bloomberg收盘价 — 基准金额上的外汇差异为12个基点。"
- "证券ABC123在映射表中映射到总账账户11420，但子分类账馈送了11410 — 映射差异，提交给参考数据团队。"
- "子分类账过账了两次交易（交易ID 88412和88419是重复的） — 重复过账，抑制88419。"

## 输出

对于每个追溯的差异，返回：

```json
{
  "key": "...",
  "root_cause": "如上所述的一句话",
  "owner": "ops | reference-data | accounting | upstream-system",
  "expected_clear_date": "YYYY-MM-DD 或 null",
  "action": "monitor | adjust | raise-ticket | suppress"
}
```

只有解决者才能写入调整 — 此技能仅诊断，不过账。

## 金融术语和知识解释

### 1. 根本原因分析详解

**定义**：根本原因分析是识别问题根本原因的系统方法。

**分析方法**：
| 方法 | 描述 | 应用场景 |
|------|------|----------|
| **5个为什么** | 连续提问"为什么" | 简单问题 |
| **鱼骨图** | 从多维度分析 | 复杂问题 |
| **故障树分析** | 逻辑演绎 | 系统性问题 |
| **帕累托分析** | 80/20原则 | 优先处理 |

**分析步骤**：
```
1. 定义问题 → 清晰描述差异
2. 收集证据 → 获取交易记录、系统日志
3. 分析原因 → 分类差异类型
4. 验证假设 → 测试根本原因
5. 制定措施 → 预防再次发生
6. 跟踪改进 → 验证措施有效性
```

**输出格式**：
```json
{
  "问题": "描述差异",
  "根本原因": "根本原因说明",
  "责任人": "负责部门/人员",
  "解决方案": "解决措施",
  "预期解决日期": "YYYY-MM-DD"
}
```

### 2. 审计线索

**定义**：审计线索是追踪交易从原始凭证到最终报告的完整路径。

**线索要素**：
| 要素 | 内容 |
|------|------|
| **交易ID** | 唯一标识符 |
| **时间戳** | 记录时间 |
| **操作人员** | 谁执行的操作 |
| **来源系统** | 数据来源 |
| **变更历史** | 所有修改记录 |

**追溯路径**：
```
原始凭证 → 交易系统 → 清算系统 → 子分类账 → 总账 → 财务报表
```

**重要性**：
- 支持审计验证
- 追踪错误来源
- 确保数据完整性
- 符合法规要求

### 3. T+2结算详解

**定义**：T+2结算是指交易发生后第二个工作日完成清算和交割。

**结算周期演变**：
| 时期 | 周期 | 说明 |
|------|------|------|
| 传统 | T+3 | 三天结算 |
| 2017年起 | T+2 | 缩短至两天 |
| 未来 | T+1 | 正在推进 |

**结算流程**：
```
T日：交易执行
├── 买方：支付指令
└── 卖方：交付证券

T+1日：清算处理
├── 确认交易详情
└── 计算净额

T+2日：交割完成
├── 买方收到证券
└── 卖方收到资金
```

**时间差异影响**：
- 交易日过账 vs 结算日过账
- 暂时性差异（T+2内自动消除）
- 需要跟踪未结算交易

### 4. WM/R汇率详解

**定义**：WM/R汇率是由WM公司发布的基准汇率。

**WM/R简介**：
- **发布方**：WM公司（现归LSEG/Refinitiv）
- **发布时间**：伦敦时间下午4点
- **覆盖货币**：超过150种货币对
- **应用领域**：资产估值、业绩归因

**特点**：
- **权威性**：全球广泛采用
- **透明性**：基于实际交易数据
- **稳定性**：减少估值波动

**使用场景**：
- 基金每日估值
- 投资组合业绩计算
- 财务报告折算

### 6. 基点详解

**定义**：基点是金融中常用的最小单位。

**换算关系**：
```
1基点 = 0.01% = 0.0001
100基点 = 1%

示例：
利率从3.25%上升到3.50% = 上升25基点
汇率波动12基点 = 0.12%
```

**应用场景**：
| 场景 | 使用 |
|------|------|
| 利率 | 描述利率变动 |
| 汇率 | 描述汇率波动 |
| 费用 | 描述费率差异 |
| 业绩 | 描述业绩差异 |

### 8. 重复过账

**定义**：重复过账是指同一交易被记录多次。

**原因分析**：
| 原因 | 说明 |
|------|------|
| **系统错误** | 重复触发过账 |
| **人工错误** | 重复录入 |
| **馈送重复** | 上游系统重复发送 |
| **处理故障** | 重试导致重复 |

**检测方法**：
- 按交易ID去重
- 检查时间戳和金额
- 对比历史记录

**处理方式**：
- **抑制**：标记重复项不处理
- **删除**：移除重复记录
- **追溯**：确认原始记录

---

### 9. 抑制机制

**定义**：抑制是指忽略或标记重复/错误交易。

**抑制类型**：
| 类型 | 说明 |
|------|------|
| **自动抑制** | 系统自动识别并标记 |
| **手动抑制** | 人工确认后标记 |
| **临时抑制** | 暂时忽略，后续处理 |
| **永久抑制** | 永久排除 |

**抑制流程**：
```
1. 识别重复/错误交易
2. 确认抑制原因
3. 记录抑制日志
4. 标记状态为已抑制
5. 定期复查
```

**注意事项**：
- 抑制需有记录
- 定期审查抑制列表
- 避免滥用抑制功能

### 10. 上游系统

**定义**：上游系统是提供数据给总账或子分类账的前置系统。

**常见上游系统**：
| 系统类型 | 示例 |
|----------|------|
| **交易系统** | OMS、EMS |
| **清算系统** | 中央清算所 |
| **托管系统** | 托管银行系统 |

**数据流向**：

**系统集成**：
- **API接口**：实时数据传输
- **文件传输**：批量数据交换

**问题排查**：
- 检查上游系统数据

## Appendix: 金融背景知识

这份文件是"差异追溯（Break Trace）"技能的详细说明。在基金会计中，当 GL Recon 发现总账和子账有差异后，Break Trace 负责下钻到"交易级别"——找到到底哪笔交易导致了差异。

---

### 1. 什么是 Break Trace？

**类比**：银行 App 显示余额少了 500 块。你不是只看"少了 500"——而是一笔一笔对账单：这笔 200 的转账是谁转的？这笔 50 的扣款是什么费用？这笔 250 的退款为什么没到？——找到"罪魁祸首"。

Break Trace 就是基金会计的"账单逐笔排查"——把差异追踪到具体是哪笔交易、在哪天、因为什么原因。

---

### 2. 差异追溯的"两点一线"

```
A 线：GL 端 → 这个 GL 条目来自哪个日记账分录？→ 哪个源系统？→ 哪天过账的？

B 线：子账端 → 对应的是哪笔交易？→ 哪个交易对手？→ 结算日期？→ 用的什么汇率？

交差点：比较 A 和 B → 日期不一样？汇率不一样？账户映射错了？——这就是根因。
```

---

### 给小白的一句话

> Break Trace 就是基金会计的"侦探放大镜"——GL Recon 发现总账和子账差 $5,000，你不能说"我查不出来"就完了。你要像侦探一样下钻到交易级：总账这边说苹果股票买价 $180，子账说 $178——差 $2 × 1000 股 = $2,000。剩下的 $3,000 继续查，直到每一分钱的差异都找到真实的"根因"。
