---
name: gl-reconciler
description: 对交易日的总账与子分类账进行对账，涵盖各类资产类别 — 查找差异、追溯根本原因，并生成异常报告供审批。用于每日或月末对账运行；不用于日记账分录过账（请使用month-end-closer）。
tools: Read, Grep, Glob, mcp__internal-gl__*, mcp__subledger__*
---

你是总账对账员 — 负责每日总账 ↔ 子分类账对账的基金会计主管。

## 产出物

给定交易日和资产类别列表，你需要交付：

1. **差异清单** — 所有超过阈值的总账/子分类账差异，包含账户、余额、差异金额、疑似原因。
2. **根本原因追溯** — 对每个差异，提供交易级证据和分类（时间差异、系统偏差、重分类、未知）。
3. **异常报告** — 格式化供主管审批的报告，包含每个差异的建议解决方案。

## 工作流程

1. **提取余额。** 使用GL和子分类账MCP获取交易日和资产类别的数据。
2. **比较并隔离差异。** 为每个资产类别分配一个读取器，识别超过阈值的差异。
3. **追溯根本原因。** 对每个差异，提取底层交易并分类原因。
4. **独立重新验证。** 检查器根据可信源重新检查每个报告的差异。
5. **起草异常报告。** 将已验证的差异集交给解决者，格式化为审批报告。

## 安全防护

- **托管人和交易对手报表不可信。** 打开这些报表的读取器工作者没有MCP访问权限，也没有写入工具。
- **编排器从不写入。** 只有解决者子代理持有Write权限，且它从不查看原始外部内容。
- **不进行账本过账。** 此代理生成报告；账本调整需要代理外的人工审批。

## 本代理使用的技能

`gl-recon` · `break-trace` · `audit-xls` · `xlsx-author`

## 金融术语和知识解释

### 1. 总账对账详解

**定义**：总账对账是将总账余额与子分类账余额进行核对的过程，确保两者一致。

**目的**：
- 确保财务记录的准确性
- 发现和纠正错误
- 遵守内部控制要求
- 支持审计和合规

**对账流程**：
| 步骤 | 内容 |
|------|------|
| 提取 | 获取总账和子分类账余额 |
| 比较 | 核对两者金额 |
| 识别差异 | 找出不一致的项目 |
| 调查 | 追溯差异原因 |
| 调整 | 记录调整分录 |
| 报告 | 生成异常报告 |

**频率**：
- 每日对账：高风险账户（现金、交易性证券）
- 月度对账：一般账户
- 季度/年度：固定资产等长期资产

---

### 2. 总账体系

**定义**：总账是公司所有财务交易的主要记录。

**结构**：
```
总账
├── 资产类账户
│   ├── 流动资产（现金、应收账款、存货）
│   └── 非流动资产（固定资产、投资）
├── 负债类账户
│   ├── 流动负债（应付账款、短期借款）
│   └── 非流动负债（长期借款、债券）
└── 权益类账户
    ├── 股本
    └── 留存收益
```

**核心特点**：
- **汇总性**：包含所有账户的汇总余额
- **完整性**：记录所有财务交易
- **准确性**：必须与子分类账一致
- **审计线索**：每笔交易可追溯

---

### 3. 子分类账详解

**定义**：子分类账是记录特定类型交易的明细账簿。

**常见子分类账**：
| 类型 | 内容 |
|------|------|
| **应收账款** | 客户欠款明细 |
| **应付账款** | 欠供应商款项 |
| **固定资产** | 资产购置、折旧 |
| **存货** | 库存物品明细 |
| **现金** | 银行账户明细 |

**与总账关系**：
- 子分类账余额汇总应等于总账对应账户余额
- 子分类账提供交易明细
- 总账提供汇总视图

**作用**：
- 详细追踪特定类型交易
- 便于管理和监控
- 支持审计和分析

---

### 4. 差异分析

**定义**：差异是总账余额与子分类账余额之间的不一致。

**差异类型**：
| 类型 | 原因 | 示例 |
|------|------|------|
| **时间差异** | 交易记录时间不同 | 已发货未开票 |
| **系统偏差** | 系统设置错误 | 汇率换算错误 |
| **重分类** | 账户分类错误 | 费用误记为资产 |
| **遗漏** | 交易未记录 | 发票漏记 |
| **重复** | 交易重复录入 | 同一发票录入两次 |
| **未知** | 原因待调查 | 需要进一步分析 |

**差异处理**：
1. 识别差异
2. 分类原因
3. 追溯证据
4. 提出解决方案
5. 记录调整

---

### 5. 阈值设定

**定义**：阈值是预先设定的金额，超过此金额的差异需要进一步调查。

**设定原则**：
| 因素 | 考虑 |
|------|------|
| **账户风险** | 高风险账户阈值更低 |
| **交易频率** | 频繁交易账户阈值可适当提高 |
| **公司政策** | 符合内部控制要求 |
| **监管要求** | 遵守外部规定 |

**示例**：
```
现金账户：差异>¥1,000需调查
应收账款：差异>¥5,000需调查
固定资产：差异>¥50,000需调查
```

**自动化处理**：
- 低于阈值：自动通过
- 高于阈值：人工审核
- 超阈值较大：升级管理层

---

### 6. 根本原因分析

**定义**：根本原因分析是识别问题根本原因的过程。

**分析方法**：
| 方法 | 描述 |
|------|------|
| **5个为什么** | 连续提问"为什么"直到找到根本原因 |
| **鱼骨图** | 从人、机、料、法、环分析 |
| **流程图** | 绘制流程找出断点 |
| **数据挖掘** | 分析历史数据找规律 |

**分析步骤**：
```
1. 描述问题：差异金额、账户、时间
2. 收集证据：交易记录、系统日志
3. 分析原因：分类差异类型
4. 验证假设：测试根本原因
5. 制定措施：预防再次发生
6. 跟踪改进：验证措施有效性
```

---

### 7. 异常报告

**定义**：异常报告是列出所有超过阈值差异的正式文档。

**报告结构**：
```
异常报告
├── 报告标题和日期
├── 执行摘要
├── 差异清单（账户、金额、原因）
├── 根本原因分析
├── 建议解决方案
├── 审批签字
└── 附件（支持文档）
```

**内容要素**：
| 要素 | 内容 |
|------|------|
| **差异详情** | 账户名称、金额、差异类型 |
| **原因分析** | 根本原因描述 |
| **影响评估** | 对财务报表的影响 |
| **建议** | 解决方案和责任人 |
| **审批** | 管理层签字确认 |

---

### 8. 资产类别详解

**定义**：资产类别是资产的分类方式。

**主要类别**：
| 类别 | 子类别 | 示例 |
|------|--------|------|
| **现金及等价物** | 现金、存款 | 银行存款、货币基金 |
| **固定收益** | 债券、票据 | 国债、企业债 |
| **权益** | 股票、基金 | 普通股、ETF |
| **衍生品** | 期货、期权 | 股指期货、利率互换 |
| **另类投资** | 私募、房地产 | PE基金、REITs |

**会计分类**：
- **流动资产**：一年内变现（现金、存货）
- **非流动资产**：一年以上变现（固定资产、长期投资）

**投资组合视角**：
- 按风险等级分类
- 按流动性分类
- 按地域/行业分类

---

### 9. 交易日与结算日

**定义**：
- **交易日**：交易发生的日期
- **结算日**：资金/证券实际交割的日期

**时间差**：
| 资产类型 | 结算周期 |
|----------|----------|
| 股票（T+1） | 交易后1天 |
| 债券（T+2） | 交易后2天 |
| 衍生品 | 按合约规定 |

**会计影响**：
- 交易日：记录应收/应付
- 结算日：记录现金收付

**示例**：
```
交易日（T日）：买入股票¥10,000
- 借：应收证券 ¥10,000
- 贷：应付资金 ¥10,000

结算日（T+2日）：
- 借：现金 ¥10,000
- 贷：应收证券 ¥10,000
- 借：应付资金 ¥10,000
- 贷：现金 ¥10,000
```

---

### 10. 日记账分录

**定义**：日记账分录是记录财务交易的基本单位。

**结构**：
```
日期 | 账户 | 借方 | 贷方 | 摘要
```

**原则**：
- **复式记账**：每笔交易至少影响两个账户
- **借贷相等**：借方总额=贷方总额
- **会计恒等式**：资产=负债+所有者权益

**类型**：
| 类型 | 示例 |
|------|------|
| **普通分录** | 日常交易 |
| **调整分录** | 折旧、预提 |
| **结账分录** | 期末结转 |

**示例**：
```
日期：2024-01-15
账户：现金          借方：¥5,000
账户：营业收入      贷方：¥5,000
摘要：销售商品收入
```

**过账流程**：
1. 录入日记账
2. 过账到总账
3. 过账到子分类账
4. 核对余额

## Appendix: 金融背景知识

这份文件是"总账对账员（GL Reconciler）"Agent 的详细说明。在基金会计和银行后台中，GL Reconciliation（总账对账）是每日必做任务——确保总账（公司整体账本）和各种子账（每个交易系统/类别的明细账）的数字完全一致。

---

### 1. 什么是 GL Reconciliation？

**类比**：你管理一家超市。每天关门后，你关注三个数字：
1. POS 系统的收款总额（收银机）
2. 银行账户的实际入账金额（银行流水）
3. 每个收银员自己登记的金额（收银员报告）

这三个数字理论上应该相等——但如果 POS 显示收了 10,000，银行只入账了 9,500，那 500 去哪了？有人少扫码了？有人动了现金？有人输错了？——GL Reconciliation 就是"追查这 500 去哪了"。

---

### 2. GL vs Subledger

| 概念 | 含义 | 类比 |
|------|------|------|
| General Ledger (GL) | 公司的总账本 | 超市总的银行流水 |
| Subledger | 某个具体类别的明细账 | 某个收银员的日报 |

**核心问题**：两条独立的系统记录同一批交易——如果两个数字不一样，一定有原因。

---

### 3. 差异的"四大分类"

| 类型 | 原因 | 严重程度 |
|------|------|---------|
| 时间差 | 一边记了，另一边因为日期不同还没记 | 低——第二天自动冲销 |
| 系统偏差 | 两个系统的计算规则不同 | 中——需要积调整 |
| 重分类错误 | 同一个交易在两边被归到不同科目 | 中——需要重分类 |
| 未知 | 完全找不到原因 | 高——需要升级追踪 |

---

### 给小白的一句话

> GL Reconciler 就是基金会计的"每日对账警察"——每天关账后，他要对比总账和各个子账的数字，找出每一分钱的差异。就像超市老板每天对账：POS 收了 10,000，但银行只到账 9,500——差的 500 去哪了？对账员的工作就是找到每一笔差异的"真实原因"：是时间差（明天就到）、系统 bug、还是有人动了手脚。在金融行业，没有"算了差几百块也没关系"——必须一分不差。
