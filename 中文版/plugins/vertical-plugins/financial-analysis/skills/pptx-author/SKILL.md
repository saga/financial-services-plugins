# pptx-author

当在**无头模式**（托管代理 / CMA 模式）下运行，且需要将 PowerPoint 演示文稿作为**文件产出物**交付，而非通过 `mcp__office__powerpoint_*` 编辑实时文档时，请使用此技能。

## 输出契约

- 写入至 `./out/<name>.pptx`。如果 `./out/` 目录不存在，请先创建。
- 在最终消息中返回相对路径，以便编排层可以统一提取。

## 如何构建演示文稿

编写一段简短的 Python 脚本并使用 Bash 运行。使用 `python-pptx` 库：

```python
from pptx import Presentation
from pptx.util import Inches, Pt

prs = Presentation("./templates/firm-template.pptx")  # 如果提供了公司模板
# 或者: prs = Presentation()

slide = prs.slides.add_slide(prs.slide_layouts[5])    # 仅含标题的布局
slide.shapes.title.text = "估值摘要"
# ... 添加表格 / 图表 / 文本框 ...

prs.save("./out/pitch-<target>.pptx")
```

## 操作规范（参考实时 Office 环境下的 `pitch-deck` 技能）

- **每张幻灯片传达一个核心观点。** 标题应当陈述结论；正文内容负责支撑该结论。
- **所有数据必须可溯源至模型。** 如果数字源自 `./out/model.xlsx`，请在脚注中注明对应的工作表（Sheet）和单元格（Cell）。
- **优先使用公司模板。** 当 `./templates/` 目录中挂载了模板时请务必使用；否则采用默认布局。
- **图表处理**：当对画面精准度（Fidelity）有较高要求时，相比于 pptx 原生图表，优先选择嵌入从模型渲染生成的 PNG 图片。
- **禁止外部发送。** 此技能仅负责本地文件的写入，严禁进行电子邮件发送或云端上传。

## 何时**不**适用

如果 `mcp__office__powerpoint_*` 相关工具可用（协作插件模式），请优先使用这些工具——它们可以直接驱动用户的实时文档，并支持设置人工审阅检查点。本技能仅作为无头运行环境下的文件生成后备方案。

### 1. 为什么用代码生成 PPT 而不是手动做——"从手工作坊到自动流水线"

在普通人的认知中，做 PPT 就是"打开 PowerPoint → 打字 → 粘贴图表 → 保存"。但在投行和 PE，越来越多的高频 PPT（如季度更新、估值摘要、公司介绍页）是通过 Python 代码生成的。

**代码生成 PPT 的五个核心优势**：

**① 一致性（Consistency）**：
- 手动做 10 份 PPT → 字体可能不一样、间距可能不一样、颜色可能不一样
- 代码生成 100 份 PPT → 每一份都完全一模一样——因为代码不会"今天喜欢红色，明天喜欢蓝色"
- 重要场景：给 10 个 LP 发 10 份 PPT，不能有一份"看起来不一样"

**② 效率（Efficiency）**：
- 手动做一份 30 页的 PPT 可能需要 4-6 小时
- 代码生成：改 Excel 里的数据 → 运行脚本 → 60 秒生成
- 典型场景：老板说"把估值从 $800M 改到 $850M" → 你不需要手动改 20 页 PPT 里的数字，改 Excel 然后重新运行脚本

**③ 防错（Error Prevention）**：
- 手动复制粘贴数字 → 出错概率 1-5%（一个人的疏忽）
- 代码从 Excel 读取数字 → 出错概率 0%（逻辑固定，要么全对要么全错）
- 改一个数字要手动更新 20 个地方 → 很容易漏——代码不会"漏"

**④ 版本控制（Version Control）**：
- 手动 PPT 的版本管理：`pitch_v1.pptx`, `pitch_v2.pptx`, `pitch_v3_final.pptx`, `pitch_v3_final_V2.pptx`
- 代码生成的版本管理：用 Git 追踪脚本变化 → 谁改了哪个模板、什么时候改的、为什么改——全部可追溯
- 来源演进追踪：三个月前的提委会材料长什么样？Git log 告诉你

**⑤ 数据溯源（Data Traceability）**：
- 手动 PPT 里的数字：你记得是"从哪个 Excel 的哪个单元格拿的"吗？三天后你可能忘了
- 代码生成的 PPT ：脚本里写清楚了 `= model.xlsx!Assumptions!C5` → 永远知道来源
- 如果被 LP 质疑："$850M 是哪里来的？"→ 你可以 3 秒追溯到模型源头

**打比方——手动做 PPT vs 代码生成 PPT**：
- 手动做 PPT = 手写 30 封信 → 每封信稍有不同，写错一个字要重写
- 代码生成 PPT = 印刷机打印 30 封信 → 每封一模一样，改一个字只需改模板然后全部重印

**所以这意味着什么**：本技能的核心价值在于——用代码确保每一个投行 PPT 在内容（数字）和形式（格式）上都 100% 一致、100% 可追溯。

### 2. Placeholder（占位符）——"模板的'插卡槽'"

**Placeholder（占位符）** 是 PPT 模板中的一个关键概念——模板中预留的"空位"，等待被实际内容（文字、图表、表格）填充。

**Placeholder 在金融 PPT 中的三种类型**：

**① 文字占位符（Text Placeholder）**：
- 模板上写着"[公司名称]" → 脚本用实际公司名替换：`"苹果"`
- 模板上写着"[估值范围]" → 脚本用计算好的数字替换：`"$180 - $220 每股"`
- 关键：括号中的提示文字不会被客户看到——它们只在模板中可见

**② 图表占位符（Chart Placeholder）**：
- 模板上有一个"图表区域"（一个空框，写了"[插入估值散点图]"）
- 脚本在运行时，根据 Excel 数据生成实际的散点图，填充到这个空框里
- 关键：图表不在模板中预设——模板只定义了"图表放在哪里"和"图表多大"

**③ 表格占位符（Table Placeholder）**：
- 模板上有一个"表格区域"（空表格框架，有表头但没数据）
- 脚本运行时，从 Excel 读取各行数据，逐行填充
- 例如：模板表头是"财务指标 \| 2023 \| 2024 \| 2025E"，数据行由脚本填充

**Placeholder 的设计原则**：
- 用明确的标记符号：`[公司名称]` 而不是 `_____`
- 说明数据源：`[从 Income_Statement.xlsx!B5 填充]` 而不是 `[填数字]`
- 说明格式要求：`[百分比，保留一位小数]` 而不是只写 `[利润率]`

**打比方——Placeholder 就像"宜家家具的说明书"**：
- 模板 = 宜家的安装说明书（告诉你每个部件放在哪里）
- Placeholder = 螺丝孔（某个位置等待被填充）
- 脚本 = 安装工人（按照说明书，把螺丝拧进对应的孔里）
- 最终 PPT = 安装好的家具（所有螺丝都在对应的位置）

**所以这意味着什么**：本技能强调 Placeholder 的管理——它们是模板和最终 PPT 之间的"桥梁"。好的 Placeholder 设计让代码 100% 可靠地生成 PPT，而不需要人工干预。

### 3. 数据溯源（Data Traceability）——"每一个数字都有'户口本'"

**数据溯源（Data Traceability）** 是金融 PPT 最重要的"非视觉"质量要求——PPT 上的每一个数字，都能追溯到它的"源头"。

**为什么数据溯源在金融中如此重要**：

**场景 1：LP 质疑一个数字**
- LP："第 12 页的 EBITDA 利润率 23.5% 是哪来的？"
- 如果数字是手打的 → 你："呃……应该是从模型里抄的……让我找找……"
- 如果数字是由代码从 `model.xlsx!Assumptions!C5` 读取的 → 你："来自模型假设页，C5 单元格，基于 Q4 实际数据计算。"

**场景 2：内部审查发现问题**
- 合伙人："这个估值好像比上次低了 5%，怎么回事？"
- 如果你有数据追踪 → 你可以追溯："因为模型假设中的 WACC 从 8% 上调到 8.5%，反映了最近利率上升。"→ 有理有据
- 如果你没有 → "我也不确定……" → 不专业

**数据溯源的实现方式**：
1. **源头引用**：PPT 生成时，记录每一个数字来自哪个 Excel 的哪个单元格
2. **版本锁定**：记录使用的是模型的哪个版本（v3.2）
3. **变更日志**：记录数字改了谁、什么时候改的、为什么改

**打比方——数据溯源就像"食品成分标签"**：
- 你买一包饼干 → 包装上会写"面粉（来自山东）、糖（来自广西）、油（来自马来西亚）"
- 金融 PPT 也是一样 → 每一个数字都应该有"成分标签"：来源 + 版本 + 时间
- 没有溯源的数字 = 三无产品 → 在金融行业，这是"不合格品"

**所以这意味着什么**：本技能强调数据溯源——因为金融 PPT 不是"设计作品"，而是"决策文件"。决策文件的核心要求是：**每一个数字都必须能经受住审核**。

### 4. 估值摘要（Valuation Summary）——"一句话说清公司值多少钱"

**估值摘要（Valuation Summary）** 是投行PPT 中最重要的页面之一。它像是一场路演的"高潮"——前面的行业分析、财务分析、竞争分析，都是为了铺垫这一页：**"这个公司到底值多少钱？"**

**一个标准的 Valuation Summary 包含什么**：

| 元素 | 说明 | 例子 |
|------|------|------|
| **估值范围** | 最低-最高区间 | "$150 - $200 每股" |
| **估值方法** | 用了什么方法 | DCF、可比公司、先例交易 |
| **关键假设** | 每个方法的核心假设 | "DCF 假设 3% 终端增长率、9.5% WACC" |
| **敏感性分析** | 关键变量变化对估值的影响 | "如果 WACC 上升 0.5%，估值下降 8%" |
| **足球场图** | 不同方法得出的估值范围可视化 | 类似于足球场的"横道图" |

**如何构建一个完整的 Valuation Summary（足球场图）**：

```
估值方法                每股估值范围       中位数       结论
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
可比公司分析           ████████░░░░░░░░░░░ $155-$185   $170
先例交易分析           ██████████████░░░░░░ $175-$210   $192
DCF 估值               ██████████░░░░░░░░░ $158-$195   $176
52 周交易区间          ░░██░░░░░░░░░░░░░░░ $120-$160   $140
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

综合估值区间                                   $150-$200
建议目标价                                     $180
当前股价                                       $135
潜在上行空间                                    33%
```

**每种估值方法的"性格"**：

| 方法 | 核心原理 | "性格" | 在什么市场最有效 |
|------|---------|--------|----------------|
| **可比公司分析** | 类似公司值多少钱，你的公司就该值多少钱 | 跟随者——跟随行业或大市的情绪 | 牛市（所有同行都贵） |
| **先例交易分析** | 历史上类似的交易付了多少钱 | 含"控股权溢价"——通常最高的一档 | 并购市场活跃的时候 |
| **DCF 估值** | 公司未来的现金折到现在值多少钱 | 独立者——和当前市场情绪无关 | 熊市（理性估值锚点） |
| **交易区间** | 过去一年市场愿意以什么价格买卖 | 滞后指标——只是噪音的历史平均 | 提供安全区间的参考下界 |

**为什么三种（或四种）估值方法要并列展示**：
- 单一方法不可靠——每种方法都有盲点
- 三种方法"交叉验证"——如果三种方法都指向 $150-$200，你的估值更有说服力
- 如果结论分叉太远（DCF 说 $100，可比说 $200）→ 说明你的假设可能有问题，需要排查

**打比方——Valuation Summary 就像"房产估值报告"**：
- 可比公司 = "同小区类似户型最近卖了多少"（市场价）
- 先例交易 = "这个小区过去 1 年被收购的成交价"（含并购溢价）
- DCF = "如果永远不卖，靠租金赚现金流折现回来值多少"（内在价值）
- 综合结论 = "我建议挂牌价定在 500 万到 550 万之间"（基于三种方法的交叉验证）

**所以这意味着什么**：本技能生成的 PPT 中，估值摘要是"终极页面"——前面所有的分析和数据，最终都汇总到这一页。

### 5. 无头模式下的 PPT 渲染机制——"你不是在'做' PPT，你是在'描述' PPT"

在无头模式下，你没有 PowerPoint 软件可用——你不能"打开 PPT、插入文本框、拖动图表"。你必须用代码"描述" PPT 的每一个元素：位置、大小、颜色、内容。

**无头模式下的核心操作**：

```python
# 你不是"做" PPT——你是在"描述" PPT

# 1. 指定文本框的位置和大小
txBox = slide.shapes.add_textbox(left=Inches(1), top=Inches(2), 
                                  width=Inches(8), height=Inches(1))

# 2. 给文本框放文字
tf = txBox.text_frame
tf.text = "这是我们建议的估值范围"

# 3. 设置字体和样式
p = tf.paragraphs[0]
p.font.size = Pt(18)
p.font.bold = True
p.font.color.rgb = RGBColor(0x00, 0x2B, 0x5C)  # 公司品牌色
```

**有头模式 vs 无头模式的对比**：

| 操作 | 有头模式（手动） | 无头模式（代码） |
|------|----------------|----------------|
| 放一个标题 | 点"插入文本框" → 拖拽 → 打字 | `add_textbox(left=1, top=0.5, width=8, height=0.8)` |
| 改字体大小 | 选中文字 → 右上角选字号 | `p.font.size = Pt(18)` |
| 插入图表 | 点"插入图表" → 选类型 → 粘贴数据 | 用 openpyxl 生成 Excel 图表 → 嵌入 PPTX |
| 保存 | Ctrl+S | `prs.save("./out/pitch.pptx")` |

**所以这意味着什么**：本技能在无头模式下运行——你不需要 PowerPoint 软件。你只需要 Python 和 python-pptx 库，就可以在任何环境（包括云服务器）上"制造" PPT。
