# 模型检查器

调试和审计财务模型中的错误——循环引用、公式失效、硬编码覆盖、资产负债表失衡、现金流量不勾稽以及逻辑漏洞。当模型无法平衡、产生意外结果，或在提交给客户或投资委员会前，均可使用本工具。

## 工作流程

### 第一步：获取模型

- 接收用户的 Excel 模型（.xlsx 或 .xlsm）
- 识别模型类型：DCF、LBO、并购、三表联动、可比公司分析、回报分析或自定义
- 梳理结构：存在哪些工作表、如何相互链接、输入与输出分别在哪

### 第二步：结构检查

**工作表与布局审阅：**
- 输入项是否与计算项清晰分离？
- 是否有一致的颜色编码规范？（蓝色=输入，黑色=公式，绿色=跨表链接）
- 是否存在可能含有覆盖项的隐藏工作表或行？
- 模型流向是否合理？（假设 → 利润表 → 资产负债表 → 现金流量表 → 估值）

**公式一致性：**
- 检查公式中的硬编码数字（部分硬编码）
- 检查行/列范围内公式是否不一致（同行公式应为跨列拖拽的相同公式）
- 识别任何 #REF!、#VALUE!、#N/A、#DIV/0! 错误
- 标记格式为公式但实际含有硬编码值的单元格

### 第三步：完整性检查

**资产负债表：**
- 各期间资产 = 负债 + 净资产
- 若不平衡，量化差距并追溯失衡位置
- 检查留存收益是否正确滚动：期初留存 + 净利润 - 股利 = 本期留存
- 验证商誉和无形资产是否从并购假设流入（并购模型适用）

**现金流量表：**
- 各期间现金流量表期末现金 = 资产负债表现金
- 经营活动现金流 + 投资活动现金流 + 融资活动现金流 = 现金变动额
- 现金流量表折旧摊销与利润表匹配
- 现金流量表资本开支与资产负债表 PP&E 滚动表匹配
- 现金流量表营运资本变化与资产负债表变动（应收账款、应付账款、存货）匹配

**利润表：**
- 营收拆解与分部/产品明细勾稽
- 营业成本和毛利率与假设一致
- 所得税费用 = 税前利润 × 税率（检查递延所得税调整）
- 股数与稀释方案勾稽（期权、可转债、回购）

**循环引用：**
- 检查循环引用（利息支出 → 债务余额 → 现金 → 利息）
- 若为有意为之（LBO/三表联动模型中常见），验证迭代开关是否正常工作
- 若为无意引入，追踪循环路径并提出打破循环的方案

### 第四步：逻辑检查

**合理性验证：**
- 增长率是否合理？（无解释的100%+营收增长 = 红色预警）
- 利润率是否在行业正常范围内？标记异常值
- DCF 中终端价值是否占主导？（超过75%的企业价值来自终端价值为黄色预警）
- 预测是否存在不现实的"曲棍球棒"形态？
- 第10年时 EBITDA 复合增长是否达到荒谬的数字？

**敏感性与边界情况：**
- 零增长时会发生什么？负增长时呢？
- EBITDA 为负时模型会崩溃吗？
- 杠杆率是否会变为负数或超出现实界限？
- 是否存在零除风险？

**跨工作表一致性：**
- 被链接的单元格是否确实与来源匹配？（复制粘贴错误很常见）
- 所有工作表的日期标题是否一致？
- 单位是否匹配（千元 vs 百万元 vs 实际值）？

### 第五步：按模型类型检查常见问题

**DCF 模型：**
- 折现率适用期间错误（年中惯例 vs 年末惯例）
- 终端价值未正确折现回来
- WACC 使用账面价值而非市场价值
- 自由现金流包含利息支出（应为无杠杆自由现金流）
- 税盾重复计算

**LBO 模型：**
- 还款金额与现金归还机制不匹配
- PIK 利息未计入债务余额
- 管理层跟投未体现在回报中
- 退出倍数适用于错误的 EBITDA（过去十二个月 vs 未来十二个月）
- 交易费用和支出未从第一天股权中扣除

**并购模型：**
- 增厚/摊薄使用了错误的股数（交易前 vs 交易后）
- 协同效应逐步兑现未正确建模
- 购买价格分配不平衡
- 未计入放弃的现金利息
- 交易费用未列入资金来源与用途

**三表联动模型：**
- 营运资本变化符号约定错误
- 折旧与 PP&E 明细表不匹配
- 债务到期明细与本金偿还不匹配
- 无解释地以超过净利润的金额支付股利

### 第六步：出具报告

生成模型审计报告：

**摘要：**
- 模型类型和总体评估（干净 / 有轻微问题 / 有重大问题）
- 按严重程度划分的问题数量

**问题清单：**

| 序号 | 工作表 | 单元格/范围 | 严重程度 | 类别 | 描述 | 建议修复方案 |
|------|--------|------------|----------|------|------|------------|
| 1 | | | 严重/警告/提示 | 公式/逻辑/平衡/硬编码 | | |

**严重程度定义：**
- **严重**：模型输出错误结果（资产负债表不平衡、公式失效）
- **警告**：模型可运行但存在风险（硬编码、公式不一致、边界情况失效）
- **提示**：风格和最佳实践建议（颜色编码、布局、命名）

### 第七步：输出结果

- 问题清单（在对话中或 Excel 文件中）
- 标注了问题单元格的模型（如用户提供了文件）
- 含修复优先级的总体评估

## 重要注意事项

- 始终先检查资产负债表平衡——若不平衡，其他问题暂时可以搁置
- 硬编码覆盖是模型错误的第一大来源——务必积极搜索
- 符号约定错误（现金流出的正负号）极为常见
- "能运行"的模型仍可能是错的——用行业基准对输出结果进行合理性检验
- 若模型使用 VBA 宏，注明任何无法仅通过公式审计的宏驱动计算
- 未经询问不要修改模型——报告问题，由用户决定如何修复

### 1. 模型检查为什么是"发车前的最后一道闸门"——"不是找茬，是救你"

在投行和 PE，财务模型在发给客户、投委会或 LP 之前，必须经过严格的"模型检查"。这不是走形式，而是"最后一道防线"。

**模型检查的四个诊断维度**：

| 维度 | 检查什么 | 如果出了问题会怎样 |
|------|---------|------------------|
| **结构检查** | 工作表布局、公式一致性、颜色编码、数据流 | 模型像"意大利面条"——别人看不懂，无法审计 |
| **完整性检查** | 资产负债表配平、现金钩稽、留存收益滚动、折旧匹配 | 数字有漏洞——"钱凭空消失或凭空出现" |
| **逻辑检查** | 增长率合理性、利润率、终端价值占比、"曲棍球棒"效应 | 模型看起来"很美"但逻辑上不可能——垃圾进垃圾出 |
| **类型检查** | DCF/LBO/并购模型中各自特有的易错点 | 例如 DCF 的终端价值忘了折现——一错错 5 亿 |

**为什么必须分成四个维度**：
- 如果只检查公式（#REF! 等错误）→ 可能漏掉"逻辑合理但数字不对"的问题
- 如果只检查资产负债表配平 → 可能漏掉"增长率不切实际"的问题
- 四个维度全查，才能确保"模型健康"

**打比方——模型检查就像飞机的"航前检查"**：
- 飞行员在起飞前要逐项检查：引擎压力、燃料、导航系统、襟翼……
- 任何一个项目没检查就起飞 → 可能机毁人亡
- 财务模型的"航前检查"就是本技能的四步检查——任何一项没通过，模型就不能"外发"

**真实案例——为什么检查不应该"马虎"**：
2021 年，某 PE 基金在内部投委会上展示了一个 LBO 模型，结论是 IRR = 25%——看起来是个不错的交易。但在"发车"前，一位 VP 重新跑了模型检查，发现现金流量表中的"折旧"和利润表中的"折旧"不一致（差了 $15M）——IRR 修正后只有 18%。如果没发现这个错误，投委会就基于错误的数字做了投资决策，最终亏了 $200M+。

**所以这意味着什么**：本技能不是"找茬"，而是"救命"——模型检查是投行和 PE 的"航前检查清单"，每一条都必须过。

### 2. 公式一致性检查——"别让你的模型像补丁衣服"

**公式一致性检查（Formula Consistency Check）** 是模型检查中最基础但最容易被跳过的一步：检查同一行（或同一列）的公式是否一致。

**什么是不一致的公式**：
```
C5 = B5 * (1 + Assumptions!$B$2)   ← 正确
D5 = B5 * (1 + Assumptions!$B$2)   ← 错了！应该是 C5 * 而不是 B5 *
E5 = 2300                           ← 硬编码！和其他公式不一致
```

**为什么公式不一致很危险**：
- 表面上看，模型"能跑"——没有 #REF! 错误，数字看起来合理
- 但 D5 用的是"第一年"的基数（B5）而不是"上一年"的基数（C5）→ 增长率被重复计算了
- E5 直接硬编码了 2300 → 如果改变假设，E5 不会更新
- 这种错误非常隐蔽——没有人会逐单元格检查公式，尤其是在 5 万行的大模型中

**如何发现不一致**：
1. **视觉检查**：选中一行，看每个单元格的公式栏 → 太慢
2. **公式审计工具**：使用 Excel 的"显示公式"模式（Ctrl+`）→ 可以快速扫描
3. **结构检查**：选择一整行，使用"定位条件 → 公式不一致"
4. **脚本检查**：用 Python 或 VBA 逐单元格比对公式模式 → 最可靠

**打比方——公式一致性就像"生产线上的质检"**：
- 假设你在生产 100 个汉堡 → 前 99 个是"面包+肉饼+生菜"，第 100 个变成了"面包+肉饼+石头"
- 如果你不做质检，第 100 个汉堡就会送到客户手里 → 客户会投诉
- 公式一致性检查 = 在生产线的最后一关，确保每个"汉堡"（单元格）的配方（公式）是一致的

**所以这意味着什么**：本技能要求"检查行/列范围内公式是否不一致"——因为一个不一致的公式，足以让整个模型的输出产生系统性偏差。

### 3. 跨表钩稽关系——"三张表的'恋爱关系'不能断"

**钩稽关系（Ties/Reconciliation）** 是财务模型中三张表之间的"同步链接"——三张表不是独立的，而是互相依赖的。如果链接断了，模型就有问题。

**最重要的钩稽关系**：

**① 资产负债表配平（A = L + E）**：
- 这是最基本的会计恒等式
- 每一期都必须成立——如果有任何一期不成立，说明"钱凭空消失或出现了"
- 常见不配平原因：净利润没流入留存收益、折旧没减少固定资产、营运资本变动方向搞反了

**② 现金钩稽（Cash Tie）**：
- 现金流量表的"期末现金"必须等于资产负债表的"现金及现金等价物"
- 如果不等 → 现金流和资产负债表之间的链接断了
- 常见原因：某个现金流科目（如融资现金流中的"新股发行"）没有正确流入资产负债表

**③ 折旧钩稽（Depreciation Tie）**：
- 利润表的"折旧费用" → 现金流量表的"折旧加回" → 资产负债表的"累计折旧增加"
- 如果三个数字不一致 → 折旧计算有误

**④ 留存收益滚动（Retained Earnings Rollforward）**：
- 留存收益_期末 = 留存收益_期初 + 净利润 - 股利
- 如果这条不成立 → 利润表中的净利润没有正确流入资产负债表

**⑤ 营运资本变动（Working Capital Changes）**：
- 现金流量表中的"营运资本变动" = 本期营运资本 - 上期营运资本
- 如果这两者不一致 → 营运资本科目之间的链接断了

**打比方——钩稽关系就像"水管系统"**：
- 三张表之间的钩稽关系 = 房子的水管系统
- 如果一根水管断了 → 水会漏出来（模型不配平）
- 你需要找到"断在哪里"（不平衡的差额从哪一年开始出现）→ 然后"接回去"（修复公式链接）

**所以这意味着什么**：本技能要求检查所有关键钩稽关系——因为这五条线是三张表之间的"生命线"。断了一条，整个模型的可信度就归零。

### 4. 逻辑合理性检查——"数字可以'完美'但逻辑可以'荒谬'"

**逻辑合理性检查（Logical Reasonableness Check）** 是模型检查中"最高级"的一层——它不是检查公式对不对，而是检查"公式算出的结果是否合理"。

**为什么逻辑检查比公式检查更难**：
- 公式检查是"机械的"——Excel 会告诉你 #REF! 是错的
- 逻辑检查是"判断的"——Excel 不会告诉你"100% 增长率"不合理，它只会照算

**常见的逻辑红旗**：

**① 不切实际的增长率（Unrealistic Growth Rates）**：
- 第 3 年增长率 15% → 合理
- 第 5 年增长率 85% → "曲棍球棒"效应，需要解释原因
- 为什么不合理：一个成熟公司不可能突然"爆发"——如果真的有这样的增长，通常是因为并购或特殊事件

**② 终端价值占比过高（Excessive Terminal Value）**：
- DCF 估值 = 预测期 FCF 折现 + 终端价值折现
- 如果终端价值占企业价值的 80%+ → 黄色预警
- 为什么是问题：你的估值几乎完全取决于"无限远的未来"的假设——不确定性极大
- 华尔街常规：终端价值占比 50-70% 是正常范围

**③ 负杠杆 / 不现实的杠杆（Negative or Impossible Leverage）**：
- 在 LBO 模型中，债务/EBITDA 从 4x 变成 -2x → 某个地方算错了
- 或者债务/EBITDA = 15x（现实中银行最多借 5-6x）→ 杠杆假设脱离现实

**④ 零除风险（Division by Zero Risk）**：
- `= Revenue / EBITDA` 当 EBITDA 为 0 时 → #DIV/0! 错误，模型崩溃

**打比方——逻辑检查就像"体检报告解读"**：
- 公式检查 = 检查数值是否在正常范围（血压 120/80 → 正常）
- 逻辑检查 = 解读数值是否合理（血压 120/80 但患者 5 岁 → 不对！）
- 一个"没有错误"的模型可能输出"荒谬的结论"——这就是逻辑检查的价值

**所以这意味着什么**：本技能要求"合理性验证"——一个没有 #REF! 错误的模型，可能在"安静地"算出荒谬的答案。逻辑检查是最后一层防线。

### 5. 按模型类型的专项检查——"DCF/LBO/并购各有各的坑"

不同类型的财务模型有各自的"经典错误"——专项检查针对这些类型特有的陷阱：

**DCF 模型的常见错误**：
- 自由现金流包含了利息支出（应该是无杠杆 FCF = EBIT×(1-t) + D&A - CapEx - ΔWC）
- 终端价值用的增长率 > GDP 增长率（一家公司不可能永远比经济整体增长快）
- WACC 用了账面价值而非市场价值计算权重
- 年中折现和年末折现搞混——导致系统性偏差

**LBO 模型的常见错误**：
- 现金扫还（Cash Sweep）金额和可用现金不一致
- 管理层跟投（Management Rollover）没体现在股权结构中
- 退出 EBITDA 用的是 LTM（过去 12 个月）而不是 NTM（未来 12 个月）→ 估值基础搞错了
- PIK 利息（实物支付利息）没加到债务余额上

**并购模型的常见错误**：
- 增厚/摊薄分析用了错误的股数（交易前 vs 交易后）
- 协同效应（Synergies）分阶段实现，但模型错误地全额一次性计入
- 购买价格分配（Purchase Price Allocation）中商誉计算错误
- 交易费用没计入"资金来源与用途"（Sources & Uses）

### 给小白的一句话

> **模型检查（Check Model）是投行和 PE 财务模型的"航前检查清单"——从结构、完整性、逻辑和类型四个维度逐项排查，确保模型在发给客户/投委会之前是健康可靠的。公式一致性检查确保同一行的单元格使用相同的公式模式——一个不一致的公式足以让整个输出产生系统性偏差。钩稽关系是三张表之间的"生命线"——资产负债表配平、现金钩稽、折旧钩稽、留存收益滚动、营运资本变动，断了一条模型就不可信。逻辑检查是最"高级"的一层——不是检查公式是否有 #REF!，而是检查输出是否合理（100% 增长率、终端价值占 90%、杠杆 -2×）。DCF/LBO/并购模型各有各的"经典错误"——专项检查针对类型特有的陷阱。记住：没有经过检查的财务模型，就像没有经过航前检查的飞机——看起来能飞，但随时可能坠毁。**
