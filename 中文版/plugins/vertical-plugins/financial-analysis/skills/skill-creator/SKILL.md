# 技能创建指南 (Skill Creator)

本技能旨在为如何创建高效的技能提供指导。

## 关于技能 (Skills)

技能是模块化、自包含的软件包，通过提供专业化的知识、工作流和工具来扩展 Claude 的能力。可以将它们视为针对特定领域或任务的“新手入职指南”——它们将 Claude 从一个通用型智能体转化为具备程序化知识的专用型智能体，而这些知识是任何模型都无法完全具备的。

### 技能提供的核心价值

1. **专业工作流** - 针对特定领域的跨步骤操作程序。
2. **工具集成** - 处理特定文件格式或 API 调用。
3. **领域专业知识** - 企业特有的知识、数据架构（Schemas）及业务逻辑。
4. **捆绑资源** - 针对复杂且重复性任务的脚本、参考资料及资产。

## 核心原则

### 简洁至上 (Concise is Key)

上下文窗口 (Context Window) 是一项公共资源。技能需要与 Claude 运行所需的其他信息共享这一窗口：系统提示词、对话历史、其他技能的元数据以及用户的实际请求。

**默认假设：Claude 已经非常聪明。** 仅添加 Claude 尚未掌握的上下文。审视每一条信息：“Claude 真的需要这段解释吗？”以及“这一段内容的 Token 成本是否物有所值？”

优先使用简明的示例，而非冗长的解释。

### 设定适当的自由度

根据任务的脆弱性和变异性设定精确度：

**高自由度（基于文本的指令）**：适用于存在多种有效方法、决策依赖于具体情境，或通过启发式方法引导的情境。

**中等自由度（伪代码或带参数的脚本）**：适用于存在首选模式、允许一定差异，或配置会影响行为的情境。

**低自由度（特定脚本，极少参数）**：适用于操作复杂脆弱且易出错、一致性至关重要，或必须遵循特定顺序的情境。

想象 Claude 正行走在一条路径上：狭窄且紧邻悬崖的桥梁需要特定的护栏（低自由度），而开阔的田野则允许多条路线选择（高自由度）。

### 技能的构成要素

每个技能都包含一个必须的 `SKILL.md` 文件以及可选的捆绑资源：

```
skill-name/
├── SKILL.md (必填)
│   ├── YAML 前置元数据 (必填)
│   │   ├── name: (必填)
│   │   └── description: (必填)
│   └── Markdown 指令 (必填)
└── 捆绑资源 (可选)
    ├── scripts/          - 可执行代码 (Python/Bash 等)
    ├── references/       - 旨在根据需要加载到上下文中的文档
    └── assets/           - 用于输出的文件 (模板、图标、字体等)
```

#### SKILL.md (必填)

每个 `SKILL.md` 包含：

- **前置元数据 (Frontmatter)** (YAML)：包含 `name`（名称）和 `description`（描述）字段。这是 Claude 决定何时使用该技能的唯一读取依据，因此清晰、全面地描述技能及其应用场景至关重要。
- **正文 (Body)** (Markdown)：使用技能及其指南的说明。仅在技能被触发后才会加载（如果需要）。

#### 捆绑资源 (可选)

##### 脚本 (`scripts/`)

针对需要确定性可靠性或被反复重写的任务，提供可执行代码。

- **适用场景**：当同一段代码被反复重写，或需要高度确定性的可靠性时。
- **示例**：用于 PDF 旋转任务的 `scripts/rotate_pdf.py`。
- **优势**：节省 Token、具备确定性，且无需加载到上下文中即可执行。
- **注意**：Claude 可能仍需读取脚本以进行补丁修复或环境适配。

##### 参考资料 (`references/`)

旨在根据需要加载到上下文中的文档，用于辅助 Claude 的处理和思考。

- **适用场景**：Claude 在工作时需要参考的文档。
- **示例**：`references/finance.md`（金融架构）、`references/mnda.md`（公司保密协议模板）、`references/policies.md`（公司政策）、`references/api_docs.md`（API 规范）。
- **优势**：保持 `SKILL.md` 精简，仅在 Claude 判定需要时加载。
- **最佳实践**：如果文件过大（超过 1 万字），请在 `SKILL.md` 中包含 grep 搜索模式。
- **避免重复**：信息应存在于 `SKILL.md` 或参考文件中，而非两者皆有。优先将详细信息放入参考文件，除非它是技能的核心——这在保持 `SKILL.md` 精简的同时，确保了信息的可发现性，且不会占用过多上下文窗口。

##### 资产 (`assets/`)

不旨在加载到上下文中，而是用于 Claude 生成的最终输出中。

- **适用场景**：当技能需要包含在最终输出中的文件时。
- **示例**：品牌资产 `assets/logo.png`、PPT 模板 `assets/slides.pptx`、前端脚手架 `assets/frontend-template/`。
- **优势**：将输出资源与文档分离，使 Claude 无需加载文件即可使用它们。

#### 技能中不应包含的内容

技能应仅包含支持其功能的必备文件。切勿创建无关的辅助文档，包括：

- README.md
- INSTALLATION_GUIDE.md
- QUICK_REFERENCE.md
- CHANGELOG.md 等

技能应仅包含 AI 智能体完成任务所需的信息。不应包含关于创建过程的辅助上下文、安装测试程序或面向用户的文档。创建额外的文档文件只会增加混乱。

### 渐进式披露设计原则 (Progressive Disclosure)

技能采用三级加载系统，以高效管理上下文：

1. **元数据 (名称 + 描述)** - 始终在上下文中（约 100 字）。
2. **SKILL.md 正文** - 技能触发时加载（5k 字以内）。
3. **捆绑资源** - 根据 Claude 的需要加载（无上限，因为脚本无需读取即可执行）。

#### 渐进式披露模式

保持 `SKILL.md` 精简在 500 行以内。接近上限时应拆分内容。拆分时，务必在 `SKILL.md` 中清晰标注并在何时读取，以引导读者。

**核心原则：** 当技能支持多种变体、框架或选项时，仅在 `SKILL.md` 中保留核心流转和选择指南。将特定变体的细节（模式、示例、配置）移至独立的参考文件。

**模式 1：带参考的高层级指南**

```markdown
# PDF 处理

## 快速上手

使用 pdfplumber 提取文本：
[代码示例]

## 高级功能

- **表单填充**：详见 [FORMS.md](FORMS.md) 完整指引
- **API 参考**：详见 [REFERENCE.md](REFERENCE.md) 详细方法
- **示例**：详见 [EXAMPLES.md](EXAMPLES.md) 常见模式
```

**模式 2：特定领域组织**

针对跨多个领域的技能，按领域组织内容以避免加载无关信息：

```
bigquery-skill/
├── SKILL.md (概览与导航)
└── reference/
    ├── finance.md (收入、账单指标)
    ├── sales.md (商机、销售管线)
    ├── product.md (API 使用、功能)
    └── marketing.md (活动、归因分析)
```

**模式 3：条件细节**

展示基础内容，链接高级内容：

```markdown
# DOCX 处理

## 创建文档

使用 docx-js 创建。见 [DOCX-JS.md](DOCX-JS.md)。

## 编辑文档

**修订记录**：详见 [REDLINING.md](REDLINING.md)
**OOXML 细节**：详见 [OOXML.md](OOXML.md)
```

## 技能创建流程

创建技能包含以下步骤：

1. 通过具体案例理解技能。
2. 规划可重用的资源内容（脚本、参考资料、资产）。
3. 初始化技能（运行 `init_skill.py`）。
4. 编辑技能（实现资源并编写 `SKILL.md`）。
5. 封装技能（运行 `package_skill.py`）。
6. 基于实际使用进行迭代。

### 第 1 步：通过具体案例理解技能

只有当技能的使用模式已非常明确时，方可跳过此步。

要创建有效的技能，必须清晰理解其具体应用场景。例如，构建“图像编辑”技能时，核心问题包括：

- “该技能支持哪些功能？编辑、旋转，还是其他？”
- “你能提供一些使用该技能的具体示例吗？”
- “用户会通过什么指令触发该技能？”

### 第 2 步：规划可重用的技能内容

分析每个案例：

1. 考虑如何从零开始执行该案例。
2. 确定哪些脚本、参考资料和资产在重复执行时会有所帮助。

**示例**：在构建 `pdf-editor` 时，分析发现旋转 PDF 每次都需重写相同代码，因此在 `scripts/rotate_pdf.py` 中存储脚本非常有益。

### 第 3 步：初始化技能

从零创建新技能时，请始终运行 `init_skill.py` 脚本，它会自动生成包含目录结构和模板的文件夹。

用法：
```bash
scripts/init_skill.py <skill-name> --path <output-directory>
```

### 第 4 步：编辑技能

编辑时请记住，该技能是供另一个 Claude 实例使用的。应包含对 Claude 来说收益

### 1. 金融领域什么工作适合"技能化"——"从重复劳动中解放大脑"

在投行和 PE，分析师每天花大量时间在"重复性高、创造性低"的工作上。这些工作恰恰是最适合被"技能化"的。

**适合技能化的金融任务（高频 + 规则明确）**：

| 任务类型 | 例子 | 为什么适合技能化 |
|---------|------|----------------|
| **数据格式化** | 把 Bloomberg 导出的数据清洗为标准格式 | 规则 100% 明确——去除空格、统一日期格式、转数字类型 |
| **可比公司分析** | 从 FactSet 拉 10 家公司的财务指标做对比 | 步骤固定——拉数据 → 清洗 → 计算倍数 → 生成图表 |
| **估值散点图** | 用同行增长率和估值倍数画散点图 | 输入/输出明确——X 轴 Y 轴定义清楚，规则固定 |
| **财报回测** | 每季度把实际数据填入旧模型，更新预测 | 流程标准化——读取财报 → 更新模型 → 检查配平 → 输出 |
| **PPT 模板填充** | 把 Excel 数据填入 Pitchbook 模板 | 映射关系固定——哪个数字进哪个 Placeholder |

**不适合技能化的金融任务（低频 + 需要判断力）**：

| 任务类型 | 例子 | 为什么不适合 |
|---------|------|------------|
| **首次覆盖报告** | 写一个全新公司的 Initiation | 每个公司不同——商业模式、竞争格局、投资逻辑都需要从零构思 |
| **投资逻辑制定** | 判断"为什么买这只股票" | 需要独特判断力——不能套用模板 |
| **交易谈判** | 和卖方讨论价格和条款 | 高度动态——不受规则约束 |
| **危机处理** | 被投公司出了问题，紧急应对 | 突发性强——没有标准流程 |

**打比方——"技能化"就像"流水线"**：
- 适合技能化的任务 = 流水线上拧螺丝 → 每次都一样，可以自动化
- 不适合技能化的任务 = 设计一辆新车 → 每次都不一样，需要创造力

**所以这意味着什么**：本 Skill Creator 在金融环境中最重要的应用是"识别哪些工作可以封装为技能"——一个好的技能可以节省分析师每周 10-20 小时的重复劳动。

### 2. 自由度（Degree of Freedom）——"给 AI 多大自主权"

**自由度（Degree of Freedom）** 是本技能的核心概念——决定了 AI 在执行任务时的"自主权"有多大。在金融场景中，不同任务需要不同的自由度。

**三种自由度在金融中的实际应用**：

**① 高自由度（基于文本指令）——"AI 你自己看着办"**
- 适用场景：多种有效方法都可行，决策依赖具体情境
- 金融例子：
  - 理解客户需求（"帮我们分析一下新能源行业"）
  - 撰写投资摘要（不同的分析师有不同的侧重点）
  - 回答开放性问题（"这家公司的风险在哪里？"）
- 风险：AI 可能走偏——需要人工审核输出
- 类比：项目经理分配任务："搞定新能源行业分析"（不说具体怎么做，只给目标）

**② 中等自由度（伪代码+参数）——"AI 你按这个框架做"**
- 适用场景：存在首选模式，但允许一定变化
- 金融例子：
  - 生成估值散点图（X 轴 Y 轴已定义，但选哪几家同行可以灵活调整）
  - 写财报速评（有标准的"Beat/Miss→Guidance→Implication"框架，但具体措辞可变）
  - 格式化 Excel（有蓝/黑/绿色规范，但数字的具体位置灵活）
- 风险：适度的自由确保一致性，同时保留应对变化的灵活性
- 类比：给出"模板"："按《财报速评模板》写，但你可以调整强调的重点"

**③ 低自由度（特定脚本，极少参数）——"AI 你必须严格照做"**
- 适用场景：操作复杂、脆弱且易出错，一致性至关重要
- 金融例子：
  - 资产负债表配平检查（A = L + E，一字节都不能差）
  - 数据清洗脚本（去除空格、统一格式 → 必须精确）
  - 合规声明插入（法律文本一个字不能改）
  - 交易费用计算（公式固定，错了就是几百万美元的偏差）
- 风险：过于僵化可能导致"死板"——但对于财务准确性要求极高的任务，这是必要的
- 类比：给"SOP 操作手册"："第 3 步：将 B2 单元格的值复制到 C5，不得跳过"

**打比方——自由度就像"驾驶辅助"**：
- 高自由度 = 导航告诉你目的地 → 你自己选路
- 中等自由度 = 导航给你三条路线 → 你选一条
- 低自由度 = 自动驾驶 → 你完全不需要碰方向盘

**所以这意味着什么**：在金融技能设计中，正确设置自由度是"艺术"——太宽了不可靠，太窄了不灵活。DCF 模型应该用低自由度，行业分析应该用中等自由度。

### 3. Context Window（上下文窗口）——"金融 AI 的稀缺资源"

**Context Window（上下文窗口）** 是 AI 模型在单次对话中能"记住"的最大信息量。它就像 AI 的"短期记忆"——是一次性把所有资料"装进脑子里"的能力上限。

**为什么 Context Window 在金融技能中特别重要**：
- 一份投行 Pitchbook 可能 80 页 → 如果全部放进上下文，已经占了一半窗口
- 一个 DCF 模型可能 5 万行 → 如果全部放入上下文，窗口直接"爆"
- 一份行业报告 + 公司年报 + 同行数据 → 信息量可能远超窗口限制
- **Context Window 是"稀缺资源"——每个 Token 都要精打细算**

**本技能设计的核心原则"简洁至上"在金融中的含义**：

| 设计决策 | 为什么 | 金融例子 |
|---------|--------|---------|
| 只添加 AI 不知道的知识 | AI 已经知道"什么是 EBITDA" → 不需要在技能里再解释一次 | 不要在技能里写"EBITDA 是息税折旧摊销前利润"——AI 知道 |
| 优先用简明示例而非冗长解释 | 一个例子比 500 字解释更省 Token | "像这样画估值散点图：X=增长率，Y=EV/EBITDA"（20 Token）vs "估值散点图是一种用于..."（200 Token） |
| 把参考资料独立出来，按需加载 | 不是每次都需要所有参考资料 | `references/valuation-methodologies.md` 只在做估值分析时才加载 |

**打比方——Context Window 就像"办公桌"**：
- 你只能在办公桌上放最重要的文件（Context Window 有限）
- 你不能把整个档案柜搬过来（一次放入太多信息）
- 聪明的做法是：桌上只放当前需要的文件，其他文件在档案柜里随用随取（references/ 按需加载）

**所以这意味着什么**：创建金融技能时，必须时刻关注"Token 成本"——技能写得太长，就会占用 Context Window，挤占用户实际问题的空间。

### 4. 好的金融技能应该具备哪些特质——"金融 AI 技能的质量标准"

**① 确定性（Determinism）——"同样的输入，永远产生同样的输出"**
- 金融中最不能接受的就是"不确定性"——一个 DCF 模型跑两次不应该产生两个不同的估值
- 技能中的关键计算应该用确定性脚本（Python），而不是依赖 AI 的"自由发挥"

**② 可审计性（Auditability）——"每一步都能追溯"**
- 技能的执行过程应该留下"审计追踪"——输入是什么、处理步骤、输出是什么
- 类似金融中的"Audit Trail"——如果有人质疑输出结果，能回溯到源头

**③ 错误处理（Error Handling）——"遇到错误不能'爆炸'"**
- 如果输入数据格式错误 → 技能应该报告"第 3 行的日期格式不对"，而不是崩溃
- 如果资产负债表不配平 → 技能应该指出"差额 $15M，可能来自第 5 行的公式"
- 金融模型中的错误处理 = 生命线——一个崩溃的模型意味着时间被浪费

**④ 版本控制（Versioning）——"谁改了什么、什么时候改的"**
- 金融技能必须有版本追踪——当估值结果发生变化时，能知道是哪个假设改了
- 类似 Git 的 commit log，但在技能层面

**⑤ 领域适配（Domain Fit）——"不是为了炫技，而是为了帮用户真正完成工作"**
- 一个好的金融技能不是"技术最先进的"，而是"最终用户最需要的"
- 如果一个技能用到了最复杂的 Python 库但用户不知道怎么用 → 失败
- 如果一个技能只用了简单的 openpyxl 但用户 1 分钟就能生成估值表 → 成功

---

### 5. 金融技能的设计哲学——"从用户痛点出发"

在金融工作流中设计技能，不能"为了技能而技能"——必须从用户的实际痛点出发。

**识别金融痛点的"三问法"**：

1. **什么工作你每周都做？（频率）** → 高频工作最值得技能化
2. **什么工作你做得最痛苦？（痛点）** → 痛点越深，技能价值越大
3. **什么工作出错代价最大？（风险）** → 高风险工作最需要确定性

**投行分析师的典型痛点**：

| 痛点 | 频率 | 出错代价 | 技能化价值 |
|------|------|---------|-----------|
| 每季度更新可比公司分析 | 4 次/年 | 中 | 高——节省 10 小时/次 |
| 手动格式化 Pitchbook | 每天 | 高（失去客户信任） | 极高——节省 20 小时/次 |
| 资产负债表配平调试 | 每周 | 极高（模型不可靠） | 极高——减少错误 90% |

**打比方——创建金融技能就像"给工匠定制工具"**：
- 你不需要"万能瑞士军刀"（什么都能做但什么都不精）
- 你需要为特定任务定制的工具——一个专门的扳手比一把瑞士军刀的扳手好用 100 倍
- 最好的技能就是"专门为你的工作流定制的工具"

**所以这意味着什么**：在创建金融技能之前，先问"用户的痛点是什么"——不是"AI 能做什么"，而是"用户最需要什么"。

### 给小白的一句话

> **技能创建器（Skill Creator）在金融领域的意义是"将重复性高、规则明确的分析工作封装为可复用的 AI 技能"——让投行和 PE 分析师从每周 20+ 小时的重复劳动中解放出来。自由度决定了 AI 的自主权——DCF 模型用低自由度（公式必须精确），行业分析用中等自由度（有框架但措辞灵活）。Context Window 是金融 AI 的"稀缺资源"——每个 Token 都要用在刀刃上，不需要重复 AI 已经知道的知识。好的金融技能必须具备确定性（同样的输入永远同样的输出）、可审计性（每一步可追溯）、错误处理（遇到错误不崩溃）。创建技能的核心是"从用户痛点出发"——不是 AI 能做什么，而是用户最痛苦、最高频、出错代价最大的工作是什么。**
