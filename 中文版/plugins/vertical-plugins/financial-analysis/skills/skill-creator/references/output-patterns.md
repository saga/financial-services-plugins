# 输出范式

当技能需要产出一致且高质量的结果时，请采用这些范式。

## 模板模式 (Template Pattern)

为输出格式提供模板。根据您的需求设定相应的规范力度。

**针对严格要求（如 API 响应或数据格式）：**

```markdown
## 报告框架

请务必遵循以下模板结构：

# [分析标题]

## 执行摘要
[对核心发现的单段综述]

## 核心洞察
- 发现 1（包含支撑数据）
- 发现 2（包含支撑数据）
- 发现 3（包含支撑数据）

## 改进建议
1. 具体且具备可行性的操作建议
2. 具体且具备可行性的操作建议
```

**针对柔性引导（当适应性调整更有利时）：**

```markdown
## 报告框架

以下是推荐的默认格式，请根据实际情况灵活调整：

# [分析标题]

## 执行摘要
[概览]

## 核心洞察
[根据您的发现调整各部分内容]

## 改进建议
[针对具体语境量身定制]

请根据特定分析类型的需求，对章节进行必要调整。
```

## 示例模式 (Examples Pattern)

若技能的输出质量依赖于对范例的理解，请提供“输入/输出”对：

```markdown
## 提交信息格式

请参考以下示例生成 Git 提交信息（Commit Message）：

**示例 1：**
输入：添加了使用 JWT 令牌的用户身份验证
输出：
```
feat(auth): implement JWT-based authentication

Add login endpoint and token validation middleware
```

**示例 2：**
输入：修复了报告中日期显示不正确的缺陷
输出：
```
fix(reports): correct date formatting in timezone conversion

Use UTC timestamps consistently across report generation
```

请遵循此风格：类型(作用域): 简要描述，随后是详细说明。
```

相比于单纯的文字描述，示例能帮助 Claude 更清晰地理解所需的文风与细节精度。

> **💡 Appendix: 领域知识小贴士**
> 
> **1. 执行摘要 (Executive Summary)**：想象你是一位日理万机的基金经理，没时间翻阅几百页的年报。执行摘要就是报告的“精华浓缩版”，旨在让你在电梯间的几十秒内，就能精准捕捉到最核心的投资逻辑和风险点。
> 
> **2. 核心洞察 (Key Findings)**：在金融分析中，数据本身是冰冷的，而洞察是有温度的。它不是简单地列出“股价跌了”，而是结合宏观环境和财务指标，分析出“为什么跌”以及“这种下跌是否反映了基本面”。它是从海量信息中捞出来的“真金”。
> 
> **3. 可行性建议 (Recommendations)**：好的金融建议不应该是“要注意风险”这种废话，而应该是“建议在 105 元分批建仓，止损位设在 98 元”。它必须像导航仪一样明确，让阅读者知道下一步该踩油门还是踩刹车。

## Appendix: 金融背景知识

这份文件是"输出范式（Output Patterns）"参考文档的详细说明。在金融服务业中，输出质量直接影响客户信任——无论是研究报告、财务模型还是演示文稿，都必须遵循一致的格式和标准。

---

### 1. 为什么需要输出范式？

**类比：**
想象你订阅了一份杂志。如果每期杂志的排版、字体、风格都不一样，你会觉得很混乱，甚至怀疑这是不是"正经杂志"。但如果每期都统一格式，你就会形成阅读习惯，信任度也会提升。

金融服务的输出范式就是"杂志的统一排版"——让客户一看就知道"这是专业的"。

---

### 2. 金融输出的"三大范式"

#### 范式 1：模板模式（Template Pattern）
**适用场景**：需要严格格式控制的输出（如 API 响应、监管报告）

**核心特征**：
- 固定结构
- 必填字段
- 数据验证

**例子**：监管报告模板
```
[报告标题]
[报告日期]
[报告机构]

1. 执行摘要
2. 详细数据
3. 风险声明
4. 签字确认
```

#### 范式 2：框架模式（Framework Pattern）
**适用场景**：需要灵活但结构化的输出（如研究报告、战略分析）

**核心特征**：
- 模块化结构
- 可扩展
- 逻辑清晰

**例子**：研究报告框架
```
1. 投资摘要（1页）
2. 行业分析（2-3页）
3. 公司分析（3-5页）
4. 财务模型（2-3页）
5. 估值分析（1-2页）
6. 风险提示（1页）
```

#### 范式 3：检查清单模式（Checklist Pattern）
**适用场景**：需要确保不遗漏关键步骤的任务（如尽职调查、合规审查）

**核心特征**：
- 逐项检查
- 签字确认
- 可追溯

**例子**：尽职调查检查清单
```
□ 公司章程审查
□ 财务报表审计
□ 法律诉讼排查
□ 知识产权核查
□ 环境合规检查
```

---

### 3. 输出质量的"四重检查"

| 检查层 | 检查人 | 检查内容 |
|--------|--------|---------|
| 第一层 | 作者 | 数据准确性、逻辑自洽 |
| 第二层 | 同事 | 同行评审、盲点发现 |
| 第三层 | 主管 | 战略一致性、客户适配 |
| 第四层 | 合规 | 法规遵循、风险披露 |

---

### 4. 真实案例：某投行的报告质量问题

某投行发布的研究报告出现以下问题：

| 问题 | 原因 | 后果 |
|------|------|------|
| 数据错误 | 未做交叉验证 | 客户据此投资亏损 |
| 逻辑矛盾 | 前后章节不一致 | 报告可信度下降 |
| 格式混乱 | 未遵循模板 | 客户投诉"不专业" |
| 风险遗漏 | 检查清单未执行 | 监管罚款 |

**教训：** 输出范式不是"形式主义"，而是"质量保险"。

---

### 给小白的一句话

> 输出范式就是金融服务的"产品包装标准"——想象你买 iPhone，无论在哪里的苹果店，包装盒的质感、配件的摆放、说明书的排版都一样。这种"一致性"让你感觉到"这是大品牌、值得信赖"。金融服务的输出范式也是同样的道理：研究报告有固定格式、财务模型有颜色编码、演示文稿有排版规范。这些范式不仅提升"专业感"，更重要的是确保"质量可控"——不会因为今天换了一个分析师，报告质量就忽高忽低。
