# 输出范式

当技能需要产出一致且高质量的结果时，请采用这些范式。

## 模板模式 (Template Pattern)

为输出格式提供模板。根据您的需求设定相应的规范力度。

**针对严格要求（如 API 响应或数据格式）：**

```markdown
## 报告框架

请务必遵循以下模板结构：

# [分析标题]

## 执行摘要
[对核心发现的单段综述]

## 核心洞察
- 发现 1（包含支撑数据）
- 发现 2（包含支撑数据）
- 发现 3（包含支撑数据）

## 改进建议
1. 具体且具备可行性的操作建议
2. 具体且具备可行性的操作建议
```

**针对柔性引导（当适应性调整更有利时）：**

```markdown
## 报告框架

以下是推荐的默认格式，请根据实际情况灵活调整：

# [分析标题]

## 执行摘要
[概览]

## 核心洞察
[根据您的发现调整各部分内容]

## 改进建议
[针对具体语境量身定制]

请根据特定分析类型的需求，对章节进行必要调整。
```

## 示例模式 (Examples Pattern)

若技能的输出质量依赖于对范例的理解，请提供“输入/输出”对：

```markdown
## 提交信息格式

请参考以下示例生成 Git 提交信息（Commit Message）：

**示例 1：**
输入：添加了使用 JWT 令牌的用户身份验证
输出：
```
feat(auth): implement JWT-based authentication

Add login endpoint and token validation middleware
```

**示例 2：**
输入：修复了报告中日期显示不正确的缺陷
输出：
```
fix(reports): correct date formatting in timezone conversion

Use UTC timestamps consistently across report generation
```

请遵循此风格：类型(作用域): 简要描述，随后是详细说明。
```

相比于单纯的文字描述，示例能帮助 Claude 更清晰地理解所需的文风与细节精度。

> **💡 Appendix: 领域知识小贴士**
> 
> **1. 执行摘要 (Executive Summary)**：想象你是一位日理万机的基金经理，没时间翻阅几百页的年报。执行摘要就是报告的“精华浓缩版”，旨在让你在电梯间的几十秒内，就能精准捕捉到最核心的投资逻辑和风险点。
> 
> **2. 核心洞察 (Key Findings)**：在金融分析中，数据本身是冰冷的，而洞察是有温度的。它不是简单地列出“股价跌了”，而是结合宏观环境和财务指标，分析出“为什么跌”以及“这种下跌是否反映了基本面”。它是从海量信息中捞出来的“真金”。
> 
> **3. 可行性建议 (Recommendations)**：好的金融建议不应该是“要注意风险”这种废话，而应该是“建议在 105 元分批建仓，止损位设在 98 元”。它必须像导航仪一样明确，让阅读者知道下一步该踩油门还是踩刹车。
