# xlsx-author

当在**无头模式**（托管代理 / CMA 模式）下运行，且需要交付 Excel 工作簿作为**文件产物**，而非通过 `mcp__office__excel_*` 工具编辑实时工作簿时，请使用此技能。

## 输出规范

- 写入至 `./out/<name>.xlsx`。如果 `./out/` 文件夹不存在，请自行创建。
- 在最终消息中返回相对路径，以便编排层（Orchestration Layer）提取文件。

## 如何构建工作簿

编写一段简短的 Python 脚本并使用 Bash 运行。请调用 `openpyxl` 库：

```python
from openpyxl import Workbook
from openpyxl.styles import Font, PatternFill

wb = Workbook()
ws = wb.active; ws.title = "Inputs"
ws["B2"] = "Revenue"; ws["C2"] = 1_250_000_000
ws["C2"].font = Font(color="0000FF")           # 蓝色 = 硬编码输入值
calc = wb.create_sheet("DCF")
calc["C5"] = "=Inputs!C2*(1+Inputs!C3)"        # 黑色 = 计算公式
wb.save("./out/model.xlsx")
```

## 建模惯例（参考 `audit-xls` 标准）

- **蓝/黑/绿配色方案：** 蓝色代表硬编码输入值，黑色代表计算公式，绿色代表链接至其他工作表或文件的引用。
- **计算单元格禁止硬编码：** 所有执行计算的单元格必须使用公式；所有原始输入数据必须统一存放于“Inputs”页签。
- **命名区域（Named Ranges）：** 对于投行简报（Deck）或投资备忘录（Memo）中引用的任何数值，请务必定义命名区域。
- **平衡检查（Balance Checks）：** 包含一个“Checks”页签，用于自动核对勾稽关系（如资产负债表平衡、现金流与现金余额匹配等），并显示 TRUE/FALSE 结果。
- **单文件单模型：** 每个文件仅存放一个财务模型。除非有明确指令，否则不要在现有工作簿中追加内容。

## 何时不宜使用

如果当前环境可以调用 `mcp__office__excel_*` 系列工具（协作插件模式），请优先使用这些工具 — 它们能直接操作用户的实时工作簿并支持人工审查。本技能仅作为无头运行模式下生成物理文件的备选方案。

### 1. 为什么投行用代码生成 Excel 而不是手动填表——"流水线 vs 手工作坊"

在普通人眼中，Excel 就是"打开 → 填数字 → 保存"。但在投行和 PE，越来越多的 Excel 是通过 Python（openpyxl 库）用代码"打印"出来的。这不是炫技，而是生存必需。

**用代码生成 Excel 的三个核心优势**：

**① 100% 一致性（Consistency）**：
- 手动做 10 个 Excel，同一个数字可能因为"手误"而不一样
- 代码生成 1000 个 Excel，每一个都完全一样——因为代码不会"走神"
- 这对于客户交付来说至关重要——你不能给两个客户发"两个版本"的数据

**② 可复现性（Reproducibility）**：
- 三个月后，老板说："上次的 Excel 我想改一个假设，重新算一遍"
- 如果手动做的 → 你需要重建一切（可能已经忘了当初怎么做的）
- 如果代码生成的 → 改一行代码，重新运行 → 3 秒出结果

**③ 版本控制（Version Control）**：
- 手动 Excel 的版本控制是："model_v1.xlsx"、"model_v2.xlsx"、"model_v3_final.xlsx"、"model_v3_final_REAL.xlsx"
- 代码生成的版本控制是：用 Git 追踪代码变化 → 谁改了什么、什么时候改的、为什么改，全部有记录

**打比方——做汉堡**：
- 手动做 Excel = 你亲自去厨房，拿面包、放肉饼、加生菜、挤酱 → 每次做出的汉堡都有细微的差异
- 代码生成 Excel = 麦当劳生产线 → 100 万个汉堡完全一样的规格、一样的味道、一样的重量
- 投行需要的就是"麦当劳式"的可靠性和一致性

**无头模式（Headless Mode）的含义**：
- 在云服务器上运行时，没有 Excel 桌面界面可以操作
- 你不能"打开 Excel → 全选 → 改字体"——因为根本没有 Excel 应用
- 你必须用 Python 代码直接操作 .xlsx 文件的底层结构
- **好处**：适合自动化流水线——服务器 24 小时运行，不需要人工介入

**所以这意味着什么**：本技能的核心就是——用 Python（openpyxl）在后台"默默"构建 Excel，最终交付一个 .xlsx 文件，而不是让你在 Excel 里手动操作。

### 2. 蓝/黑/绿配色规则——"华尔街的 Excel 军规"

在投行的 Excel 模型中，单元格颜色有严格的含义。这不是"好看"的问题，而是"能不能被审计"的问题。颜色是信息载体——别人一看颜色就知道"这个数字是怎么回事"。

| 颜色 | 含义 | 规则 |
|------|------|------|
| 🔵 **蓝色** | 硬编码输入（Hardcoded Input） | 你在假设页手动填的数字——比如"营收增长率 5%" |
| ⚫ **黑色** | 公式计算（Formula/Calculation） | 由公式算出来的数字——比如 `=B2*(1+C2)` |
| 🟢 **绿色** | 跨表链接（Cross-sheet Link） | 引用了其他工作表或外部文件的数字 |

**为什么这"三板斧"这么重要**：

**① 可审计性**：任何人打开模型，一看颜色就知道：
- 蓝色 → "这个数字是假设，我可以改"
- 黑色 → "这个数字是算出来的，要改的是它的来源"
- 绿色 → "这个数字来自别的地方，我要去那边找源头"

如果一个 Excel 全是黑色（没有区分输入和计算），审计员需要花几天时间搞清楚"哪个是输入、哪个是输出"。但如果颜色规范 → 审计时间从几天缩短到几小时。

**② 防错机制**：最常见的 Excel 错误是"不小心把公式改成了数字"（硬编码覆盖）。但有颜色规范后，如果你看到一个黑色单元格里出现了数字（而不是公式），你就知道"有人手动覆盖了公式"——这是模型中的"定时炸弹"。

**③ 团队协作**：一个大模型可能有 10 个人同时工作。如果每个人都按自己的习惯配色 → 混乱。但如果所有人遵守蓝/黑/绿规则 → 任何人打开别人的 Excel，都能立刻读懂。

**历史上因为这个规则产生的灾难**：
某投行分析师在 DCF 模型的终端增长率里，把公式 `=Assumptions!B5`（绿色链接）误改成了 `0.03`（硬编码）。因为 0.03 是黑色字体（违反颜色规则），其他人没有注意到。结果估值差了 $5B——模型发给了客户，客户发现了这个错误。投行丢了这笔交易。

**所以这意味着什么**：本技能要求"蓝色代表硬编码输入值，黑色代表计算公式，绿色代表链接至其他工作表或文件的引用"——这不是"建议"，而是"铁律"。不遵守这个规则的 Excel，在投行眼里就是"废纸"。

### 3. Checks 工作表——"财务模型的'安全带'"

**Checks（检查 / 勾稽）** 工作表是财务模型中最重要的"非业务"工作表——它不参与估值计算，但它是模型的"安全带"。

**为什么需要 Checks 工作表**：
- 财务模型有成千上万个公式 → 一个地方出错，整个模型就错了
- 你不能"靠眼睛"找出错误 → 5 万行的模型，眼睛看不完
- Checks 工作表自动检查模型的关键"承重点"是否成立 → 如果不成立，立刻报警

**Checks 工作表的经典检查项**：

**① 资产负债表配平检查（Balance Sheet Balance Check）**：
```
=IF(ABS(Total_Assets - Total_Liabilities - Total_Equity) < 0.01, "OK", "UNBALANCED!")
```
- 资产必须等于负债+权益 → 如果不等，模型有漏洞
- 检查每一期（通常 5-10 年）→ 每一期都必须配平
- 如果某一年不配平 → 那一年的模型有问题

**② 现金钩稽检查（Cash Flow Tie Check）**：
```
=IF(ABS(CF_Ending_Cash - BS_Cash_and_Equivalents) < 0.01, "OK", "MISMATCH!")
```
- 现金流量表算出的"期末现金"必须等于资产负债表上的"现金及现金等价物"
- 如果不一致 → 三张表之间的链接断了

**③ 折旧一致性检查（Depreciation Consistency Check）**：
```
=IF(ABS(CF_Depreciation - IS_Depreciation) < 0.01, "OK", "MISMATCH!")
```
- 现金流量表的"折旧"必须等于利润表的"折旧"
- 如果不一致 → 折旧被加了两次或者漏加了

**④ 公式一致性检查（Formula Consistency Check）**：
```
=IF(AND(C5:C20 = D5:D20), "OK", "FORMULA BROKEN!")
```
- 同一行的公式应该是"向右拖动"的结果 → 如果中间一个单元格的公式不一样，有问题

**打比方——Checks 工作表就像汽车的"仪表盘"**：
- 速度表（Checks）告诉你速度是多少（模型是否正常）
- 如果发动机故障灯亮了（Checks 显示 "UNBALANCED!"），你就不能继续开（不能依赖这个模型的结果）
- 没有 Checks 的模型 = 没有仪表盘的汽车 → 你根本不知道它是不是"要爆炸了"

**所以这意味着什么**：本技能要求"包含一个 Checks 页签，用于自动核对勾稽关系"——因为一个没有 Checks 的财务模型，在专业投资者眼里是不可信任的。

### 4. 命名区域（Named Ranges）——"给 Excel 单元格起名字"

**命名区域（Named Range）** 是 Excel 中一个被严重低估的功能——给一个单元格或一组单元格起一个有意义的名字，然后在整个工作簿中通过名字引用它。

**为什么不直接用单元格地址（如 C5）**：
```excel
= Inputs!C5 * (1 + Inputs!C6)   ← 坏！看不懂 C5 和 C6 是什么
= Revenue_Growth * (1 + Margin)  ← 好！一看就知道在算什么
```

**命名区域在大型模型中的价值**：
- 模型有 5 万行 → 你不可能记住 "Inputs!C5 是营收增长率"
- 但如果它叫 `Revenue_Growth` → 任何人在任何地方引用它，一看就知道
- 对于审计员来说：看到 `=Revenue_Growth * Base_Revenue` → 5 秒理解逻辑
- 看到 `=Inputs!C5 * Inputs!C6` → 必须跳回 Inputs 工作表查 C5 和 C6 是什么

**投行和 PE 的命名惯例**：
- 所有命名使用英文（方便国际团队协作）
- 用下划线分隔单词（`Revenue_Growth`，不是 `RevenueGrowth`）
- 避免空格（Excel 会在空格处断开）
- 统一使用 Pascal_Snake_Case 或 UPPER_SNAKE_CASE

**所以这意味着什么**：本技能要求"对于投行简报或投资备忘录中引用的任何数值，请务必定义命名区域"——因为命名区域让 Excel 从"一堆格子"变成了"可读的文档"。

### 5. 单文件单模型原则——"不要把鸡蛋放在一个篮子里"

**单文件单模型（One Model Per File）** 是投行建模的铁律——每个 Excel 文件只能包含一个财务模型。

**为什么不能在一个 .xlsx 里放多个模型**：
- **审计困难**：审计员不知道哪个模型是"主模型" → 增加审计时间 3 倍+
- **版本混乱**：改了一个模型，另一个可能被不小心改到 → 两个都错了
- **性能下降**：多个模型在一个文件 → 公式计算变慢 → 客户拿到模型打开要等 5 分钟
- **链接脆弱**：模型之间的链接如果用了绝对路径 → 发给别人后链接全部断裂

**投行的文件命名习惯**：
```
Project_Apple_DCF_v1.xlsx        ← DCF 模型
Project_Apple_LBO_v1.xlsx        ← LBO 模型
Project_Apple_Comps_v1.xlsx      ← 可比公司分析
```
而不是：
```
Project_Apple_All_Models_v1.xlsx  ← ❌ 所有模型挤在一起
```
