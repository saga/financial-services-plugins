# 数据清洗

清洗当前活动工作表或指定范围内的混乱数据。

## 环境

- **若在 Excel 内部运行（Office 加载项 / Office JS）：** 直接使用 Office JS（`Excel.run(async (context) => {...})`）。通过 `range.values` 读取数据，通过 `range.formulas = [["=TRIM(A2)"]]` 写入辅助列公式。关于“原位操作”还是“使用辅助列”的决策逻辑依然适用。
- **若对独立 .xlsx 文件进行操作：** 使用 Python/openpyxl。

## 工作流程

### 步骤 1：确定范围

- 若已由用户给定范围（如 `A1:F200`），则直接使用。
- 否则，使用活动工作表中所有已使用的范围（Full used range）。
- 对每列进行画像分析：检测其主导数据类型（文本 / 数值 / 日期）并识别离群值。

### 2. 问题检测

| 问题 | 检测要点 |
|---|---|
| 空格 | 首尾空格、冗余的双空格 |
| 大小写 | 分类列中大小写不一致（如 `usa` / `USA` / `Usa`） |
| 文本存储的数字 | 以文本格式存储的数值；数值单元格中夹杂的 `$`, `,`, `%` 等符号 |
| 日期 | 同一列中存在混合格式（如 `3/8/26`, `2026-03-08`, `March 8 2026`） |
| 重复项 | 完全重复的行，以及近似重复（仅大小写或空格差异）的条目 |
| 空白值 | 在有数据的列中偶然出现的空单元格 |
| 混合类型 | 某列中 98% 为数字但包含 3 条文本条目的情况 |
| 编码 | 乱码字符（如 `Ã©`, `â€™`）、不可打印字符 |
| 错误 | `#REF!`, `#N/A`, `#VALUE!`, `#DIV/0!` 等 Excel 报错 |

### 步骤 3：提出建议

在修改任何内容前，展示一份摘要表：

| 列 | 问题 | 数量 | 建议修正措施 |
|---|---|---|---|

### 步骤 4：执行应用

- **优先使用公式而非写死清洗后的数值** —— 只要清洗结果可以通过公式表达（例如 `=TRIM(A2)`, `=VALUE(SUBSTITUTE(B2,"$",""))`, `=UPPER(C2)`, `=DATEVALUE(D2)`），就应在相邻的辅助列中写入公式，而非在 Python 中计算结果后覆盖原始数据。这能确保数据转换过程透明且可审计。
- 仅在用户明确要求或无法通过简单公式实现（例如修复编码/乱码）时，才采用计算值进行原位覆盖。
- 对于破坏性操作（如删除重复项、填充空白值、覆盖原始数据），必须先征得用户确认。
- 在完成每一类修复（空格 → 大小写 → 数值转换 → 日期 → 去重）后，向用户展示变更示例并获得确认，然后再执行下一类。
- 汇报操作前后的变更摘要。

> **💡 Appendix: 领域知识小贴士**
>
> 在金融世界里，数据就像是金钱，但“脏数据”就像是掺了沙子的金币。
>
> 1.  **为什么金融分析师讨厌“文本数字”？**
>     在 Excel 中，如果一个数字被存成了“文本”（单元格左上角会有个彩色小三角），所有的求和函数（SUM）和财务模型都会把它当成 0。这就像在算账时漏掉了一张关键的支票。
> 2.  **什么是“审计追踪”（Audit Trail）？**
>     这是金融合规的核心。如果我们直接把原始数据改了，别人就没法确认我们的清洗逻辑是否正确。所以我们更倾向于在旁边加一个“辅助列”写公式。这就像是在草稿纸上写明推导步骤，不仅透明，还方便后期核查。
> 3.  **日期的“跨国翻译”**
>     不同国家的日期写法天差地别：有些是“月/日/年”，有些是“日/月/年”。在处理跨国财务报表时，如果不进行“标准化”，可能会把 3 月 8 日错当成 8 月 3 日，导致利息计算或报告时序彻底混乱。
