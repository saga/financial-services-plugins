# 差异归因溯源

给定单个差异行（包含键值、总账金额、明细账金额、分类桶及疑似原因），通过溯源生成根本原因说明。

## 溯源路径

1. **获取总账（GL）端数据** —— 通过 internal-gl MCP 获取生成该总账行的日记账分录或过账记录：分录 ID、过账日期、源系统、批次 ID、编制人。
2. **获取明细账（Subledger）端数据** —— 通过 subledger MCP 获取匹配的交易信息：交易 ID、交易日/结算日、交易对手、源数据流、所用汇率。
3. **属性比对** —— 对齐过账日期、汇率及汇率日期、科目映射、数量正负号、金额正负号。属性不一致的地方通常就是原因所在。

## 原因 → 陈述建议

将根本原因编写为一句话陈述，格式为：**“⟨某侧⟩ ⟨做了什么动作⟩，因为 ⟨原因⟩”**，例如：

- “总账在结算日 (T+2) 过账，而明细账在交易日过账 —— 属于时间性差异，将于 2026-05-07 消除。”
- “明细账使用了 WM/R 16:00 汇率，而总账使用了彭博收盘汇率 —— 导致基准金额产生 12 个基点 (bps) 的汇率差异。”
- “证券 ABC123 在映射表中指向总账科目 11420，但明细账传输的是 11410 —— 属于映射差异，请提报至参考数据部门。”
- “明细账重复记录了该交易（交易 ID 88412 和 88419 为重复项） —— 属于重复过账，请剔除 88419。”

## 输出格式

对于每个完成溯源的差异，返回以下內容：

```json
{
  "key": "...",
  "root_cause": "上述格式的一句话说明",
  "owner": "ops | reference-data | accounting | upstream-system",
  "expected_clear_date": "YYYY-MM-DD 或 null",
  "action": "monitor | adjust | raise-ticket | suppress"
}
```

只有调整程序（Resolver）负责写入调整分录 —— 本技能仅负责诊断，不负责过账。

> **💡 Appendix: 领域知识小贴士**
>
> 这里的“对账”就像是在核对你的银行账单和自己的记账本。
> 1. **什么是 Break（差异/挂账）？** 当你的“总账”（大盘账）和“明细账”（每一笔小账）对不上号时，就会产生一个 Break。它说明两边的钱数或笔数没对齐。
> 2. **总账 vs 明细账 (GL vs Subledger)：** “总账”是财务部的全盘统计，“明细账”是业务部门（比如交易台或仓库）记录的每一笔买卖。
> 3. **时间性差异 (Timing Break)：** 最常见的一种“对不上”。比如你今天在网上买了东西（交易日），但银行后天才扣款（结算日）。这两天之间，你的记账本和银行余额就会出现临时的不对称。
> 4. **映射差异 (Mapping Break)：** 系统里有个“翻译官”（映射表），负责把业务数据翻译成会计科目。如果翻译官把“铅笔”翻译成了“固定资产”而不是“办公用品”，账目就会出错。
> 5. **审计跟踪 (Audit Trail)：** 就像侦探破案。从差异的结果开始，根据证据（ID、日期、批次）一步步往回找，直到发现最初是哪一步录入错误或时间没对上。

## Appendix: 金融背景知识

这份文件是"差异归因溯源（Break Trace）"技能的详细说明。在基金行政中，当 GL Recon 发现了差异（Break）之后，下一步就是"溯源"——从差异项一路追溯回原始交易记录，找到差异产生的"第一现场"。

---

### 1. 什么是 Break Trace？

**类比：**
想象你是一个侦探。银行报表和对账单对不上，差了 5 万元。你不能直接说"哦，差 5 万"——你要追查：
1. 这 5 万是哪笔交易产生的？
2. 是什么时候产生的？
3. 为什么两边不一样？（汇率？时间差？人为错误？）

Break Trace 就是基金会计的"侦探工作"——从差异追溯到根源。

---

### 2. Break Trace 的"溯源路径"

```
差异行（Break Item）
  ↓ 追踪 GL 端
  找到日记账分录（JE）
  → 分录 ID、过账日期、源系统、批次 ID、编制人
  ↓ 追踪子账端
  找到原始交易（Trade）
  → 交易 ID、交易日/结算日、交易对手、汇率
  ↓ 属性比对
  - 过账日期是否一致？
  - 汇率及汇率日期是否一致？
  - 科目映射是否一致？
  - 正负号是否一致？
  - 金额是否一致？
  ↓ 
根因 → 一句话陈述
```

---

### 3. Break 的"五大根因"

| 根因类型 | 典型陈述 | 处理方式 |
|---------|---------|---------|
| 时间差异 | "GL 在 T+2 过账，子账在 T 日记账——时间性差异，将于 X 日消除" | 等待自动消除 |
| 汇率差异 | "子账用了 WM/R 16:00 汇率，GL 用了彭博收盘汇率——汇率差异 12bps" | 统一汇率标准 |
| 映射差异 | "证券 ABC 映射到 GL 11420，但子账传的是 11410——映射错误" | 修正参考数据 |
| 重复过账 | "子账重复记录了交易 ID 88412 和 88419——重复过账" | 剔除重复项 |
| 手动错误 | "GL 手动调整金额时多输了一个 0——人为错误" | 冲正+重做 |

---

### 4. 一句话陈述的格式

**标准格式：** "⟨某侧⟩ ⟨做了什么动作⟩，因为 ⟨原因⟩"

| 例子 |
|------|
| "子账在结算日过账，而 GL 在交易日过账——属于正常时间性差异。"
| "GL 使用了 manual override 调整汇率，而子账使用系统自动汇率——需确认调整原因。"
| "子公司账户未及时向总账系统传输数据——系统延迟导致。"
| "交易员在子账中修改了交易价格，但 GL 未同步更新——需补录调整分录。"

---

### 5. 真实案例：一个"幽灵差异"的溯源

**背景**：某基金 GL 与子账差 $50,000，所有常规检查都找不到原因

| 溯源步骤 | 发现 |
|---------|------|
| 追踪 GL 端 | 一笔 $50,000 的调整分录，编制人"system"，来源"auto-reversal" |
| 追踪子账端 | 无对应记录 |
| 扩大搜索范围 | 发现上个月有一笔 $-50,000 的临时调整 |
| 根因 | 上月的临时调整设置了自动冲回，但原始交易已永久修正——导致反向调整多余 |

**教训**：系统自动逻辑有时比手动错误更隐蔽。

---

### 给小白的一句话

> Break Trace 就是基金会计界的"法医鉴定"——当 GL Recon 发现总账和子账对不上（好比死者身上的伤痕），Break Trace 就要像法医一样从"伤口"一路追溯到"凶器"（原始交易记录），找出"死因"（差异根因）。这个过程有时很简单（汇率用错了），有时很复杂（一个系统 Bug 在 3 个月前埋下的伏笔）。但无论如何，找到根因之前不能轻易说"算了、差不多"——因为一个 $50,000 的小差异背后，可能藏着 $500 万的大问题。
