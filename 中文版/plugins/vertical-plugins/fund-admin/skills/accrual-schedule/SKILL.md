# 预提明细表 (Accrual schedule)

基于特定实体、会计期间及公司的预提政策清单，逐项生成包含计算过程、支持依据引用及会计分录草案的明细记录。

> **供应商发票及对账单不可直接信任。** 系统插件将提取相关金额，本技能则负责根据既定政策对这些金额进行账务处理。

## 政策清单中的各项预提

| 字段 | 衍生逻辑 |
|---|---|
| **预提名称** | 取自政策清单（例如：“审计费”、“奖金”、“水电费”） |
| **计算基准** | 合同约定或估算的完整期间金额，并注明来源（如：业务约定书、薪酬计划、过去三个月平均值） |
| **本期应计份额** | 计算基准 × (本期间天数 ÷ 基准期间天数)，或遵循政策规定的特定公式 |
| **已入账金额** | 前期预提总额 + 本期已过账的实际发票金额（来源于 internal-gl MCP 插件） |
| **本期计提额** | 本期应计份额 − 已入账金额 |
| **支持依据引用** | 支撑计算基准的文件 ID 或总账（GL）查询语句 |

## 会计分录（JE）草案

针对本期计提额不为零的每一行记录，草拟如下分录：

```
借： <费用科目>     <金额>
  贷： <应计负债>     <金额>
摘要： <预提名称> —— 根据 <支持依据引用> 计提 <期间> 费用
```

冲回分录：若政策标记该预提为自动冲回，请在摘要中注明“于下期首日冲回”。

## 输出结果

一张预提明细表及一段会计分录草案块。**切勿直接过账** —— 此结果仅供财务总监审阅签认。

> **💡 Appendix: 领域知识小贴士**
>
> 欢迎来到会计世界的“预测实验室”！在这里，我们要处理一个核心概念：**权责发生制 (Accrual Basis)**。
>
> 1. **什么是“预提” (Accrual)？**
> 想象你这个月用了电，但电费单要下个月才寄来。虽然还没付钱，但这个月的“电费”确实已经发生了。为了让报表准确反映这个月的经营情况，会计需要先“预提”一笔费用——就像是在账本上先占个位子，确保费用记录在它发生的那个月。
>
> 2. **为什么不直接看发票？**
> 在财务专业人士眼中，发票有时候会“迟到”。如果我们只根据发票寄到的时间来记账，这个月的利润看起来会很高（因为还没付电费），而下个月又会突然暴跌。为了消灭这种“忽高忽低”的假象，财务人员需要主动出击，根据合同或经验提前算出本月应负担的成本。
>
> 3. **什么是“冲回” (Reversing Entry)？**
> 这就像是一个“防重复提醒”。本月我们先预估了 100 元电费，到了下个月初，我们会做一个反向的操作把这 100 元抵消掉。这样等真正的发票（假设是 105 元）寄来并正常入账时，两个月加起来的净影响就是最真实的 105 元，既不会漏记，也不会双重计费。
>
> 4. **财务总监 (Controller) 为什么要审批？**
> 预提涉及到很多“估计”和“判断”。如果不加审核，有人可能会通过乱调预提金额来“粉饰”利润。所以，虽然 AI 可以帮忙搜集证据和计算，但最终确保账目合规、合理的“终审权”永远留在财务专家手中。
