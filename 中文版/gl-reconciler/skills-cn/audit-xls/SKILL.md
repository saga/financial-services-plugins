# 审计电子表格 (Audit Spreadsheet)

审计公式和数据以确保准确性并发现错误。审核深度由范围决定——从针对选定区域的快速公式检查，到完整的财务模型完整性审计。

## 第一步：确定审计范围

如果用户已指定范围，请直接使用。否则**询问用户**：

> 您希望执行什么范围的审计？
> - **选定区域 (selection)** — 仅针对当前选中的单元格范围
> - **当前工作表 (sheet)** — 仅针对当前的活动工作表
> - **模型 (model)** — 整个工作簿，包括财务模型完整性检查（资产负债表平衡、现金勾稽、科目滚算、逻辑合理性）

**模型**范围是深度最细致的——在将现金流折现模型 (DCF)、杠杆收购模型 (LBO)、三表模型、并购模型、可比公司分析 (Comps) 或任何综合财务模型发送给客户或投资委员会 (IC) 之前，请务必使用此范围。

---

## 第二步：公式级检查（适用于所有范围）

无论范围如何，均执行以下检查：

| 检查项 | 关注重点 |
|---|---|
| 公式错误 | `#REF!`, `#VALUE!`, `#N/A`, `#DIV/0!`, `#NAME?` |
| 公式内硬编码 | 例如 `=A1*1.05` — 其中的 `1.05` 应当是一个单元格引用 |
| 不一致的公式 | 某个公式破坏了所在行或列邻近单元格的计算模式 |
| 范围错位 | `SUM`/`AVERAGE` 等函数遗漏了首行、末行或首列、末列 |
| 被覆盖的公式 | 单元格看起来像公式，但实际上被改写成了硬编码数值 |
| 循环引用 | 无论是有意还是无意产生的 |
| 断开的跨表链接 | 引用了已被移动或删除的单元格 |
| 单位/量级不匹配 | 千位与百万位混用，或百分比被存为整数 |
| 隐藏行/页签 | 可能包含被覆盖的数值或过时的计算逻辑 |

## 第三步：模型完整性检查（仅限模型范围）

如果范围是**模型**，请识别模型类型（DCF / LBO / 三表模型 / 并购 / 可比分析 / 自定义），并运行下述相应的完整性检查。

### 3a. 结构审核

| 检查项 | 关注重点 |
|---|---|
| 输入与公式分离 | 输入值是否与计算逻辑清晰分开？ |
| 配色规范 | 是否遵循“蓝色=输入，黑色=公式，绿色=链接”或其他一致的配色方案？ |
| 页签逻辑 | 页签顺序是否逻辑清晰（假设 → 利润表 → 资产负债表 → 现金流量表 → 估值）？ |
| 日期页眉 | 所有页签的日期轴是否一致？ |
| 单位 | 全表单位是否统一（千位 vs 百万位 vs 绝对数值）？ |

### 3b. 资产负债表 (Balance Sheet)

| 检查项 | 测试方法 |
|---|---|
| 资产负债表平衡 | 每个期间：总资产 = 总负债 + 股东权益 |
| 留存收益滚算 | 期初留存收益 + 净利润 - 分红 = 期末留存收益 |
| 商誉/无形资产 | 是否来源于并购假设（若是并购模型） |

如果资产负债表不平衡，请**量化每个期间的差异并追踪断裂点**——在修复此问题之前，其他任何检查都毫无意义。

### 3c. 现金流量表 (Cash Flow Statement)

| 检查项 | 测试方法 |
|---|---|
| 现金余额勾稽 | 每个期间：现金流量表期末现金 = 资产负债表现金科目 |
| 现金流加总 | 经营活动现金流 (CFO) + 投资活动现金流 (CFI) + 筹资活动现金流 (CFF) = 现金变动净额 |
| 折旧摊销 (D&A) 匹配 | 现金流量表中的 D&A = 利润表中的 D&A |
| 资本支出 (CapEx) 匹配 | 现金流量表中的 CapEx = 资产负债表中固定资产 (PP&E) 的变动 |
| 营运资本 (WC) 变动 | 正负号是否与资产负债表变动一致（应收账款 $\Delta AR$、应付账款 $\Delta AP$、存货 $\Delta Inventory$） |

### 3d. 利润表 (Income Statement)

| 检查项 | 测试方法 |
|---|---|
| 收入拆解 | 收入总额是否与细分部门/产品明细项匹配 |
| 税项 | 所得税费用 = 税前利润 $\times$ 税率（允许递延税项调整） |
| 股份数 | 是否与稀释明细表一致（期权、可转债、回购） |

### 3e. 循环引用 (Circular References)

- 在 LBO 或三表模型中，利息 → 债务余额 → 现金 → 利息形成的循环引用通常是有意为之。
- 如果是有意的：确认是否存在“迭代计算开关”且能正常运作。
- 如果是无意的：追踪闭环逻辑并标记如何打破循环。

### 3f. 逻辑与合理性

| 检查项 | 触发预警的情况 |
|---|---|
| 增长率 | 营收增长率 >100% 且无合理解释 |
| 利润率 | 明显超出行业常规水平 |
| 终值占比 | 终值 (Terminal Value) 占 DCF 企业价值 (EV) 的比例 >~75% |
| 曲棍球杆式增长 | 预测年份后期出现不切实际的爆发式增长 |
| 复利效应 | EBITDA 在第 10 年因复利计算出天文数字 |
| 极端情况 | 模型在增长率为 0% 或负值、EBITDA 为负、杠杆率为负时崩溃 |

### 3g. 特定模型类型的常见错误

**现金流折现模型 (DCF):**
- 折现率应用期间错误（年中折现 vs 年末折现）
- 终值未折现回当前价值
- WACC 使用账面价值而非市值
- 自由现金流 (FCF) 包含了利息支出（应使用无杠杆现金流）
- 税盾 (Tax Shield) 被双重计入

**杠杆收购模型 (LBO):**
- 债务偿还额度与现金清扫 (Cash Sweep) 机制不符
- PIK 利息（实物支付利息）未计入本金滚算
- 管理层滚投权益未在收益中体现
- 退出倍数应用了错误的 EBITDA（过往十二个月 LTM vs 未来十二个月 NTM）
- 初始股权投资中未扣除各项手续费/开支

**并购模型 (Merger):**
- 增厚/稀释分析使用了错误的股数（交易前 vs 交易后）
- 协同效应未按阶段逐步录入
- 购买价格分配 (PPA) 不平衡
- 未计入因动用现金而产生的利息损失
- 资金来源与用途 (Sources & Uses) 中漏掉交易费用

**三表模型 (3-statement):**
- 营运资本变动的正负号错误
- 折旧额与固定资产 (PP&E) 明细表不符
- 债务到期计划与本金偿付金额不匹配
- 分红在无解释的情况下超过了净利润

## 第四步：输出报告

输出发现项表格：

| # | 工作表 | 单元格/范围 | 严重程度 | 类别 | 问题描述 | 建议修复方案 |
|---|---|---|---|---|---|---|

**严重程度说明：**
- **致命 (Critical)** — 结果错误（资产负债表不平、公式损坏、现金勾稽失败）
- **警告 (Warning)** — 存在风险（硬编码、公式不一致、极端场景失效）
- **提示 (Info)** — 格式/最佳实践（配色规范、布局、命名方式）

对于**模型**范围，请在开头添加汇总行：

> 模型类型：[DCF/LBO/三表/...] — 总体评估：[整洁 / 存在微小问题 / 存在重大问题] — [N] 项致命, [N] 项警告, [N] 项提示

**未经允许请勿修改任何内容** — 先报告，在收到指令后再进行修复。

## 注意事项

- **资产负债表平衡优先** — 如果不平衡，下游的所有计算结果都值得怀疑。
- **硬编码覆盖是“隐形错误”的头号来源** — 请进行地毯式搜索。
- **正负号惯例错误**（如将现金流出错误记为正数）极其常见。
- 如果模型涉及 VBA 宏，请注明哪些通过宏实现的计算无法通过公式直接审计。

> **💡 Appendix: 领域知识小贴士**
>
> 即使你是财务小白，了解以下几个核心概念也能让你瞬间看懂财务模型的精髓：
>
> 1.  **资产负债表平衡 (BS Balance)**：想象你在玩平衡木。左边是公司拥有的东西（资产，如现金、厂房），右边是这些东西是怎么来的（借的钱叫负债，自己的钱叫股东权益）。左右永远相等是财务学的“第一定律”。
> 2.  **勾稽 (Tie-out)**：这是财务圈的专业黑话，意思就是“对账”。比如公式算出来的期末有多少钱，必须和报表里写的数对得上，哪怕错了一分钱，整个模型也算是不及格。
> 3.  **硬编码 (Hardcoding)**：指在算式里直接填数字（比如 `=A1*1.05`）而不是引用单元格。这在金融圈是大忌！因为没人知道那个 `1.05` 是从哪拍脑门想出来的，且以后修改时极易被漏掉，像模型里的“定时炸弹”。
> 4.  **循环引用 (Circular Reference)**：A 的计算需要 B，B 的计算又需要 A。在财务模型中，利息支出减少利润，利润减少现金，现金少了吧，借钱就多，借钱一多利息又变高了——这就是一个合法的闭环，Excel 需要开启“迭代计算”才能算清这笔账。
> 5.  **DCF (现金流折现)**：一种估值方法。核心逻辑是：今天的 100 块比明年的 100 块值钱。把公司未来几十年能赚的钱全都“折价”算成今天的价值，加起来就是这个公司现在的身价。
