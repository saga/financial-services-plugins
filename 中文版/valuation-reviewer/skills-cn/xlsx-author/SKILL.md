# xlsx-author

在**无头（headless）**模式（托管代理 / CMA 模式）下运行，且需要交付 Excel 工作簿作为**文件产物**，而非通过 `mcp__office__excel_*` 编辑实时工作簿时，请使用此技能。

## 输出约定

- 写入至 `./out/<name>.xlsx`。如果 `./out/` 目录不存在，请先创建。
- 在最终消息中返回相对路径，以便编排层进行文件收集。

## 如何构建工作簿

编写一段简单的 Python 脚本并通过 Bash 运行。使用 `openpyxl` 库：

```python
from openpyxl import Workbook
from openpyxl.styles import Font, PatternFill

wb = Workbook()
ws = wb.active; ws.title = "Inputs"
ws["B2"] = "Revenue"; ws["C2"] = 1_250_000_000
ws["C2"].font = Font(color="0000FF")           # 蓝色 = 硬编码输入
calc = wb.create_sheet("DCF")
calc["C5"] = "=Inputs!C2*(1+Inputs!C3)"        # 黑色 = 公式
wb.save("./out/model.xlsx")
```

## 建模规范（参考 `audit-xls`）

- **蓝 / 黑 / 绿。** 蓝色代表硬编码输入（Inputs），黑色代表公式（Formulas），绿色代表指向其他工作表或文件的链接。
- **计算单元格严禁硬编码。** 所有计算单元格必须为公式；所有输入变量应统一存放于“Inputs”页签。
- **命名区域（Named ranges）。** 凡是会被幻灯片（Deck）或备忘录（Memo）引用的数值，均需定义命名区域。
- **平衡校验（Balance checks）。** 包含一个“Checks”校验页签，用于核对勾稽关系（如资产负债表是否平衡、现金流量表是否与现金余额一致等），并直观显示 TRUE/FALSE 结果。
- **一文件一模型。** 除非有明确要求，否则不要向现有工作簿追加内容。

## 何时不用

如果 `mcp__office__excel_*` 工具可用（Cowork 插件模式），请优先使用这些工具 —— 它们可以驱动用户的实时工作簿并支持人工审核。本技能仅作为无头运行模式下生成文件的备选方案。

> **💡 Appendix: 领域知识小贴士**
> 
> **1. 为什么要区分“蓝色”和“黑色”？**
> 在华尔街标准的金融建模中，颜色是有“特殊含义”的。**蓝色**代表你手动填入的原始数据（比如去年的营收），**黑色**代表由公式自动算出来的结果。这样别人看你的表时，一眼就能知道哪些数字是可以改动的“变量”，哪些是绝对不能乱动的“逻辑”，这能极大地减少模型出错的概率。
> 
> **2. 什么是“勾稽关系”（Tie）？**
> 财务报表不是孤立的，它们之间像齿轮一样互相咬合。例如，资产负债表里的“现金”必须等于现金流量表最后算出来的“期末现金余额”。这种“报表之间必须严丝合缝”的关系就叫“勾稽”。在 Excel 里写一个简单的 `Check` 公式（比如：`=A1-B1=0`），如果结果是 FALSE，就说明模型内部的逻辑逻辑链条断了。
> 
> **3. 为什么不直接在计算格子里写数字（杜绝硬编码）？**
> 想象一下，如果你在 100 个计算公式里都直接写了“5%”这个增长率，一旦老板想看“6%”的情况，你就得手动改 100 次，而且极易漏掉几个。专业的做法是把“5%”放在专门的 Inputs 页签的一个格子里，所有公式都引用这个格子。这样你只需要改这一个地方，整个模型就会瞬间自动更新。这就是“牵一发而动全身”的建模艺术。
