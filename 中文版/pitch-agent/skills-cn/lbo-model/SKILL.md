## 克劳德 (CLAUDE) 核心指令 - 请先阅读

### 环境：Office JS 还是 Python

**如果在 Excel 内部运行（Office 加载项 / Office JS 环境）：**
- 直接使用 Office JS (`Excel.run(async (context) => {...})`) —— **不要**使用 Python/openpyxl。
- 通过 `range.formulas = [["=B5*B6"]]` 写入公式 —— Office JS 公式会在动态工作簿中原生重新计算。
- 遵循同样的“公式优于硬编码”原则：设置 `range.formulas`，严禁在需要计算的单元格设置 `range.values`。
- 使用 `range.format.font.color` / `range.format.fill.color` 来应用蓝/黑/紫/绿配置方案。
- 无需单独的重算步骤 —— Excel 会原生处理计算。
- **合并单元格陷阱**：不要在调用 `.merge()` 后再对已合并区域设置 `.values`（会抛出 `InvalidArgument` 错误，因为区域仍记录原始维度）。正确做法是：仅向左上角单元格写入值 (`ws.getRange("A7").values = [["SOURCES & USES"]]`)，然后再对整个区域进行合并和格式化 (`ws.getRange("A7:F7").merge(); ws.getRange("A7:F7").format.fill.color = "#1F4E79";`)。

**如果生成独立的 .xlsx 文件（非实时 Excel 会话）：**
- 使用下文所述的 Python/openpyxl。
- 写入公式字符串 (`ws["D20"] = "=B5*B6"`)，并在交付前运行 `recalc.py`。

本技能后续部分以 openpyxl 为例编写，但同样原则适用于 Office JS —— 只需转换 API 调用方式即可。

### 核心原则
* **所有计算必须使用 Excel 公式** —— 严禁在 Python 中预计算结果并硬编码到单元格中。使用 openpyxl 时，应写入 `cell.value = "=B5*B6"`（公式字符串），而非 `cell.value = 1250`（预计算结果）。模型必须是动态的，随输入值的改变而自动更新。
* **沿用模板结构** —— 遵循 `examples/LBO_Model.xlsx` 或用户提供模板的组织方式。不要自创布局。
* **使用正确的单元格引用** —— 所有公式应引用相应的单元格。严禁手动输入本应来自其他单元格的数字。
* **保持正负号约定一致性** —— 遵循模板使用的符号习惯（有些使用负数代表流出，有些使用正数）。全程保持一致。
* **分模块作业，逐步与用户确认** —— 完整完成一个模块，向用户展示成果，运行该模块的验证检查，并在获得确认**后再**进入下一个模块。**不要**一次性建完整个模型后再展示 —— 后续模块依赖前期模块，如果回报分析做完才发现“资金来源与用途”表有错，会导致大量重复返工。

### 公式颜色约定
* **蓝色 (0000FF)**：硬编码输入项 —— 手动录入的数字，不引用其他单元格。
* **黑色 (000000)**：包含计算的公式 —— 任何使用运算符或函数的公式（如 `=B4*B5`, `=SUM()`, `=-MAX(0,B4)`）。
* **紫色 (800080)**：链接至**同工作表**单元格 —— 没有任何计算的直接引用（如 `=B9`, `=B45`）。
* **绿色 (008000)**：链接至**不同工作表**单元格 —— 跨表引用（如 `=Assumptions!B5`, `='Operating Model'!C10`）。

### 背景填充调色板 — 专业蓝灰系（除非用户/模板另有规定）
* **保持简约** —— 单元格填充仅限蓝色和灰色。严禁引入绿色、黄色、红色或多种强调色。专业的 LBO 模型体现的是克制感。
* **默认填充调色板：**
  * **模块页眉**（如 Sources & Uses, Operating Model 等）：深蓝色 `#1F4E79` 搭配白色加粗字体。
  * **列标题**（如 Year 1, Year 2 等）：浅蓝色 `#D9E1F2` 搭配黑色加粗字体。
  * **输入单元格**：浅灰色 `#F2F2F2`（或纯白色） —— 蓝色**字体**是信号，填充是次要的。
  * **公式/计算单元格**：白色，无填充。
  * **关键产出指标**（如 IRR, MOIC, 退出股权价值）：中蓝色 `#BDD7EE` 搭配黑色加粗字体。
* **这就是全部配色。** 3 种蓝 + 1 种灰 + 白色。如果模板有自定义配色，请遵循模板。
* 注意：上述蓝/黑/紫/绿**字体**颜色是为了区分输入、公式和链接。这与此处定义的**填充**调色板是独立的，两者相辅相成。

### 数字格式标准
* **货币 (Currency)**：`$#,##0;($#,##0);"-"` 或基于模板要求的 `$#,##0.0`。
* **百分比 (Percentages)**：`0.0%`（保留一位小数）。
* **倍数 (Multiples)**：`0.0"x"`（保留一位小数）。
* **MOIC/精确比率**：`0.00"x"`（保留两位小数以体现精确度）。
* **所有数值单元格**：右对齐。

## 模板分析阶段 - 必须首行执行

在填充任何公式前，全面审查模板：

1. **映射结构** —— 确定各模块位置及相互关系。标记哪些模块的数据会流向其他模块。

2. **理清时间轴** — 哪些列代表哪些期间？是否存在“交易日（Closing）”或“备考（Pro Forma）”列？预测期从哪里开始？

3. **区分输入框与公式框** —— 模板通常使用颜色代码、边框或阴影来区分。请尊重这些约定。

4. **仔细阅读现有标签** —— 行标签会直接告诉您预期的计算逻辑。不要凭空猜测，仔细阅读模板要求。

5. **检查现有公式** —— 某些模板可能已预填了部分内容。除非特别要求，否则不要覆盖已在正常工作的公式。

6. **记录模板特有约定** —— 正负号约定、小计结构、模块组织方式、不同组件是否分拆在独立标签页中等。

## 公式填充 - 通用方法

对于每个需要公式的单元格，遵循以下优先级：

### 第 1 步：核对模板
* 该单元格是否已有公式？若是，验证其正确性并跳过。
* 是否有注释说明预期的计算逻辑？
* 行/列标签是否已明确了计算方法？
* 相邻单元格是否呈现出应遵循的规律？

### 第 2 步：核对用户指令
* 用户是否指定了特定的计算方法？
* 既定的假设是否会影响此公式？
* 是否提到了任何特殊需求？

### 第 3 步：应用行业标准实操
* 如果模板和用户均未指定，请采用标准的 LBO 模型惯例。
* 记录下您所做的任何假设。
* 如果确实不确定，请询问用户。

## 常见难点区域

以下计算模式在 LBO 模型中经常出错。遇到时请格外留意：

### 模块平账 (Balancing)
* 当两个模块必须相等时（如 资金来源 = 资金用途），通常有一个项目是“差额项（Plug）”。
* 确定哪个项目是差额项，并将其计算为其他项目的差值。

### 税务计算
* 税务公式应仅引用相关的收入行和税率。
* **不应**引用无关模块（如债务计划表）。
* 考虑亏损是否产生税盾效用，还是直接忽略。

### 利息与循环引用
* 如果利息计算引用的余额受现金流影响，可能会导致循环引用。
* 使用**期初余额**（而非平均或期末余额）来打破循环引用。
* 逻辑链条：利息 → 现金流 → 偿还 → 期末余额（如果利息引用期末余额，则形成闭环）。

### 债务偿还 / 现金清扫 (Cash Sweeps)
* 当存在多个债务层级时，通常有偿还优先级。
* 现金清扫应遵循偿付顺序瀑布流。
* 余额不能为负 —— 适当使用 `MAX` 或 `MIN` 函数。

### 回报计算 (IRR/MOIC)
* 现金流的正负号必须正确：投资额 = 负数，回笼资金 = 正数。
* 如果使用 `XIRR`，需要对应的日期列。
* 如果使用 `IRR`，现金流应处于连续的期间。
* MOIC = 总回笼资金 / 总投资额。

### 敏感性分析表 (Sensitivity Tables)
* **使用奇数维度**（5×5 或 7×7） —— 严禁使用 4×4 或 6×6。奇数维度能确保有一个精确的中心单元格。
* **中心单元格 = 基准情景**。围绕模型的实际假设对称构建行轴和列轴的数值（例如：若基准入场倍数为 10.0x，则坐标轴设为 `[8.0x, 9.0x, 10.0x, 11.0x, 12.0x]`）。中心单元格的 IRR/MOIC **必须**等于模型的实际 IRR/MOIC 产出 —— 这是证明表格勾稽正确的证据。
* **高亮中心单元格** —— 使用中蓝色填充 (`#BDD7EE`) + 加粗字体，使其在视觉上锚定基准情景。
* Excel 的“模拟分析-数据表”功能可能无法在 openpyxl 下运行 —— 请改写显式引用行列标题的公式。
* 每个单元格应显示**不同**的数值 —— 如果全部相同，说明公式未能正确随输入变量变动。
* 使用混合引用（例如：行输入用 `$A5`，列输入用 `B$4`）。

## 需避免的常见错误

| 错误 | 后果 | 如何修复 |
|-------|-----------------|------------|
| 硬编码计算值 | 输入改变时模型不会更新 | 始终使用引用源单元格的公式 |
| 复制公式后引用错误 | 公式指向了错误的单元格 | 验证所有链接，使用恰当的 `$` 绝对引用 |
| 循环引用错误 | 模型无法完成计算 | 针对利息类计算使用期初余额，打破环路 |
| 模块不平账 | 本应匹配的总计出现偏差 | 确保有一个项目设为差额项（公式计算差值） |
| 出现不可能的负余额 | 超额支用可用资金 | 适当使用 `MAX(0, ...)` 或 `MIN` 函数 |
| IRR/回报率错误 | 符号错误或区间不完整 | 检查现金流正负号，确保公式覆盖所有期间 |
| 敏感性分析表数值全相同 | 公式未随输入变量变动 | 检查单元格引用 —— 需使用混合引用 (`$A5`, `B$4`) |
| 滚存表数据不连贯 | 期初不等于上期期末 | 验证期间之间的链接逻辑 |
| 正负号约定不统一 | 应加项变减项或反之 | 全程严格遵循模板的正负号约定 |

> **💡 Appendix: 领域知识小贴士**
> 
> *   **LBO（杠杆收购）**：想象一下你想买一套 100 万的房子，但手里只有 20 万。于是你找银行借了 80 万，买下房子后，用收来的租金去还房贷。几年后把房子 150 万卖掉，还清剩下的贷款，剩下的钱都是你的。LBO 就是公司版的“贷款买房”。
> *   **资金来源与用途 (Sources & Uses)**：这就是一张“收支明细单”。“用途”告诉你买公司、付中介费、收回旧欠款一共要花多少钱；“来源”告诉你这些钱里，多少是借的债，多少是老板自己掏的腰包（股权）。这两边必须一分钱都不差地“配平”。
> *   **IRR（内部收益率）与 MOIC（投资倍数）**：这是评判投资赚不赚钱的两把尺子。MOIC 简单粗暴，就是“回头钱除以本金”，比如投 1 块赚回来 3 块，就是 3 倍；IRR 则考虑了时间，1 年赚 3 倍和 10 年赚 3 倍，含金量完全不同，IRR 越高代表赚钱的速度越快。
> *   **敏感性分析表**：金融界的“压力测试”。如果我们预估公司每年增长 10%，但万一只有 5% 呢？或者退出时价格卖得低一点呢？这张表会告诉你，在不同的“入场价格”和“增长速度”组合下，我们的收益会如何变动，帮投资人看清风险边界。
