# DCF 模型构建器

## 概览

本技能遵循投资银行标准，为权益估值创建机构级质量的 DCF 模型。每次分析都会生成一个详细的 Excel 模型（并在 DCF 工作表的底部附带敏感性分析）。

## 工具

- 默认使用用户提供的所有信息以及可用于数据溯源的 MCP 服务器。

## 关键约束——请先阅读

这些约束适用于整个 DCF 模型构建过程。在开始前请仔细审阅：

**环境：Office JS 与 Python/openpyxl：**
- **如果在 Excel 内部运行（Office 加载项 / Office JS 环境）：** 直接使用 Office JS —— 不要使用 Python/openpyxl。通过 `range.formulas = [["=D19*(1+$B$8)"]]` 写入公式。无需单独的重算步骤；Excel 会原生计算。使用 `range.format.*` 进行样式设置。同样的“公式优于硬编码”规则适用：为派生单元格设置 `.formulas`，绝不要设置 `.values`。
- **如果生成独立的 .xlsx 文件（无实时 Excel 会话）：** 按下文所述使用 Python/openpyxl，然后在交付前运行 `recalc.py`。
- 本技能的其余部分使用 openpyxl 示例 —— 在该环境下请转化为 Office JS API 调用，但所有原则（公式字符串、单元格批注、章节检查点、敏感性分析表循环）均完全相同。

**⚠️ Office JS 合并单元格陷阱：** 在构建带有合并单元格的章节标题时，不要先调用 `.merge()` 然后在合并后的区域设置 `.values` —— Office JS 仍会报告区域的原始尺寸，并抛出 `InvalidArgument: The number of rows or columns in the input array doesn't match the size or dimensions of the range`。相反，应先将值写入左上角单元格，然后再合并并格式化整个区域：

```js
// 错误 —— 会抛出 InvalidArgument：
const hdr = ws.getRange("A7:H7");
hdr.merge();
hdr.values = [["市场数据与关键输入"]];  // 1×1 数组对比 1×8 区域 → 失败

// 正确 —— 先在单个单元格赋值，然后合并并格式化区域：
ws.getRange("A7").values = [["市场数据与关键输入"]];
const hdr = ws.getRange("A7:H7");
hdr.merge();
hdr.format.fill.color = "#1F4E79";
hdr.format.font.bold = true;
hdr.format.font.color = "#FFFFFF";
```

这适用于 DCF 中的每一个合并章节标题（市场数据、场景块、现金流预测、终值、估值摘要、敏感性分析表）。

**公式优于硬编码（不可协商）：**
- 每一个预测、利润率、折现因子、现值（PV）和敏感性分析单元格必须是一个实时 Excel 公式 —— 绝不能是在 Python 中计算并以数字形式写入的数值。
- 使用 openpyxl 时：`ws["D20"] = "=D19*(1+$B$8)"` 是正确的；`ws["D20"] = calculated_revenue` 是错误的。
- 仅允许以下内容使用硬编码数字：(1) 原始历史输入，(2) 假设驱动因素（增长率、WACC 输入、永续增长率 g），(3) 当前市场数据（股价、债务余额）。
- 如果你发现自己正在 Python 中进行计算并写入结果 —— 请停止。当用户更改假设时，模型必须能够随之联动。

**与用户逐步核实（不要一站式构建）：**
- 数据检索后 → 向用户展示原始输入块（营收、利润率、股数、净负债），并在预测前确认。
- 营收预测后 → 展示预测的顶线数据和增长率，并在构建利润率结构前确认。
- FCF 构建后 → 展示完整的 FCF 附表，并在计算 WACC 前确认逻辑。
- WACC 之后 → 展示计算过程和输入参数，并在折现前确认。
- 终值 + 现值（PV）之后 → 展示权益桥（企业价值 EV → 权益价值 → 每股价值），并在构建敏感性分析表前确认。
- 在每个阶段捕捉错误 —— 如果在构建敏感性分析表后才发现利润率假设错误，意味着必须重建下游的所有内容。

**敏感性分析表：**
- **使用奇数的行数和列数**（标准：5×5，有时 7×7） —— 这确保了有一个真正的中心单元格。
- **中心单元格 = 基准情形。** 构建坐标轴数值，使中间行标题和中间列标题正好等于模型的实际假设（例如，如果基准 WACC = 9.0%，则中间行是 9.0%；如果永续增长率 g = 3.0%，则中间列是 3.0%）。因此，中心单元格的输出必须等于模型实际隐含的每股价格 —— 这是检查表格构建是否正确的质量控制手段。
- **高亮显示中心单元格**，使用中蓝色填充（`#BDD7EE`） + 加粗字体，以便一眼看出哪个单元格是基准情形。
- 使用完整的 DCF 重算公式填充所有单元格（通常为 3 个表 × 25 个单元格 = 75 个）。
- 使用 openpyxl 循环（或 Office JS 循环）以编程方式写入公式。
- 不允许使用占位符文本，不允许线性近似，不需要任何手动步骤。
- 每个单元格必须针对该假设组合重新计算完整的 DCF。

**单元格批注：**
- 在创建每个硬编码值的**同时**添加单元格批注。
- 格式：“来源：[系统/文件]，[日期]，[引用内容]，[如有，URL]”。
- 在移动到下一章节前，每个蓝色输入单元格都必须有批注。
- 不要推迟到最后或写上“TODO: 添加来源”。

**模型布局规划：**
- 在编写任何公式之前，先定义所有章节的行位置。
- 先编写所有标题和标签。
- 其次编写所有章节分隔符和空行。
- 然后使用锁定后的行位置编写公式。
- 公式创建后立即进行测试。

**公式重算：**
- 交付前运行 `python recalc.py model.xlsx 30`。
- 修复所有错误，直到显示“Success（成功）”。
- 必须确保零公式错误（#REF!, #DIV/0!, #VALUE! 等）。

**场景块：**
- 为悲观（Bear）、基准（Base）、乐观（Bull）情形分别创建独立的块。
- 在每个块内横向展示预测年度的假设。
- 使用 IF 公式：`=IF($B$6=1,[悲观单元格],IF($B$6=2,[基准单元格],[乐观单元格]))`。
- 核实公式是否引用了正确的场景块单元格。

## DCF 流程工作流

### 第一步：数据检索与验证

从 MCP 服务器、用户提供的数据以及网络抓取数据。

**数据源优先级：**
1. **MCP 服务器**（如已配置） - 来自 Daloopa 等提供商的结构化财务数据。
2. **用户提供的数据** - 来自其研究的历史财务数据。
3. **网络搜索/获取** - 适时获取当前股价、贝塔系数、债务和现金。

**验证清单：**
- 核实净负债与净现金（对估值至关重要）。
- 确认稀释后的发行在外股数（检查近期回购/发行情况）。
- 验证历史利润率是否与商业模式一致。
- 将营收增长率与行业基准进行交叉核对。
- 验证税率是否合理（通常为 21-28%）。

### 第二步：历史分析（3-5 年）

分析并记录：
- **营收增长趋势**：计算复合年均增长率（CAGR），识别驱动因素。
- **利润率演变**：跟踪毛利率、EBIT 利润率、FCF 利润率。
- **资本密集度**：折旧摊销（D&A）和资本支出（CapEx）占营收的百分比。
- **营运资本效率**：净营运资本（NWC）变动额占营收增长的百分比。
- **回报指标**：ROIC、ROE 趋势。

创建展示以下内容的摘要表：
```
历史指标 (LTM):
营收: $X 百万
营收增长: X% CAGR
毛利率: X%
EBIT 利润率: X%
D&A 占营收比例: X%
CapEx 占营收比例: X%
FCF 利润率: X%
```

### 第三步：构建营收预测

**方法论：**
1. 从最新的实际营收（LTM 或最近一个财年）开始。
2. 对每个预测年度应用增长率。
3. 同时展示金额数值和计算出的增长百分比。

**增长率框架：**
- 第 1-2 年：较高的增长，反映近期的能见度。
- 第 3-4 年：逐渐向行业平均水平放缓。
- 第 5 年及以后：接近永续增长率。

**公式结构：**
- 营收(第 N 年) = 营收(第 N-1 年) × (1 + 增长率)
- 增长率 %(第 N 年) = 营收(第 N 年) / 营收(第 N-1 年) - 1

**三场景方法：**
```
悲观情形: 保守增长 (例如: 8-12%)
基准情形: 最可能的情形 (例如: 12-16%)
乐观情形: 乐观增长 (例如: 16-20%)
```

### 第四步：经营费用建模

**固定/变动成本分析：**

经营费用应体现现实的经营杠杆：
- **销售与市场 (S&M)**：通常占营收的 15-40%，取决于商业模式。
- **研发 (R&D)**：对于技术型公司，通常占 10-30%。
- **一般及行政 (G&A)**：通常占营收的 8-15%，随公司规模扩大呈现杠杆效应。

**核心原则：**
- 所有百分比均基于**营收**，而非毛利润。
- 模拟经营杠杆：百分比应随营收规模扩大而下降。
- 保持 S&M、R&D、G&A 的独立分项。
- 计算 EBIT = 毛利润 - 总经营费用（OpEx）。

**利润率扩张框架：**
```
当前状态 → 目标状态 (第 5 年)
毛利率: X% → Y% (根据规模、效率给出理由)
EBIT 利润率: X% → Y% (营收增长 + 费用杠杆的结果)
```

### 第五步：自由现金流计算

**按正确顺序计算 FCF：**

```
EBIT
(-) 税收 (EBIT × 税率)
= 税后净营业利润 (NOPAT)
(+) 折旧与摊销 (D&A，非现金费用，占营收 %)
(-) 资本支出 (CapEx，占营收 %，通常为 4-8%)
(-) 净营运资本变动 (Δ NWC)
= 无杠杆自由现金流 (Unlevered FCF)
```

**营运资本建模：**
- 按营收变动额（营收 Delta）的百分比计算。
- 典型范围：营收变动额的 -2% 到 +2%。
- 负数 = 现金来源（营运资本释放）。
- 正数 = 现金占用（营运资本增加）。

**维护性 vs 增长性资本支出：**
- 维持性资本支出：维持当前运营（约占营收 2-3%）。
- 增长性资本支出：支持扩张（额外占营收 2-5%）。
- 总资本支出应与公司的增长战略保持一致。

### 第六步：资本成本 (WACC) 研究

**用于计算权益成本的 CAPM 方法论：**

```
权益成本 = 无风险利率 + 贝塔系数 × 股票风险溢价

其中：
- 无风险利率 = 当前 10 年期国债收益率
- 贝塔系数 = 5 年期股票相对于市场指数的月度贝塔值
- 股票风险溢价 = 5.0-6.0% (市场标准)
```

**债务成本计算：**

```
税后债务成本 = 税前债务成本 × (1 - 税率)

通过以下方式确定税前债务成本：
- 信用评级 (如有)
- 公司债券的当前收益率
- 财务报表中的利息支出 / 总债务
```

**资本结构权重：**

```
权益市场价值 = 当前股价 × 发行在外股数
净负债 = 总债务 - 现金及现金等价物
企业价值 = 市值 + 净负债

权益权重 = 市值 / 企业价值
债务权重 = 净负债 / 企业价值

WACC = (权益成本 × 权益权重) + (税后债务成本 × 债务权重)
```

**特殊情况：**
- **净现金头寸**：如果现金 > 债务，则净负债为负。
  - 债务权重可能为负。
  - WACC 计算据此调整。
- **无债务**：WACC = 权益成本。

**典型的 WACC 范围：**
- 大盘蓝筹、稳定：7-9%
- 成长型公司：9-12%
- 高增长/高风险：12-15%

### 第七步：折现率应用（5-10 年预测期）

**中期折现惯例：**
- 假设现金流发生在年中。
- 折现期：0.5, 1.5, 2.5, 3.5, 4.5 等。
- 折现因子 = 1 / (1 + WACC)^折现期

**现值（PV）计算：**
```
对于每个预测年度：
FCF 的现值 = 无杠杆 FCF × 折现因子

示例 (第 1 年):
FCF = $1,000
WACC = 10%
折现期 = 0.5
折现因子 = 1 / (1.10)^0.5 = 0.9535
现值 = $1,000 × 0.9535 = $954
```

**预测期选择：**
- **5 年**：大多数分析的标准。
- **7-10 年**：具有较长增长跑道的高增长公司。
- **3 年**：成熟、稳定的业务。

### 第八步：终值计算

**永续增长法（首选）：**

```
终值年份 FCF = 最终预测年 FCF × (1 + 永续增长率)
终值 = 终值年份 FCF / (WACC - 永续增长率)

关键约束：永续增长率 < WACC (否则价值无限)
```

**永续增长率选择：**
- 保守：2.0-2.5% (GDP 增长率)
- 中等：2.5-3.5%
- 进取：3.5-5.0% (仅限于行业领导者)

**不得超过**：无风险利率或长期 GDP 增长率。

**退出乘数法（备选）：**
```
终值 = 最终预测年 EBITDA × 退出乘数

其中退出乘数来源于：
- 行业可比交易乘数
- 先例交易乘数
- 典型范围：8-15x EBITDA
```

**终值的现值：**
```
终值的现值 = 终值 / (1 + WACC)^最终时期

其中最终时期会考虑时间：
采用中期惯例的 5 年模型：时期 = 4.5
```

**终值合理性检查：**
- 应占企业价值（EV）的 50-70%。
- 如果 >75%，模型可能过度依赖终值假设。
- 如果 <40%，检查终值假设是否过于保守。

### 第九步：企业价值到权益价值的路径图

**估值摘要结构：**

```
(+) 预测期 FCF 的现值之和 = $X 百万
(+) 终值的现值 = $Y 百万
= 企业价值 (EV) = $Z 百万

(-) 净负债 [如果是负数则 + 净现金] = $A 百万
= 权益价值 = $B 百万

÷ 稀释后的发行在外股数 = C 百万股
= 隐含每股价格 = $XX.XX

当前股价 = $YY.YY
隐含收益率 = (隐含价格 / 当前价格) - 1 = XX%
```

**关键调整：**
- **净负债 = 总债务 - 现金及现金等价物**
  - 如果为正：从 EV 中减去（降低权益价值）。
  - 如果为负（净现金）：加回 EV（增加权益价值）。
- **使用稀释后股数**：包括期权、受限股（RSU）、可转换证券。
- **其他调整**（如适用）：
  - 少数股东权益
  - 养老金负债
  - 融资租赁义务

**估值输出格式：**
```csv
估值组成部分,金额 ($M)
显性预测期 FCF 现值,X.X
终值现值,Y.Y
企业价值 (EV),Z.Z
(-) 净负债,A.A
权益价值,B.B
,,
发行在外股数 (M),C.C
隐含每股价格,$XX.XX
当前每股价格,$YY.YY
隐含上涨/(下跌)空间,+XX%
```

### 第十步：敏感性分析

在 DCF 工作表底部构建**三个敏感性分析表**，展示估值随不同假设的变化情况：

1. **WACC 对比 永续增长率** - 展示企业价值对折现率和永续增长率的敏感度。
2. **营收增长 对比 EBIT 利润率** - 展示顶线增长和经营杠杆的影响。
3. **贝塔系数 对比 无风险利率** - 展示对权益成本组成部分的敏感度。

**实现方式**：这些是简单的 2D 网格（**不是** Excel 的“模拟分析(Data Table)”功能），每个单元格都有公式。每个单元格必须包含针对该特定假设组合的完整 DCF 重新计算。详见“关键约束”章节中关于使用 openpyxl 以编程方式填充全部 75 个单元格的详细要求。

### 场景块选择模式 —— 遵循此方法

**假设按每个场景的独立块进行组织：**

**关键结构 —— 每一节标题占三行：**

```csv
悲观情形假设 (章节标题, 跨列合并单元格)
假设项目,FY1,FY2,FY3,FY4,FY5
营收增长 (%),12%,10%,9%,8%,7%
EBIT 利润率 (%),45%,44%,43%,42%,41%

基准情形假设 (章节标题, 跨列合并单元格)
假设项目,FY1,FY2,FY3,FY4,FY5
营收增长 (%),16%,14%,12%,10%,9%
EBIT 利润率 (%),48%,49%,50%,51%,52%

乐观情形假设 (章节标题, 跨列合并单元格)
假设项目,FY1,FY2,FY3,FY4,FY5
营收增长 (%),20%,18%,15%,13%,11%
EBIT 利润率 (%),50%,51%,52%,53%,54%
```

**每个场景块必须有一个列标题行**，紧接在章节标题下方显示预测年份（FY2025E, FY2026E 等）。如果没有这个，用户将无法分辨哪个假设值对应哪个年份。

**如何引用假设 —— 创建一个整合列：**
1. 案例选择器单元格（例如 B6）包含 1=悲观, 2=基准, 或 3=乐观。
2. 创建一个整合列，使用 INDEX 或 OFFSET 公式从正确的场景块中提取数据。
3. 预测公式引用该整合列（保持清晰的单元格引用）。
4. 每个场景块都包含预测年度内的一整套 DCF 假设。

**推荐的整合列模式（使用 INDEX）：**
`=INDEX(B10:D10, 1, $B$6)`

**不要这样 —— 散落在各处的 IF 语句：**
`=IF($B$6=1,[悲观块单元格],IF($B$6=2,[基准块单元格],[乐观块单元格]))`

整合列方法集中了逻辑，使模型更易于审计。

### 正确的营收预测模式

**使用 INDEX 公式创建一个整合列，然后在预测中引用它：**

**第一步 —— FY1 增长的整合列：**
`=INDEX([悲观 FY1 增长]:[乐观 FY1 增长], 1, $B$6)`

**第二步 —— 营收预测引用整合列：**
`第 1 年营收: =D29*(1+$E$10)`

其中：
- D29 = 前一年营收
- $E$10 = FY1 增长的整合列单元格（包含 INDEX 公式）
- $B$6 = 案例选择器 (1=悲观, 2=基准, 3=乐观)

**这种方法比在每个预测公式中嵌入 IF 语句更整洁**，并且更容易审计正在使用哪些场景假设。

### 正确的 FCF 公式模式

**使用带有 INDEX 公式的整合列，然后在 FCF 计算中引用它们：**

**整合列方法：**
```csv
项目,公式,参考说明
折旧摊销 (D&A),=E29*$E$21,$E$21 = D&A % 整合列
资本支出 (CapEx),=E29*$E$22,$E$22 = CapEx % 整合列
Δ NWC,=(E29-D29)*$E$23,$E$23 = NWC % 整合列
无杠杆 FCF,=E57+E58-E60-E62,E57=NOPAT E58=D&A E60=CapEx E62=Δ NWC
```

**每个整合列单元格包含一个 INDEX 公式**，该公式根据案例选择器从适当的场景块中提取数据。这使预测公式保持简洁且可审计。

在编写公式之前，确认场景块的行位置并设置整合列。

### 正确的单元格批注格式

**每一个硬编码值都需要这种格式：**

“来源：[系统/文件]，[日期]，[引用内容]，[如有，URL]”

**示例：**
```csv
项目,来源批注
股价,来源: 市场数据脚本 2025-10-12 收盘价
发行在外股数,来源: 10-K FY2024 第 45 页 注释 12
历史营收,来源: 10-K FY2024 第 32 页 合并报表
贝塔系数,来源: 市场数据脚本 2025-10-12 5 年期月度贝塔
共识预期,来源: 管理层指引 2024年三季度业绩电话会议
```

### 正确的假设表结构

**关键：每个场景块需要三个结构元素：**

1. **章节标题行**（合并单元格）：例如“悲观情形假设”。
2. **显示年份的列标题行** —— 这是必需的，请勿跳过。
3. **带有假设值的数值行**。

**结构：**
```csv
悲观情形假设 (章节标题 - 跨 A:G 列合并)
假设项目,FY1,FY2,FY3,FY4,FY5
营收增长 (%),X%,X%,X%,X%,X%
EBIT 利润率 (%),X%,X%,X%,X%,X%
永续增长率,X%,,,,
WACC,X%,,,,

基准情形假设 (章节标题 - 跨 A:G 列合并)
假设项目,FY1,FY2,FY3,FY4,FY5
营收增长 (%),X%,X%,X%,X%,X%
EBIT 利润率 (%),X%,X%,X%,X%,X%
永续增长率,X%,,,,
WACC,X%,,,,

乐观情形假设 (章节标题 - 跨 A:G 列合并)
假设项目,FY1,FY2,FY3,FY4,FY5
营收增长 (%),X%,X%,X%,X%,X%
EBIT 利润率 (%),X%,X%,X%,X%,X%
永续增长率,X%,,,,
WACC,X%,,,,
```

**如果没有显示预测年份（FY2025E, FY2026E 等）的列标题行，用户将无法分辨哪个假设值对应哪个年份。该行是强制性的。**

**然后创建一个整合列**（通常是右侧的下一列），使用 INDEX 公式根据案例选择器从选定的场景块中提取数据。你的预测公式将引用这个整合列。

### 正确的行规划流程

**1. 首先编写所有标题和标签：**
```csv
行,内容
1,[公司名称] DCF 模型
2,股票代码 | 日期 | 财年截止日
4,案例选择器
7,关键假设
26,假设项目标题
27-31,增长假设
...,...
```

**2. 编写所有章节分隔符和空行**

**3. 然后使用锁定后的行位置编写公式**

**4. 公式创建后立即进行测试**

**把这想象成建筑施工：**
- 正确做法：先浇筑地基，再建造墙壁（结构稳固）。
- 错误做法：先建造墙壁，再浇筑地基（墙壁倒塌）。

**Excel 版本：**
- 正确做法：先添加标题，再编写公式（公式稳固）。
- 错误做法：先编写公式，再添加标题（公式断开）。

### 正确的敏感性分析表实现

**重要提示**：这些**不是** Excel 的“模拟分析表(Data Table)”功能。这些是简单的网格，你在其中使用 openpyxl 编写常规公式。是的，这意味着总共约 75 个公式（3 个表 × 每个表 25 个单元格），但这很直接且是必需的。

**以编程方式填充公式：**

每个敏感性分析表必须完整填充公式，这些公式会针对每种假设组合重新计算隐含的股价。**不要使用 Excel 的模拟分析表功能**（它需要手动干预，且无法通过 openpyxl 自动化）。

**实现方法 —— 具体示例：**

**表格结构 —— 5×5 网格（奇数维度，基准情形居中）：**

如果模型的基准 WACC = 9.0% 且基准永续增长率 = 3.0%，则围绕这些值对称构建坐标轴：

```csv
WACC 对比 永续增长率,  2.0%,  2.5%,  3.0%,  3.5%,  4.0%
              8.0%,       [fml], [fml], [fml], [fml], [fml]
              8.5%,       [fml], [fml], [fml], [fml], [fml]
              9.0%,       [fml], [fml], [★  ], [fml], [fml]   ← 中间行 = 基准 WACC
              9.5%,       [fml], [fml], [fml], [fml], [fml]
             10.0%,       [fml], [fml], [fml], [fml], [fml]
                                   ↑
                          中间列 = 基准永续增长率 g
```

**★ = 中心单元格。** 它的公式输出必须等于模型实际隐含的每股价格（源自估值摘要）。对该单元格应用中蓝色填充（`#BDD7EE`）和加粗字体，以便基准情形在视觉上得到锚定。

**坐标轴数值规则**：`axis_values = [base - 2*step, base - step, base, base + step, base + 2*step]` —— 围绕基准值对称，奇数确保了中心。

**公式模式 - 单元格 B88 (WACC=8.0%, 永续增长率=2.0%):**

B88 中的公式应使用以下内容重新计算隐含价格：
- 来自行标题的 WACC：`$A88` (8.0%)
- 来自列标题的永续增长率：`B$87` (2.0%)

**推荐方法：** 引用主要的 DCF 计算内容，但替换掉这些值。

**示例公式结构：**
`=([使用 $A88 作为折现率的 FCF 现值之和] + [使用 B$87 作为增长率且 $A88 作为 WACC 的终值] - [净负债]) / [股数]`

**关键 —— 为 5x5 网格中的每个单元格编写公式（每个表 25 个单元格，总计 75 个）。** 使用 openpyxl 在循环中以编程方式编写这些公式。不要跳过这一步或留下占位符文本。

**Python 实现模式：**
```python
# 填充敏感性分析表的伪代码
for row_idx, wacc_value in enumerate(wacc_range):
    for col_idx, term_growth_value in enumerate(term_growth_range):
        # 构建使用 wacc_value 和 term_growth_value 的公式
        formula = f"=<使用 {wacc_value} 和 {term_growth_value} 的 DCF 重算公式>"
        ws.cell(row=start_row+row_idx, column=start_col+col_idx).value = formula
```

**敏感性分析表在打开模型时必须立即可用，无需用户进行任何手动步骤。**

<common_mistakes>

本章节包含在构建 DCF 模型时应避免的所有错误模式。

### 错误：简化的敏感性分析表近似值或占位符文本

**不要使用线性近似：**

```
// 错误 - 线性近似
B97: =B88*(1+(0.096-0.116))    // 假设是线性关系

// 错误 - 除法快捷方式
B105: =B88/(1+(E48-0.07))      // 没有重新计算完整 DCF
```

**不要留下占位符文本：**
```
// 错误 - 占位符注释
"说明：请使用 Excel 模拟分析功能 (数据 → 模拟分析 → 模拟分析表) 来填充敏感性分析表。"

// 错误 - 空白单元格
[因为“这太复杂”而让单元格保持空白]
```

**不要混淆术语：**
- ❌ “敏感性分析表需要 Excel 的模拟分析表功能”（错误 —— 这是一个特定的 Excel 工具，我们无法使用）
- ✅ “敏感性分析表是每个单元格都有公式的简单网格”（正确 —— 这是我们要构建的内容）

**为什么这些捷径是错误的：**
- 线性近似公式实际上并没有重新计算 DCF —— 它们只是应用了简单的数学调整。
- 估值关系不是线性的，因此结果将不准确。
- 占位符文本需要用户手动干预。
- 交付的模型无法立即使用。
- 不专业或不具备对客演示水平。
- 空格 = 不完整的交付物。

**常见的推脱理由：**
“写 75 个以上的公式感觉很复杂，所以我给用户留个便条，让他们手动完成。”

**现实：** 当你在 Python 中使用 openpyxl 进行循环时，编写 75 个公式非常直接。每个公式都遵循相同的模式 —— 只需替换行/列值。这是交付物中必不可少的部分。

**相反：** 在每一个敏感性分析单元格中填充公式，以便针对该特定的假设组合重新计算完整的 DCF。

### 错误：缺少单元格批注

**不要这样做：**
- 创建了所有硬编码输入但没有批注。
- 想着“我待会儿再加”。
- 写上“TODO: 添加来源”。
- 让蓝色输入内容没有文档支持。

**为什么它是错误的：**
- 无法验证数据来源。
- 违反了 xlsx 技能要求。
- 无法通过审计。
- 浪费后期修复的时间。

**相反：** 在创建每个硬编码值的**同时**添加单元格批注。

### 错误：公式行引用偏移

**症状：**
FCF 章节引用了错误的假设行：
`D&A:  =E29*$E$34    // 应该是 $E$21，但引用了错误的行`
`CapEx: =E29*$E$41   // 应该是 $E$22，但行号发生了偏移`

**为什么会发生这种情况：**
1. 先编写了公式。
2. 然后插入了标题。
3. 所有的行引用都发生了偏移。
4. 现在公式指向了错误的单元格 → 导致 #REF! 错误。

**相反：** 先锁定行布局，然后再编写公式。

### 错误：跨场景为每个假设只设一行

**不要这样构造假设：**
```csv
假设项目,悲观,基准,乐观
营收增长 FY1,10%,13%,16%
营收增长 FY2,9%,12%,15%
```
这种纵向布局使得很难看清每个场景内跨年度的演变逻辑。

**为什么它是错误的：**
- 难以看清每个场景内假设随年份演变的逻辑。
- 难以在整个预测期内比较不同场景的假设。
- 审核场景逻辑时不够直观。

**相反：**
- 为每个场景（悲观、基准、乐观）创建独立的块。
- 在每个块内，横向展示预测年度的假设。
- 这使得每个场景的假设作为一套完整的逻辑更易于审阅。

### 错误：没有边框

**不要交付一个没有边框的模型：**
- 没有章节划分。
- 所有单元格混在一起。
- 难以阅读且不专业。

**为什么它是错误的：**
- 无法对客演示。
- 难以导航查阅。
- 看起来很业余。

**相反：** 在所有主要章节周围添加边框。

### 错误：错误的字体颜色或没有字体颜色区分

**不要这样做：**
- 所有文字均为黑色。
- 只使用填充颜色（不更改字体颜色）。
- 弄混了哪些单元格应该是蓝色，哪些应该是黑色。

**为什么它是错误的：**
- 无法区分输入值和公式。
- 审计变得不可能。
- 违反了 xlsx 技能要求。

**相反：** 对所有硬编码输入使用蓝色字体，对所有公式使用黑色字体，对跨表链接使用绿色字体。

### 错误：经营费用基于毛利润计算

**不要这样做：**
`S&M: =E33*0.15    // E33 = 毛利润 (错误)`

**为什么它是错误的：**
- 经营费用随营收规模变化，而非毛利润。
- 会产生不切实际的利润率演变逻辑。
- 并非公司实际运作的逻辑。

**相反：**
`S&M: =E29*0.15    // E29 = 营收 (正确)`

### 前 5 大错误总结

1. **公式行引用偏移** → 在编写公式之前定义所有行位置。
2. **缺少单元格批注** → 在创建单元格的同时添加批注，不要留到最后。
3. **简化的敏感性分析表** → 为所有单元格填充完整的 DCF 重算公式，而不是近似值。
4. **场景块引用错误** → 确保 IF 公式可以从正确的悲观/基准/乐观块中提取数据。
5. **没有边框** → 添加专业的章节边框，确保具备对客演示水平。

此外，还要注意以下错误：

### WACC 计算错误
- 在资本结构中混用账面价值和市场价值。
- 错误地混合使用权益贝塔与资产/去杠杆贝塔。
- 债务成本中税率应用错误。
- 错误的无风险利率（必须使用当前的 10 年期国债收益率）。
- 未能根据净负债 vs 净现金头寸进行调整。

### 增长假设缺陷
- 永续增长率 > WACC（会产生无限价值）。
- 预测增长率与历史表现不一致。
- 忽视行业增长约束。
- 营收增长与单体经济模型（Unit Economics）不符。
- 利润率扩张缺乏经营层面的合理依据。

### 终值错误
- 使用了错误的增长方法（永续法 vs 退出乘数法）。
- 终值占企业价值的百分比 >80%（暗示过度依赖）。
- 终值利润率与稳态假设不一致。
- 终值的折现期错误。

### 现金流预测错误
- 经营费用基于毛利润而非营收。
- D&A/CapEx 百分比与商业模式不匹配。
- 营运资本变动计算不当。
- 年度间税率不一致。
- NOPAT 计算错误。

**这些是最常见的错误。在开始任何 DCF 构建之前，请重新阅读本节。**

</common_mistakes>
