# 总账对账员 — 托管代理模板

## 概述

针对特定交易日和资产类别集合，发现总账与子账之间的差异，追踪根因，并生成供财务总监审批的异常报告。

与 [`gl-reconciler`](../../plugins/agent-plugins/gl-reconciler) Cowork插件同源 —— 本目录是用于 `POST /v1/agents` 的托管代理操作手册。

## 部署

```bash
export ANTHROPIC_API_KEY=sk-ant-...
export GL_MCP_URL=...           # 只读总账MCP
export SUBLEDGER_MCP_URL=...    # 只读子账MCP
../../scripts/deploy-managed-agent.sh gl-reconciler
```

## 引导事件

参见 [`steering-examples.json`](./steering-examples.json)。以交易日和资产类别列表启动会话；后续事件可以重新追踪单个差异。

## 安全与交接

该代理读取交易对手/托管行对账单 —— 这些由外部人员撰写的文档可能携带对抗性指令。模板结构确保此类文档中的载荷无法触及shell、写入工具或公司系统：

| 层级 | 接触不可信文档？ | 工具 | 连接器 |
|---|---|---|---|
| **`reader`** | **是** | `Read`, `Grep` 仅限 | 无 |
| **编排器** | 否 | `Read`, `Grep`, `Glob`, `Agent` | 只读总账 + 子账MCP |
| **`resolver`**（写入持有者） | 否 | `Read`, `Write`, `Edit` | 无 |

`reader` 仅返回长度受限、模式验证的JSON（由 `scripts/validate.py` 验证）。`critic` 在编排器将差异集交给 `resolver` 之前，独立地根据可信来源重新验证每个差异。`resolver` 将异常报告写入 `./out/`；它从不打开外部人员文件。

**交接：** 如需将已验证的差异输入月末结算流程，编排器在最终输出中发出指向 `month-end-closer` 的 `handoff_request`；`scripts/orchestrate.py`（或您的Temporal/Airflow工作节点）将其作为新引导事件路由。参见该脚本中的白名单 + 载荷验证模式。

**不保证：** 以上操作均不写入记录系统。总账调整需要代理外部的人工审批。

## 金融术语和知识解释

### 1. 总账（General Ledger, GL）

**定义**：总账是企业会计系统的核心，记录所有财务交易的汇总信息，是编制财务报表的基础。

**总账结构**：
- **科目表（Chart of Accounts）**：所有会计科目的完整列表
- **借方/贷方**：每笔交易的双重记录
- **余额**：每个科目的累计金额
- **会计期间**：按月、季度、年度划分

**关键科目**：
| 科目类别 | 示例 | 借方/贷方规则 |
|---------|------|-------------|
| 资产 | 现金、应收账款、存货 | 借增贷减 |
| 负债 | 应付账款、长期借款 | 贷增借减 |
| 权益 | 股本、留存收益 | 贷增借减 |
| 收入 | 销售收入、利息收入 | 贷增借减 |
| 费用 | 工资、租金、折旧 | 借增贷减 |

**总账 vs 子账**：
- **总账**：汇总层级，记录各科目的合计余额
- **子账（Subledger）**：明细层级，记录每笔具体交易
- **对账**：确保总账余额等于子账余额的汇总

---

### 2. 对账（Reconciliation）

**定义**：对账是比较两个或多个数据来源的财务数据，确保它们一致的过程。

**对账类型**：

| 对账类型 | 对比双方 | 频率 |
|---------|---------|------|
| 银行对账 | 总账现金 vs 银行对账单 | 每日/每周 |
| 总账-子账对账 | 总账余额 vs 子账汇总 | 每月 |
| 交易对手对账 | 内部记录 vs 交易对手确认 | 每日 |
| 托管行对账 | 内部持仓 vs 托管行报告 | 每日 |
| 应收/应付对账 | 总账余额 vs 明细账龄 | 每月 |

**对账流程**：
1. **提取数据**：从总账和子账系统提取对应期间数据
2. **匹配交易**：按交易编号、日期、金额等字段匹配
3. **识别差异**：找出无法匹配或金额不一致的项目
4. **分析根因**：调查差异产生的原因
5. **编制调整**：制作日记账分录（JE）进行调整
6. **审批执行**：经财务总监审批后入账

**常见差异原因**：
- 时间性差异：一方已记录，另一方尚未记录
- 遗漏：某一方漏记交易
- 错误：金额、科目或方向录入错误
- 重复：同一笔交易被记录两次
- 系统故障：数据传输或处理错误

---

### 3. 异常报告（Exception Report）

**定义**：异常报告是对账过程中发现的差异汇总文档，详细列明每项差异的性质、金额、根因和建议调整。

**报告结构**：
1. **汇总页**：差异总数、总金额、按类型分类
2. **明细表**：每项差异的详细信息
3. **根因分析**：差异产生的原因分类
4. **调整建议**：建议的日记账分录
5. **审批签字**：财务总监和会计经理签字

**示例格式**：

| 差异编号 | 科目 | 总账金额 | 子账金额 | 差异金额 | 根因 | 建议调整 |
|---------|------|---------|---------|---------|------|---------|
| BRK001 | 应收账款 | 1,250,000 | 1,235,000 | 15,000 | 遗漏发票 | 借：应收账款 15,000 |
| BRK002 | 现金 | 890,000 | 885,000 | 5,000 | 银行手续费 | 借：银行手续费 5,000 |

**审批流程**：
1. 对账员编制异常报告
2. 会计经理复核差异分析
3. 财务总监审批调整建议
4. 会计执行日记账分录
5. 重新对账确认差异已消除

---

### 4. 资产类别（Asset Classes）

**定义**：资产类别是具有相似特征和风险收益属性的金融工具分组，是对账和风险管理的基本单位。

**主要资产类别**：

| 资产类别 | 特征 | 典型工具 |
|---------|------|---------|
| 股票（Equities） | 所有权凭证，收益不确定 | 普通股、优先股 |
| 固定收益（Fixed Income） | 债权凭证，收益相对固定 | 国债、公司债、市政债 |
| 现金及等价物 | 流动性最高，风险最低 | 银行存款、国库券、货币基金 |
| 衍生品 | 价值依赖于基础资产 | 期货、期权、互换 |
| 另类投资 | 传统资产之外 | 房地产、私募股权、大宗商品 |

**对账中的应用**：
- 按资产类别分组对账，提高效率
- 不同资产类别有不同的对账频率和精度要求
- 复杂衍生品可能需要专门的对账工具

---

### 5. 交易对手（Counterparty）

**定义**：交易对手是金融交易的另一方，可以是机构、个人或其他实体。

**交易对手类型**：
- **金融机构**：银行、券商、保险公司
- **企业客户**：供应商、客户
- **托管行（Custodian）**：保管资产的第三方机构
- **清算所（Clearing House）**：衍生品交易的中央对手方

**交易对手风险**：
- **信用风险**：交易对手违约的可能性
- **结算风险**：交易一方已付款但另一方未履约
- **操作风险**：交易对手内部流程失误

**对账中的重要性**：
- 定期与交易对手确认交易细节
- 及时发现和解决争议
- 监控交易对手信用状况

---

### 6. 三级安全隔离详解

**读取层（reader）**：
- **职责**：读取外部对账单和交易确认书
- **风险**：外部文档可能包含恶意指令
- **防护**：
  - 只读权限，无法写入系统
  - 返回长度受限的JSON
  - 模式验证确保输出格式正确
  - 不执行文档中的任何指令

**处理层（Orchestrator + critic）**：
- **职责**：分析差异、验证数据、做出判断
- **权限**：可以调用Agent工具，但只能读取内部系统
- **critic角色**：独立验证每个差异，确保准确性
- **连接器**：只读访问总账和子账MCP

**写入层（resolver）**：
- **职责**：生成异常报告和日记账分录草稿
- **特点**：唯一拥有Write权限的节点
- **安全**：从不直接接触外部文档
- **输出**：写入 `./out/` 目录，不直接修改总账

### 7. 日记账分录（Journal Entry, JE）

**定义**：日记账分录是会计系统中记录每笔交易的最基本单位，遵循借贷平衡原则。

**基本结构**：
```
日期：2024-03-15
借：应收账款 - 客户A    10,000
贷：销售收入            10,000
摘要：确认销售收入 - 发票#12345
```

**分录类型**：

| 类型 | 用途 | 示例 |
|------|------|------|
| 常规分录 | 日常业务交易 | 销售、采购、费用 |
| 调整分录 | 期末调整 | 折旧、摊销、预提 |
| 对账分录 | 消除差异 | 银行手续费、遗漏交易 |
| 冲销分录 | 纠正错误 | 红字冲销、重新入账 |
| 结转分录 | 期间结转 | 收入费用结转至留存收益 |

**审批要求**：
- 常规分录：会计经理审批
- 调整分录：财务总监审批
- 大额分录（如>100万）：CFO审批
- 所有分录需保留完整审计轨迹

### 8. MCP连接器配置

**总账MCP（GL MCP）**：
- **用途**：读取总账系统数据
- **访问权限**：只读，防止代理修改财务记录
- **数据范围**：科目余额、交易明细、会计期间信息
- **典型URL**：`https://internal-erp.company.com/mcp/gl`

**子账MCP（Subledger MCP）**：
- **用途**：读取子账系统明细数据
- **访问权限**：只读
- **数据范围**：应收账款明细、应付账款明细、存货明细等
- **典型URL**：`https://internal-erp.company.com/mcp/subledger`

**安全考虑**：
- 只读访问防止代理意外修改数据
- 所有查询都有审计日志
- 基于角色的访问控制（RBAC）
- 网络隔离，仅限内部网络访问

---

### 9. 月末结算流程

**定义**：月末结算是在每个会计期间结束时，完成所有财务记录、调整和报告的过程。

**标准流程**：
1. **数据收集**：收集所有交易数据和支持文档
2. **对账**：执行所有必要的对账程序
3. **调整**：录入期末调整分录（应计、折旧等）
4. **审查**：管理层审查财务报表草稿
5. **关闭**：关闭会计期间，防止后续修改
6. **报告**：生成并分发财务报表

**与对账的关系**：
- 对账是月末结算的关键步骤
- 未解决的对账差异会阻碍期间关闭
- 对账结果直接影响财务报表准确性

**交接机制**：
- GL Reconciler完成对账后，将已验证的差异交接给Month-End Closer
- Month-End Closer将差异纳入期末调整
- 最终生成完整的结算材料包
