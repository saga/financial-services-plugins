# 追溯差异根本原因

给定单个差异行（键、总账值、子账值、分类、可能原因），追溯至来源并生成根本原因陈述。

## 追溯路径

1. **提取总账侧** — 通过internal-gl MCP，获取生成此总账行的日记账分录或过账：分录ID、过账日期、源系统、批次ID、制备人。
2. **提取子账侧** — 通过subledger MCP，获取匹配的交易：交易ID、交易/结算日期、交易对手、源数据推送、使用的汇率。
3. **对比属性** — 对齐过账日期、汇率/日期、账户映射、数量符号、金额符号。差异的属性通常就是原因。

## 原因 → 陈述

将根本原因写成一句话，格式为 **"⟨侧⟩ ⟨做了什么⟩ 因为 ⟨原因⟩"**，例如：

- "总账在结算日期（T+2）过账，而子账在交易日期过账 — 时序差异，将在2026-05-07清除。"
- "子账使用WM/R下午4点汇率；总账使用彭博收盘价 — 基础货币金额12个基点的外汇差异。"
- "证券ABC123在映射表中映射到总账账户11420，但子账推送了11410 — 映射差异，提交参考数据团队。"
- "子账将交易过账两次（交易ID 88412和88419是重复的）— 重复过账，屏蔽88419。"

## 输出

对每个追溯的差异，返回：

```json
{
  "key": "...",
  "root_cause": "上面格式的一句话",
  "owner": "ops | reference-data | accounting | upstream-system",
  "expected_clear_date": "YYYY-MM-DD 或 null",
  "action": "monitor | adjust | raise-ticket | suppress"
}
```

只有解析器写入调整分录 — 本技能仅诊断，不过账。

## Appendix: 金融背景知识

### 什么是差异追溯（Break Trace）？

在对账发现差异后，需要深入调查每笔差异的根本原因。这涉及追踪交易从源头（如交易系统）到最终过账（总账）的完整审计轨迹。

### 关键术语解释

**审计轨迹（Audit Trail）**：从交易发起到最终记账的完整记录链，包含时间戳、操作人、系统 ID等，用于追溯和验证。

**日记账分录（Journal Entry, JE）**：记录财务交易的标准会计凭证，包含借方和贷方。每笔交易在总账中至少对应一条日记账分录。

**分录ID（Entry ID）**：唯一标识一条日记账分录的编号，用于追溯。

**批次ID（Batch ID）**：批量过账时，系统生成的批次编号，包含多笔分录。

**源系统（Source System）**：产生交易的原始系统，如交易系统（OMS）、风险系统、工资系统等。

**交易对手（Counterparty）**：金融交易中与你方相对的另一方，如经纪商、交易对手银行、客户等。

**WM/ R 4pm汇率**：World Markets/Reuters下午4点汇率，是基金行业常用的外汇定价标准，由WM公司（现为伦敦证券交易所集团一部分）与Reuters合作提供。

**彭博收盘价（Bloomberg Close）**：彭博终端提供的每日市场收盘价，与WM/R汇率可能有微小差异（通常几个基点）。

**基點（Basis Point, bps）**：万分之一（0.01%）。12个基点的差异 = 0.12%的差异。

**参考数据（Reference Data）**：描述金融工具、实体、账户的静态数据，如证券ID、账户映射关系等。映射错误通常需要参考数据团队修正。

**屏蔽（Suppress）**：在对账系统中标记某笔差异为"已识别，无需处理"，通常适用于已知的、可解释的差异（如时序差异）。

**raise-ticket**：向IT或运营团队提交工单，要求修正系统问题（如重复过账、映射错误）。

**owner（负责人）**：
- **ops（运营部）**：负责日常交易处理和核对
- **reference-data（参考数据团队）**：维护证券、账户映射
- **accounting（会计部）**：负责总账和财务报表
- **upstream-system（上游系统）**：交易、风险等产生数据的源系统
