# 追溯差异根本原因

给定单个差异行（键、总账值、子分类账值、类别、可能原因），追溯到源并生成根本原因说明。

## 追溯路径

1. **提取总账方** — 通过internal-gl MCP，获取生成此总账行的日记账分录或过账：分录ID、过账日期、源系统、批次ID、编制人。
2. **提取子分类账方** — 通过子分类账MCP，获取匹配的交易：交易ID、交易/结算日期、交易对手、源馈送、使用的汇率。
3. **比较属性** — 对齐过账日期、汇率/日期、账户映射、数量符号、金额符号。不同的属性通常就是原因。

## 原因 → 说明

将根本原因写为一句话，格式为 **"⟨哪一方⟩ ⟨做了什么⟩ 因为 ⟨原因⟩"**，例如：

- "总账在结算日(T+2)过账，而子分类账在交易日过账 — 时间差异，将于2026-05-07结清。"
- "子分类账使用WM/R下午4点汇率；总账使用Bloomberg收盘价 — 基准金额上的外汇差异为12个基点。"
- "证券ABC123在映射表中映射到总账账户11420，但子分类账馈送了11410 — 映射差异，提交给参考数据团队。"
- "子分类账过账了两次交易（交易ID 88412和88419是重复的） — 重复过账，抑制88419。"

## 输出

对于每个追溯的差异，返回：

```json
{
  "key": "...",
  "root_cause": "如上所述的一句话",
  "owner": "ops | reference-data | accounting | upstream-system",
  "expected_clear_date": "YYYY-MM-DD 或 null",
  "action": "monitor | adjust | raise-ticket | suppress"
}
```

只有解决者才能写入调整 — 此技能仅诊断，不过账。

## 金融术语和知识解释

### 1. 根本原因分析 (Root Cause Analysis)

根本原因分析是识别问题根本原因的系统方法，用于预防问题再次发生。

### 2. 审计线索 (Audit Trail)

审计线索是追踪交易从原始凭证到最终报告的完整路径，用于验证和追溯。

### 3. T+2结算 (T+2 Settlement)

T+2结算是指交易发生后第二个工作日完成清算和交割。

### 4. WM/R汇率 (WM/R Rate)

WM/R汇率是由WM公司（现归Refinitiv所有）发布的基准汇率，常用于资产估值。

### 5. Bloomberg收盘价 (Bloomberg Close)

Bloomberg收盘价是Bloomberg终端在交易日结束时发布的官方汇率。

### 6. 基点 (Basis Points)

基点是金融中常用的单位，1个基点等于0.01%。

### 7. 参考数据 (Reference Data)

参考数据是描述金融工具、交易对手和市场的基础数据，如证券代码、账户映射等。

### 8. 重复过账 (Duplicate Posting)

重复过账是指同一交易被记录多次，会导致账目不平衡。

### 9. 抑制 (Suppress)

抑制是指忽略或标记重复或错误的交易，使其不影响对账结果。

### 10. 上游系统 (Upstream System)

上游系统是指提供数据给总账或子分类账的前置系统，如交易系统、清算系统等。
