## ⚠️ 关键渲染限制

**LibreOffice用于验证但无法准确渲染PowerPoint文件。** 它会弄乱字体、渐变、形状位置、文本换行和一些表格格式。

**这意味着什么：** 在LibreOffice中通过视觉验证的幻灯片在Microsoft PowerPoint中可能仍有问题。验证循环捕捉结构问题（缺失内容、断开的表格、占位符格式保留）但**无法**捕捉字体替换、微妙的偏移或渐变问题。

**必需操作：** 交付输出时始终包含此声明：
> "此文件使用LibreOffice验证。由于存在渲染差异，请在分发前在Microsoft PowerPoint中审查。"

## 模板填充工作流

复制并跟踪进度：

```
推介演示文稿进度：
- [ ] 阶段1：提取和验证源数据
- [ ] 阶段2：将内容映射到模板部分
- [ ] 阶段3：用正确格式填充幻灯片
- [ ] 阶段4：验证 → 修复 → 重复直到干净
- [ ] 阶段5：最终验证
```

### 阶段1：数据提取

1. **创建原始模板的备份** — 在任何修改之前复制到`[filename]_backup.pptx`。直接XML编辑或意外错误可能损坏文件。
2. 识别所有源材料（Excel、CSV、PDF报告、Word文档、数据库、Web源）
3. 从每个源提取相关数据点
4. 对照原始源验证所有数字
5. 标准化单位和货币（将所有数字转换为模板中使用的主要单位/货币）
6. 注意需要验证的任何计算 → 参见[`calculation-standards.md`](reference/calculation-standards.md)了解公式

### 阶段2：内容映射

1. **打开并视觉审查模板** — 在修改之前了解其结构、样式和现有内容
2. 分析模板结构 — 识别所有占位符区域和内容框
3. 将源数据映射到相应模板部分 → 参见[`slide-templates.md`](reference/slide-templates.md)了解映射指导
4. 识别占位符指导框（来自任务创建者的彩色说明框）
5. 注意任何数据缺口或不匹配 → 参见[`slide-templates.md`](reference/slide-templates.md#handling-data-template-mismatches)了解解决方案
6. 映射时，注意幻灯片上有哪些占位符框以及它们应填入什么数据

### 阶段3：模板填充

1. **移除或重新格式化占位符框** — 彩色说明框显示创建什么，不是如何格式化。删除它们并在其位置创建正确格式的内容。参见[关键反模式](#critical-anti-patterns---never-do-these)。
2. 用映射的内容填充每个部分（首先聚焦内容）
3. **然后应用格式**以匹配模板样式 → 参见[`formatting-standards.md`](reference/formatting-standards.md)
4. 创建表格作为实际表格对象（绝不使用管道/制表符分隔的文本） → 参见[`xml-reference.md`](reference/xml-reference.md#table-implementation)
5. 创建箭头/形状作为PowerPoint对象 → 参见[`xml-reference.md`](reference/xml-reference.md#arrow-shapes)
6. 如果任务文件中提供了公司标志则插入；如果不可用，标记给用户："[未提供LOGO — 请提供公司标志]"

### 阶段4：验证 → 修复 → 重复

**这是一个反馈循环。重复直到所有检查通过或触发升级。**

```bash
# 转换为图像进行视觉验证
soffice --headless --convert-to pdf presentation.pptx
pdftoppm -jpeg -r 150 presentation.pdf slide
```

**验证清单（检查每张幻灯片图像）：**
- [ ] 文本在背景上可读？
- [ ] 表格是实际对象（列对齐，不是管道/制表符分隔的文本）？
- [ ] 图表/表格填充指定区域？
- [ ] 要点格式在部分内一致？
- [ ] 相同级别框的字体大小匹配？
- [ ] 数字格式一致（$M vs $MM）？
- [ ] 所有数字都有引用？
- [ ] 没有占位符文本残留？

### 阶段5：最终验证

运行完整检查：
- [ ] 所有数字与源匹配
- [ ] 计算正确（总计求和、百分比相加、增长率匹配端点）
- [ ] 格式一致（字体、颜色、间距）
- [ ] 所有图表都有来源引用
- [ ] 没有视觉溢出或重叠
- [ ] 公司标志就位（或标记缺失）

## 关键反模式 — 绝不要做这些

这些错误会毁坏演示文稿的专业性：

| 反模式 | 为什么错误 | 正确做法 |
|----------|----------|----------|
| **保留占位符框** | 彩色指导框在最终演示文稿中可见 | 删除并替换为正确格式的内容 |
| **管道/制表符"表格"** | 不是真实表格，无法正确渲染 | 使用XML创建实际表格对象 |
| **字体不一致** | 看起来业余 | 全程使用模板字体（通常为Arial/Helvetica） |
| **数字格式混合** | $500M和$500,000,000在同一演示文稿中 | 选择$M或$MM并坚持使用 |
| **无来源引用** | 数字无归因 | 每张图表/表格都有来源脚注 |
| **文本溢出** | 文本跑出框外 | 检查边界，必要时调整字体大小 |
| **直接编辑XML无备份** | 文件损坏风险 | 始终先创建备份 |

## 常见故障与修复

| 问题 | 症状 | 修复 |
|------|------|------|
| 表格渲染错误 | 列未对齐，文本重叠 | 使用[`xml-reference.md`](reference/xml-reference.md#table-implementation)中的XML模式 |
| 箭头位置错误 | 箭头不在形状之间连接 | 使用[`xml-reference.md`](reference/xml-reference.md#arrow-shapes)中的正确XML |
| 字体替换 | 不同幻灯片上的字体不同 | 强制所有文本使用模板字体 |
| 数字格式漂移 | 同一幻灯片上$1.2B和$1200M | 标准化为单一格式 |
| 占位符残留 | 彩色框在最终输出中可见 | 删除所有指导框 |
| 缺失来源 | 图表无脚注 | 为每张图表添加来源行 |

## Appendix: 金融背景知识

### 什么是投行推介演示文稿（Pitch Deck）？

投行推介演示文稿是投资银行用来赢得新业务的正式演示文稿，通常包含：
- 公司概览和市场定位
- 财务表现和估值
- 可比公司分析
- 交易 rationale 和战略逻辑
- 建议的交易结构和条款

### 关键术语解释

**推介（Pitch）**：投行向潜在客户展示其能力和建议以赢得业务委托的过程。

**模板（Template）**：预先设计的演示文稿框架，包含占位符，待填入客户特定数据。

**占位符（Placeholder）**：模板中的彩色指导框，显示应填入什么内容，必须删除并替换为正确格式的内容。

**XML（可扩展标记语言）**：PowerPoint文件的底层格式。直接编辑XML可精确控制形状、表格、箭头等。

**LibreOffice**：开源办公套件，可用于验证PowerPoint文件，但渲染不准确。

**一致预期（Consensus）**：华尔街分析师对公司业绩预测的平均值。

**CAGR（复合年增长率）**：一项投资或指标在特定时期内的年均增长率。

**EBITDA**：息税折旧摊销前利润，衡量公司核心运营现金流。

**估值倍数（Valuation Multiples）**：如EV/收入、EV/EBITDA、P/E等，用于比较公司估值。

**数据源引用（Source Citation）**：每张图表/表格都应注明数据来源（如"FactSet 2024年11月"、"10-K 2024年Q3"）。

**备份（Backup）**：在修改前复制原始文件，防止意外损坏或错误。

**视觉验证（Visual Verification）**：将PowerPoint转换为PDF再转为JPEG图像，检查每张幻灯片的外观。

**格式标准（Formatting Standards）**：字体、颜色、间距、对齐等的一致应用，确保专业外观。

**计算标准（Calculation Standards）**：财务公式的正确应用，如CAGR = (期末/期初)^(1/年数) - 1。
